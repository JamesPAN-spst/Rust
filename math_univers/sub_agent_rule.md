## 二、子 Agent 必须执行的工作流程

### 步骤 1：学习阶段（不可跳过）
1. 完整阅读 `generation_rule.md`
2. 阅读示例页面 `examples/ch1.1_vectors.html`，理解：
   - 页面 HTML 结构
   - CSS class 命名规范
   - 交互组件使用方式
   - 知识点讲解风格
   - 练习题设计风格
3. 阅读本文件 `sub_agent_rule.md`

### 步骤 2：规划阶段
1. 在负责的单元文件夹中创建 `workflow.md`
2. 详细列出每章每个知识点的：
   - 核心概念
   - 计划使用的讲解方式（类比、对比、代码示例等）
   - 计划使用的交互组件
   - 快速测试题目方向
3. 规划课后练习的题目方向和难度分布
4. 规划章节间的过渡逻辑

### 步骤 3：内容生成阶段
对于每个章节：
1. 创建 `index.html` 文件
2. 按 generation_rule 中的页面结构模板编写
3. 确保所有交互组件使用标准 class 名称
4. 确保 KaTeX 公式语法正确
5. 确保 Rust 代码示例完整且可运行

**生成顺序：**
- 先生成 Ch X.1（基础概念章），确保风格正确
- 再依次生成 Ch X.2、Ch X.3
- 最后生成 Ch X.4（实战章），因为它依赖前三章内容

### 步骤 4：自我审阅阶段
1. 逐一检查审阅清单（见 generation_rule.md 第八节）
2. 按评分规则对每个章节打分
3. 将评分结果写入 `workflow.md` 的「工作后审阅」部分
4. 对评分低于 80 分的章节进行修改

### 步骤 5：提交阶段
1. 确认所有文件已保存
2. 更新 `workflow.md` 的最终状态
3. 汇报工作完成

## 三、内容质量要求（细化）

### 3.1 每个知识点模块的最低要求
- ✅ 有引导式提问或思考提示
- ✅ 有形式化定义或公式
- ✅ 有至少一个具体例子
- ✅ 有 Rust 代码示例（Ch X.4 实战章必须有完整项目代码）
- ✅ 有 1-2 道快速测试题

### 3.2 代码示例标准
```rust
// ✅ 好的代码示例：有注释，有上下文，有输出
/// 计算两个向量的点积
fn dot_product(a: &[f64], b: &[f64]) -> f64 {
    // 使用迭代器实现——Rust 风格的写法
    a.iter()
        .zip(b.iter())
        .map(|(x, y)| x * y)
        .sum()
}

fn main() {
    let v1 = vec![1.0, 2.0, 3.0];
    let v2 = vec![4.0, 5.0, 6.0];
    
    let result = dot_product(&v1, &v2);
    println!("v1 · v2 = {}", result);
    // 输出：v1 · v2 = 32
    // 因为 1×4 + 2×5 + 3×6 = 4 + 10 + 18 = 32
}
```

```rust
// ❌ 坏的代码示例：无注释，无上下文，功能不明
fn dp(a: &[f64], b: &[f64]) -> f64 {
    a.iter().zip(b.iter()).map(|(x, y)| x * y).sum()
}
```

### 3.3 快速测试标准
```
✅ 好的快速测试：聚焦易错点
Q: 向量 a = [1, 2, 3] 和 b = [4, 5, 6] 的叉积结果是？
A. 标量 32
B. 向量 [-3, 6, -3]
C. 向量 [3, -6, 3]
D. 标量 0
正确答案：B
解析：叉积结果是向量，不是标量（排除 A、D）。
按公式 a × b = [a₂b₃ - a₃b₂, a₃b₁ - a₁b₃, a₁b₂ - a₂b₁]
= [2×6 - 3×5, 3×4 - 1×6, 1×5 - 2×4] = [-3, 6, -3]

❌ 坏的快速测试：过于简单，无诊断价值
Q: 向量可以相加吗？
A. 可以
B. 不可以
```

### 3.4 课后练习标准
- 基础题必须覆盖本章所有知识点
- 进阶题必须要求综合运用至少 2 个知识点
- 挑战题可以涉及证明或高级概念
- 附加题必须包含 Rust 代码编写或工程场景

## 四、页面一致性要求

### 4.1 文件命名规范
```
math_univers/
├── unitX_topic/
│   ├── chX.Y_subtopic/
│   │   └── index.html
│   └── workflow.md
```

### 4.2 CSS Class 命名规范
所有组件必须使用 generation_rule.md 中定义的标准 class 名称：
- `.chapter-header` — 章节头
- `.learning-objectives` — 学习目标
- `.prerequisites` — 前置知识
- `.knowledge-point` — 知识点模块
- `.think-box` — 思考提示
- `.code-playground` — 代码区
- `.quick-quiz` — 快速测试
- `.collapsible` — 折叠区
- `.comparison-table` — 对比表
- `.note-box` — 提示框（含 `.info` / `.warning` / `.tip` / `.danger` 修饰符）
- `.step-by-step` — 步骤分解
- `.chapter-summary` — 章节总结
- `.exercises` — 练习区
- `.further-reading` — 延伸阅读

### 4.3 ID 命名规范
- 知识点：`id="kp-X"` （X 为序号）
- 快速测试：`data-answer="X"` （X 为正确选项字母）
- 练习题：`id="ex-X"` （X 为题号）

## 六、特殊情况处理

### 6.1 知识点依赖其他单元
- 使用「前置知识提示」组件指向相关章节
- 提供简短回顾，不要求读者跳转阅读

### 6.2 数学符号复杂度
- 简单符号内联使用 KaTeX：`<span class="katex-inline">公式</span>`
- 复杂公式单独成行：`<div class="katex-display">公式</div>`
- 所有 KaTeX 公式使用标准 LaTeX 语法

### 6.3 代码示例长度
- 单个代码示例不超过 40 行
- 超过 40 行的代码使用折叠组件
- 实战章（Ch X.4）可以有完整项目代码，但需要分段讲解
