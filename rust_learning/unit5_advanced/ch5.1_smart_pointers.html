Ch 5.1 智能指针

智能指针不只是"包一层的引用"——它们各自回答了一个所有权问题：谁来分配？谁来共享？谁来决定可变性？理解每种指针的使用时机，是写出地道 Rust 代码的关键。

🎯 学习目标

  • 掌握 Box<T> 的堆分配语义与递归类型应用
  • 理解 Rc<T> 的共享所有权模型及引用计数开销
  • 对比 Arc<T> 与 Rc<T>，明确线程安全边界
  • 掌握 RefCell<T> 的内部可变性及运行时借用检查
  • 了解 Cell<T> 和 Cow<T> 的适用场景

📋 前置知识

需要对 Ownership、Borrowing、Lifetime 有清晰理解。智能指针本质上是在所有权模型上做更灵活的安排，如果基础概念不稳，这一章会感觉"到处都是例外"。

5.1.1 Box：最简单的堆分配

🤔

Rust 默认把数据放在栈上，什么时候必须把数据放到堆上？

Box<T> 是 Rust 中最基础的堆分配智能指针。它把数据从栈搬到堆，栈上只留一个固定大小的指针。最经典的场景是递归类型——编译器无法计算无限嵌套的大小，而 Box 提供了一个间接层让大小变得确定。

正式规则：Box<T> 拥有堆上的 T，当 Box 被 drop 时，堆内存自动释放。它实现了 Deref，所以你可以像使用 &T 一样透明地使用它。

Rustbox_recursive.rs
// 递归类型必须借助 Box 间接引用
enum List {
    Cons(i32, Box<List>),
    Nil,
}

fn main() {
    let list = List::Cons(1,
        Box::new(List::Cons(2,
            Box::new(List::Cons(3,
                Box::new(List::Nil))))));

    // Box 也可以用于大型数据的堆分配
    let big_array = Box::new([0u8; 1024]);
    println!("堆上数组长度: {}", big_array.len());
}
输出 
堆上数组长度: 1024

5.1.2 Rc：单线程下的共享所有权

🧠

如果一个值需要被多个地方"同时拥有"，Rust 的单一所有权怎么办？

Rc<T>（Reference Counted）通过引用计数让多个变量共享同一份堆上数据的所有权。每次 Rc::clone 只增加计数器，不做深拷贝。当最后一个 Rc 被 drop 时，数据才被释放。

关键限制：Rc 不是线程安全的——引用计数的加减不是原子操作。如果你需要跨线程共享，必须使用 Arc。

Rustrc_sharing.rs
use std::rc::Rc;

fn main() {
    let data = Rc::new(vec![1, 2, 3]);
    let a = Rc::clone(&data);
    let b = Rc::clone(&data);

    println!("引用计数: {}", Rc::strong_count(&data));
    println!("a = {:?}", a);
    println!("b = {:?}", b);
}
输出 
引用计数: 3
a = [1, 2, 3]
b = [1, 2, 3]
Rc::clone(&rc) 只增加引用计数（成本极低），不会复制堆上数据。而 T::clone() 通常会执行深拷贝。社区习惯用 Rc::clone 来明确表示"这只是增加一个引用"。

5.1.3 Arc：跨线程的共享所有权

🔒

多个线程需要读同一份配置数据，不需要修改，最轻量的方案是什么？

Arc<T>（Atomically Reference Counted）与 Rc 功能类似，但引用计数操作使用原子指令，因此可以安全地在线程间共享。原子操作有一定性能开销，所以如果确定不跨线程，优先用 Rc。

组合模式：Arc<T> 只提供共享的不可变访问。如果需要跨线程修改内部数据，搭配 Mutex 或 RwLock 使用：Arc<Mutex<T>>。

Rustarc_threads.rs
use std::sync::Arc;
use std::thread;

fn main() {
    let config = Arc::new(String::from("production"));
    let mut handles = vec![];

    for i in 0..3 {
        let cfg = Arc::clone(&config);
        handles.push(thread::spawn(move || {
            println!("线程 {i} 读取配置: {cfg}");
        }));
    }

    for h in handles {
        h.join().unwrap();
    }
}
输出 
线程 0 读取配置: production
线程 1 读取配置: production
线程 2 读取配置: production

5.1.4 RefCell：编译期借不了，运行时再借

🔓

如果你持有一个 &self 的不可变引用，却需要修改内部某个字段，怎么办？

RefCell<T> 把借用检查从编译期推迟到运行时。通过 borrow() 获取不可变引用，通过 borrow_mut() 获取可变引用——如果违反借用规则，程序会在运行时 panic 而不是编译失败。这就是所谓"内部可变性"模式。

风险提醒：运行时 panic 比编译错误更危险。只在编译器的静态分析确实无法满足你的合理需求时才使用 RefCell，比如在 Rc<RefCell<T>> 中实现图结构的共享可变节点。

Rustrefcell_interior.rs
use std::cell::RefCell;

fn main() {
    let data = RefCell::new(vec![1, 2, 3]);

    // 不可变借用
    println!("读取: {:?}", data.borrow());

    // 可变借用——即使 data 本身不是 mut
    data.borrow_mut().push(4);

    println!("修改后: {:?}", data.borrow());
}
输出 
读取: [1, 2, 3]
修改后: [1, 2, 3, 4]
Rc 让多个地方共享所有权，RefCell 允许通过共享引用来修改内部值。两者组合在构建树、图等需要共享可变节点的数据结构时非常常见。但要注意：运行时借用冲突会导致 panic。

5.1.5 Cell 与 Cow：轻量级的灵活性

💡

每次都用 RefCell 会不会太重了？有没有更轻量的方案？

Cell<T> 适用于 Copy 类型的内部可变性——它通过整体替换值而非返回引用来避免借用冲突，没有运行时借用检查的开销。Cow<T>（Clone on Write）则是一种延迟克隆策略：只在需要修改时才把借用的数据克隆一份，读多写少时非常高效。

选择指南：Copy 小类型用 Cell；需要共享引用+运行时可变用 RefCell；读多写少且想避免无谓复制用 Cow；跨线程共享用 Arc；单线程共享所有权用 Rc；简单堆分配用 Box。

Rustcell_cow.rs
use std::cell::Cell;
use std::borrow::Cow;

fn main() {
    // Cell：Copy 类型的内部可变性
    let counter = Cell::new(0u32);
    counter.set(counter.get() + 1);
    counter.set(counter.get() + 1);
    println!("Cell counter: {}", counter.get());

    // Cow：读时借用，写时克隆
    let input: Cow<str> = Cow::Borrowed("hello");
    // 不修改，零拷贝
    println!("原始: {input}");

    let mut owned: Cow<str> = Cow::Borrowed("hello");
    // 修改时自动克隆为 Owned
    owned.to_mut().push_str(" world");
    println!("修改后: {owned}");
}
输出 
Cell counter: 2
原始: hello
修改后: hello world
如果类型实现了 Copy（如 i32、bool、f64），Cell 更轻量——它直接替换值而非借用，所以根本不需要运行时借用检查。而 RefCell 适合非 Copy 类型或者需要返回引用的场景。

📌 章节总结

  • Box<T> 是最简单的堆分配，适合递归类型和大数据。
  • Rc<T> 提供单线程共享所有权，引用计数开销极低。
  • Arc<T> 是 Rc 的线程安全版本，使用原子操作。
  • RefCell<T> 提供运行时借用检查的内部可变性，违规时 panic。
  • Cell<T> 适合 Copy 类型的轻量内部可变性；Cow<T> 实现延迟克隆。
  • 选择智能指针时，先问"谁拥有？谁能改？要不要跨线程？"
