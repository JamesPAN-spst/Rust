Unit 5 · 高级特性

Ch 5.2 泛型与 Trait 对象

泛型让一份代码适用多种类型，Trait 对象让不同类型在运行时统一操作。理解静态分发与动态分发的取舍，是 Rust 类型系统的核心功力。

01泛型函数与结构体

🎯 为什么需要了解这个

不用泛型，你会为 i32 写一个 largest，为 f64 再写一个，代码完全重复。泛型用 <T> 抽象掉具体类型，一份代码覆盖所有可能。

🔍 核心思路

泛型是编译期的类型参数——告诉编译器"这个位置可以是任何满足约束的类型"。

  • 泛型函数：fn foo<T>(x: T)
  • 泛型结构体：struct Pair<T> { first: T, second: T }
  • 裸泛型什么都不能做：T 没有约束时，编译器不知道 T 有什么方法
📐 规则与语法
RUSTgenerics_basics.rs
// 泛型函数：T 加 PartialOrd 约束才能比较
fn largest<T: PartialOrd>(a: T, b: T) -> T {
    if a >= b { a } else { b }
}

// 泛型结构体
struct Pair<T> { first: T, second: T }

impl<T: std::fmt::Display> Pair<T> {
    fn show(&self) {
        println!("({}, {})", self.first, self.second);
    }
}

fn main() {
    println!("{}", largest(10, 20));
    println!("{}", largest(3.14, 2.72));
    Pair { first: "hello", second: "world" }.show();
}
输出
20
3.14
(hello, world)
🌐 工程实践

标准库中泛型无处不在：Vec<T>、Option<T>、Result<T, E>、HashMap<K, V>。它们都是编译期单态化的——用 Vec<i32> 和 Vec<String> 不会产生运行时多态开销。

✏️ 示例分析与易错点

常见错误：裸泛型调方法 → fn print<T>(x: T) { println!("{x}"); } 编译失败，因为 T 没有 Display 约束。修复：fn print<T: Display>(x: T)。

⚡ 快速检测
fn foo<T>(x: T) 中能对 x 做什么操作？

02Trait Bound、where 子句、impl Trait

🎯 为什么需要了解这个

裸泛型什么都做不了，Trait Bound 是给泛型"解锁能力"的钥匙。三种写法（内联、where、impl Trait）适用于不同复杂度。

🔍 核心思路
  • 内联：<T: Display + Clone> 简洁，适合一两个约束
  • where 子句：约束多或复杂时更清晰
  • impl Trait：参数位置是语法糖，返回位置表示"由实现决定的具体类型"
📐 规则与语法
RUSTtrait_bounds.rs
use std::fmt::Display;

// 内联写法
fn show<T: Display>(x: T) { println!("{x}"); }

// where 子句
fn process<T, U>(t: T, u: U) -> String
where
    T: Display + Clone,
    U: Into<String>,
{
    format!("{} - {}", t, u.into())
}

// impl Trait 返回类型
fn make_greeting(name: &str) -> impl Display {
    format!("Hello, {name}!")
}

fn main() {
    show(42);
    println!("{}", process("key", String::from("value")));
    println!("{}", make_greeting("Rustacean"));
}
输出
42
key - value
Hello, Rustacean!
🌐 工程实践

习惯约定：一两个简单约束用内联，三个以上或涉及关联类型用 where。impl Trait 返回类型常见于迭代器和闭包。

✏️ 示例分析与易错点

impl Trait 返回类型不能返回多种类型：

RUSTimpl_trait_error.rs
// ❌ 编译错误——两个分支返回不同具体类型
fn make_shape(circle: bool) -> impl Display {
    if circle { "⚪" } else { 42 }  // &str vs i32
}
// ✅ 用 Box<dyn Display> 解决
fn make_shape(circle: bool) -> Box<dyn Display> {
    if circle { Box::new("⚪") } else { Box::new(42) }
}
编译错误（第一种写法）
`if` and `else` have incompatible types

03单态化：零成本泛型

🎯 为什么需要了解这个

"泛型有运行时开销吗？"——很多从 Java/Python 来的开发者会担心这个问题。答案是：Rust 的泛型通过单态化实现零运行时开销。理解这个机制能让你放心使用泛型而不担心性能。

🔍 核心思路

编译器把 fn foo<T> 在使用时"展开"成 foo_i32、foo_string 等专用版本——每个具体类型一份机器码。

  • 零运行时开销：没有类型检查、没有虚函数查找
  • 代价在编译期：更长的编译时间、更大的二进制文件
📐 规则与语法
RUSTmonomorphization.rs
fn double<T: Copy + std::ops::Add<Output=T>>(x: T) -> T {
    x + x
}

fn main() {
    // 编译器生成 double_i32 和 double_f64 两个版本
    println!("{}", double(5));      // double::
    println!("{}", double(3.14));   // double::
}
输出
10
6.28
🌐 工程实践

放心用泛型：不要为了"减少编译时间"而避免泛型，除非在极端情况下（如模板膨胀严重的 Bevy 系统）。

看编译后大小：cargo bloat 可以检查哪些泛型函数展开后占用最多空间。

✏️ 示例分析与易错点

常见误解：

  • "泛型和 Java 的泛型一样有类型擦除" → Rust 泛型不擦除，编译期完全展开。
  • "dyn Trait 和泛型一样零开销" → dyn Trait 有 vtable 间接调用开销。

04dyn Trait：动态分发

🎯 为什么需要了解这个

有时你需要在运行时处理不同类型——比如一个 Vec 存储不同形状的对象，或者插件架构中加载未知类型。泛型在编译期确定，满足不了这种需求。dyn Trait 提供运行时多态。

🔍 核心思路

Trait 对象通过胖指针（数据指针 + vtable 指针）实现动态分发：

  • vtable 是一张方法查找表，存储了具体类型的每个方法地址
  • 调用时先查表，再跳转——比直接调用多一次间接寻址
  • dyn Trait 是 DST（动态大小类型），不能直接放栈上，必须用 Box<dyn Trait> 或 &dyn Trait
📐 规则与语法
RUSTdyn_trait.rs
trait Drawable {
    fn draw(&self);
    fn area(&self) -> f64;
}

struct Circle { radius: f64 }
struct Rect   { w: f64, h: f64 }

impl Drawable for Circle {
    fn draw(&self)   { println!("⚪ r={}", self.radius); }
    fn area(&self) -> f64 { std::f64::consts::PI * self.radius * self.radius }
}
impl Drawable for Rect {
    fn draw(&self)   { println!("▭ {}x{}", self.w, self.h); }
    fn area(&self) -> f64 { self.w * self.h }
}

fn main() {
    let shapes: Vec<Box<dyn Drawable>> = vec![
        Box::new(Circle { radius: 5.0 }),
        Box::new(Rect { w: 4.0, h: 3.0 }),
    ];
    for s in &shapes {
        s.draw();
        println!("  area = {:.2}", s.area());
    }
}
输出
⚪ r=5
  area = 78.54
▭ 4x3
  area = 12.00
🌐 工程实践

异构集合：Vec<Box<dyn Trait>> 是存储不同类型对象的标准方式。

插件架构：定义 trait 接口，插件 impl trait，主程序通过 dyn Trait 调用。

性能敏感路径少用：vtable 间接调用阻止内联优化。热路径上优先用泛型。

✏️ 示例分析与易错点

常见错误：忘了 dyn 关键字（旧版 Rust 允许省略，现在报 warning/error）；把 non-object-safe trait 用 dyn 包装。

05对象安全规则

🎯 为什么需要了解这个

不是所有 trait 都能用 dyn 包装。尝试 Box<dyn Clone> 会编译失败——因为 Clone 不"对象安全"。理解对象安全规则，能让你设计出既支持泛型又支持 trait 对象的接口。

🔍 核心思路

对象安全的核心限制：vtable 必须能在不知道具体类型的情况下工作。

  • 方法不能返回 Self——因为 vtable 调用时不知道 Self 的大小
  • 方法不能有泛型参数——因为 vtable 需要每个方法固定一个签名
  • Clone 不安全：fn clone(&self) -> Self 返回 Self
  • Display 安全：fn fmt(&self, f: &mut Formatter) -> Result 不含 Self
📐 规则与语法
RUSTobject_safety.rs
trait Describe {
    fn name(&self) -> &str;        // ✅ 对象安全
    fn to_json(&self) -> String;   // ✅ 返回 String 而非 Self
    // fn duplicate(&self) -> Self; // ❌ 返回 Self
    // fn convert<U>(&self) -> U;  // ❌ 泛型参数
}

struct Item { label: String }
impl Describe for Item {
    fn name(&self) -> &str { &self.label }
    fn to_json(&self) -> String { format!("{{\"name\":\"{}\"}}", self.label) }
}

fn print_desc(d: &dyn Describe) {
    println!("{}: {}", d.name(), d.to_json());
}

fn main() {
    let item = Item { label: String::from("widget") };
    print_desc(&item);
}
输出
widget: {"name":"widget"}
🌐 工程实践

设计 trait 时考虑对象安全：如果你的 trait 需要用 dyn，避免返回 Self 和泛型方法。

变通方法：

  • 返回 Box<dyn Trait> 代替 Self
  • 使用关联类型代替泛型方法参数
  • 用 where Self: Sized 标注非对象安全方法，让 trait 其余部分仍可用 dyn
✏️ 示例分析与易错点

常见错误：尝试 Box<dyn Clone> → 编译报 "the trait Clone cannot be made into an object"。解决方案：自定义一个 DynClone trait。
