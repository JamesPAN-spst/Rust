Ch 5.3 宏系统

宏是 Rust 元编程的核心工具——它在编译期生成代码，可以减少重复、创造 DSL、自动派生实现。但宏也是"最容易写出难以维护代码"的特性。理解什么时候该用宏、什么时候不该用，是成熟 Rust 工程师的标志。

🎯 学习目标

  • 掌握 macro_rules! 声明式宏的基本语法和匹配规则
  • 理解重复模式 $(...)* 和宏的卫生性（hygiene）
  • 明确宏的使用边界——什么时候函数或泛型是更好的选择
  • 了解三种过程宏（derive、属性、函数式）的功能定位
  • 掌握宏展开调试的实用技巧

📋 前置知识

需要扎实的 Rust 基础语法、Trait 系统和泛型知识。宏本质上是在生成 Rust 代码，如果对生成目标的语法不熟悉，写宏和调试宏都会很痛苦。

5.3.1 macro_rules!：声明式宏入门

🤔

vec![1, 2, 3] 这种写法为什么不是函数调用？它背后发生了什么？

macro_rules! 定义的宏通过模式匹配输入的 token 序列，然后展开为新的代码。它不像函数那样在运行时工作——宏在编译期完成文本替换级别的代码生成。每个模式分支用 => 指定匹配规则和展开模板。

正式规则：macro_rules! 使用片段说明符（fragment specifiers）匹配不同种类的语法元素：$x:expr 匹配表达式，$t:ty 匹配类型，$i:ident 匹配标识符，等等。

Rustmacro_basics.rs
// 最简单的声明式宏
macro_rules! say_hello {
    () => {
        println!("Hello from macro!");
    };
    ($name:expr) => {
        println!("Hello, {}!", $name);
    };
}

// 模拟 vec! 的简化版
macro_rules! my_vec {
    ( $( $x:expr ),* ) => {
        {
            let mut temp = Vec::new();
            $( temp.push($x); )*
            temp
        }
    };
}

fn main() {
    say_hello!();
    say_hello!("Rustacean");

    let v = my_vec![10, 20, 30];
    println!("{:?}", v);
}
输出 
Hello from macro!
Hello, Rustacean!
[10, 20, 30]

5.3.2 重复模式与卫生性

🔄

宏展开时创建的变量名，会不会和调用者的变量名冲突？

重复模式 $( ... ),* 让宏能处理可变数量的参数——* 表示零次或多次，+ 表示一次或多次。Rust 宏具有"卫生性"（hygiene）：宏内部定义的变量名默认不会和外部冲突，编译器会自动给它们加上不同的语法上下文。

卫生性保证：宏内部的 let temp = ... 不会覆盖调用处已有的 temp 变量。这让宏比 C 的文本替换宏安全得多。但卫生性也有限度——过程宏的卫生性规则有所不同。

Rustrepetition_hygiene.rs
// 重复模式：为每个字段生成 getter
macro_rules! make_getters {
    ( $struct_name:ident { $( $field:ident : $ty:ty ),* } ) => {
        struct $struct_name {
            $( $field: $ty, )*
        }

        impl $struct_name {
            $( fn $field(&self) -> &$ty { &self.$field } )*
        }
    };
}

make_getters!(Person { name: String, age: u32 });

fn main() {
    let temp = 999; // 外部的 temp 不受宏影响
    let p = Person { name: String::from("Alice"), age: 30 };
    println!("{}, age {}", p.name(), p.age());
    println!("外部 temp 仍然是: {temp}");
}
输出 
Alice, age 30
外部 temp 仍然是: 999

macro_rules! 对局部变量有卫生性保护，但对类型名、函数名、模块路径等的卫生性保护有限。过程宏默认使用 Span::call_site()，变量名在调用者作用域解析，行为更接近文本替换。理解这些差异对避免宏的命名陷阱很重要。

5.3.3 什么时候不该用宏？

⚠️

宏能做的事，函数和泛型是不是也能做？为什么很多库的宏代码特别难读？

宏的最大问题是可读性和调试难度。错误信息指向展开后的代码而非宏定义，IDE 支持也不如函数完善。原则是：如果函数或泛型能做到的事，不要用宏。宏应该只用在"必须在编译期生成不同结构的代码"的场景。

宏的适用场景：① 可变参数数量（函数做不到）；② 需要生成新的类型/函数/impl（泛型做不到）；③ DSL 语法糖。不适用场景：普通的代码复用、简单的条件逻辑、可以用泛型解决的多类型问题。

Rustmacro_vs_function.rs
// ❌ 不需要宏的场景——用函数就够了
// macro_rules! add { ($a:expr, $b:expr) => { $a + $b }; }

// ✅ 用函数更清晰
fn add(a: i32, b: i32) -> i32 { a + b }

// ✅ 宏的正当场景：可变参数 + 生成重复代码
macro_rules! impl_display {
    ( $( $t:ty => $fmt:expr ),* ) => {
        $(
            impl std::fmt::Display for $t {
                fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
                    write!(f, $fmt, self.0)
                }
            }
        )*
    };
}

struct Meters(f64);
struct Seconds(f64);
impl_display!(Meters => "{:.1}m", Seconds => "{:.1}s");

fn main() {
    println!("{}", add(1, 2));
    println!("{}", Meters(3.14));
    println!("{}", Seconds(9.8));
}
输出 
3
3.1m
9.8s

5.3.4 过程宏：derive、属性与函数式

🏗️

#[derive(Debug, Clone)] 是怎么自动为你的结构体生成代码的？

过程宏（procedural macros）是另一类宏，它们是真正的 Rust 函数——接收 TokenStream 输入，返回 TokenStream 输出。分为三种：derive 宏（自动为类型派生实现）、属性宏（类似 #[test]）、函数式宏（看起来像函数调用但在编译期展开）。过程宏必须定义在单独的 proc-macro crate 中。

三种过程宏：① #[derive(MyTrait)] — 在现有类型上添加实现；② #[my_attr] — 对任意 item 做变换；③ my_macro!(...) — 函数式调用，输入任意 token。

Rustproc_macro_usage.rs
// 过程宏的使用端代码（定义端需要单独 crate）

// derive 宏——最常见的过程宏形式
#[derive(Debug, Clone, PartialEq)]
struct Config {
    name: String,
    value: i32,
}

// 类似 serde 的用法：
// #[derive(Serialize, Deserialize)]
// struct ApiResponse { ... }

fn main() {
    let c1 = Config { name: "timeout".into(), value: 30 };
    let c2 = c1.clone();

    println!("{:?}", c1);
    println!("相等: {}", c1 == c2);
}
输出 
Config { name: "timeout", value: 30 }
相等: true

过程宏需要在编译期运行，而编译器处理当前 crate 时需要先编译好宏的代码。如果宏和使用它的代码在同一个 crate 中，就会产生循环依赖。单独的 proc-macro crate 保证宏代码先于使用方编译完成。

5.3.5 宏调试：看清展开后的代码

🔍

宏报错时，错误指向一堆你没写过的代码——怎么知道宏到底展开成了什么？

调试宏最重要的工具是 cargo expand（需要安装 cargo-expand）——它能展示所有宏展开后的完整代码。另外，编译器自带的 -Zunpretty=expanded（nightly）也能查看展开结果。在写复杂宏时，养成"先展开看看"的习惯可以节省大量调试时间。

调试三板斧：① cargo expand 查看完整展开；② 用 compile_error! 在宏内部输出诊断信息；③ 先写出你期望生成的代码，再反推宏的匹配模式。

Bashmacro_debug.sh
# 安装 cargo-expand
# cargo install cargo-expand

# 查看所有宏展开后的代码
# cargo expand

# 只看某个模块
# cargo expand module_name

# nightly 编译器自带的展开
# rustc -Zunpretty=expanded src/main.rs
说明 
cargo expand 会输出所有 derive、macro_rules! 等展开后的完整 Rust 代码
Rustcompile_error_debug.rs
// 用 compile_error! 做宏内部断言
macro_rules! only_ident {
    ($i:ident) => {
        println!("Got ident: {}", stringify!($i));
    };
    ($other:expr) => {
        compile_error!("only_ident! 只接受标识符，不接受表达式");
    };
}

fn main() {
    only_ident!(hello);
    // only_ident!(1 + 2);  // 取消注释会触发编译错误
}
输出 
Got ident: hello

📌 章节总结

  • macro_rules! 通过模式匹配 token 序列来生成代码，适合可变参数和重复模式。
  • Rust 宏具有卫生性，内部变量不会与外部冲突。
  • 如果函数或泛型能解决问题，不要用宏——宏的可读性和调试成本更高。
  • 过程宏（derive、属性、函数式）是真正的编译期函数，功能更强但需要单独 crate。
  • cargo expand 和 compile_error! 是宏调试的核心工具。

🧪 课后测试

  1. ⭐ 基础： 写一个 hashmap! 宏，支持 hashmap!{ "a" => 1, "b" => 2 } 语法创建 HashMap。
  2. ⭐⭐ 进阶： 用 macro_rules! 实现一个 newtype! 宏，自动为 newtype 生成 struct 定义和 Display 实现。
  3. ⭐⭐⭐ 挑战： 分析一段使用 macro_rules! 的库代码（如 lazy_static! 或 bitflags!），用 cargo expand 查看展开结果并解释每个匹配分支的作用。
  4. 🏆 附加： 设计一个声明式宏，可以根据 "字段名: 类型" 的列表生成完整的 builder pattern 代码。讨论这种用法和过程宏方案的取舍。
