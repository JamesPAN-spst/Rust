Ch 5.3 宏系统

宏是 Rust 元编程的核心——在编译期用代码生成代码。从 macro_rules! 的模式匹配到过程宏的 TokenStream 操作，Rust 宏系统既强大又特殊。

01macro_rules! 基础

🎯 为什么需要了解这个

你每天写的 println!、vec!、format! 都是宏。理解宏的语法，才能读懂标准库源码，也能在需要时编写自己的 DSL。

🔍 核心思路

macro_rules! 是模式匹配 + 代码模板：

  • 左边写匹配模式（类似函数参数），用"片段说明符"捕获 Rust 语法元素
  • 右边写展开结果（代码模板），用 $name 插入捕获的内容
  • 多个分支用 ; 分隔，从上到下尝试匹配
📐 规则与语法
RUSTmacro_basics.rs
// 简单宏：一个分支
macro_rules! say_hello {
    () => { println!("Hello!"); };
}

// 带参数的宏
macro_rules! greet {
    ($name:expr) => {
        println!("Hello, {}!", $name);
    };
}

// 多分支
macro_rules! calculate {
    (add $a:expr, $b:expr) => { $a + $b };
    (mul $a:expr, $b:expr) => { $a * $b };
}

fn main() {
    say_hello!();
    greet!("Rust");
    println!("{}", calculate!(add 2, 3));
    println!("{}", calculate!(mul 4, 5));
}
输出
Hello!
Hello, Rust!
5
20
🌐 工程实践

常见片段说明符：expr（表达式）、ident（标识符）、ty（类型）、stmt（语句）、block（块）、tt（token tree，最灵活）、literal（字面量）。

✏️ 示例分析与易错点

常见错误：忘记宏名后面的 !；分支末尾漏写 ;。

02重复模式与卫生性

🎯 为什么需要了解这个

宏最有用的特性是处理可变数量的参数——像 vec![1,2,3] 可以接受任意多个元素。重复模式就是实现这一功能的关键语法。

🔍 核心思路
  • 重复语法：$($x:expr),* → 匹配零个或多个逗号分隔的表达式
  • * = 零个或多个，+ = 一个或多个，? = 零个或一个
  • 展开时：$($x),* 重复产出每一项
  • 卫生性：宏内部的变量名自动"重命名"，不会与调用处冲突
📐 规则与语法
RUSTrepetition.rs
// 模拟 vec! 宏
macro_rules! my_vec {
    ( $( $elem:expr ),* ) => {{
        let mut v = Vec::new();
        $( v.push($elem); )*
        v
    }};
}

// 用 HashMap 简写
macro_rules! hashmap {
    ( $( $key:expr => $val:expr ),* $(,)? ) => {{
        let mut m = std::collections::HashMap::new();
        $( m.insert($key, $val); )*
        m
    }};
}

fn main() {
    let v = my_vec![10, 20, 30];
    println!("{:?}", v);
    
    let scores = hashmap! {
        "Alice" => 95,
        "Bob" => 87,
    };
    println!("{:?}", scores);
}
输出
[10, 20, 30]
{"Alice": 95, "Bob": 87}
🌐 工程实践

尾逗号处理：$(,)? 允许最后一项后有可选的逗号，提升使用体验。

卫生性限制：声明式宏是"部分卫生"的——局部变量名受保护，但模块路径、trait 需要手动完整路径。

✏️ 示例分析与易错点

常见错误：展开时忘记对应的重复符号 $(...)*；重复分隔符不匹配（匹配用 , 展开也要用 ,）。

03何时不该用宏

🎯 为什么需要了解这个

宏很有诱惑力——"看！这几行宏生成了几百行代码！"但宏的代价是可读性、可调试性和 IDE 支持严重下降。知道边界比知道能力更重要。

🔍 核心思路

决策流程：

  1. 能用函数/泛型解决 → 不用宏
  2. 需要可变参数或语法扩展 → 考虑 macro_rules!
  3. 需要操作类型定义或 derive → 考虑过程宏
📐 规则与语法
RUSTmacro_vs_function.rs
// ❌ 不该用宏——普通函数就够了
macro_rules! add {
    ($a:expr, $b:expr) => { $a + $b };
}

// ✅ 直接用函数
fn add(a: i32, b: i32) -> i32 { a + b }

// ✅ 该用宏——需要可变参数
macro_rules! log_values {
    ($($v:expr),+) => {
        $(
            println!("{} = {:?}", stringify!($v), $v);
        )+
    };
}

fn main() {
    let x = 42;
    let name = "Rust";
    log_values!(x, name, x + 1);
}
输出
x = 42
name = "Rust"
x + 1 = 43
🌐 工程实践

合理使用场景：可变参数日志、DSL 构建（如测试框架的 assert_eq!）、自动 derive、样板代码消除。

过度使用信号：代码中大量 !(...) 调用；IDE 无法跳转定义；错误信息让人摸不着头脑。

✏️ 示例分析与易错点

stringify! 的妙用：把表达式的源码文本变成字符串。只有宏能做到这点——函数拿到的是值，看不到源码。

04过程宏

🎯 为什么需要了解这个

#[derive(Debug, Serialize)] 和 #[tokio::main] 都是过程宏。它们比 macro_rules! 强大得多——可以检查类型结构、生成任意代码。理解过程宏让你能读懂生态系统中最重要的框架。

🔍 核心思路

过程宏用 Rust 函数操作 TokenStream（token 流），三种类型：

  • derive 宏：#[derive(MyTrait)] → 为类型自动生成 impl
  • 属性式宏：#[route(GET, "/")] → 修改/替换被标注的代码
  • 函数式宏：sql!("SELECT ...") → 类似 macro_rules! 但功能更强
📐 规则与语法
RUSTproc_macro_derive.rs
// 在 proc-macro crate 中定义 derive 宏
use proc_macro::TokenStream;
use quote::quote;
use syn::{parse_macro_input, DeriveInput};

#[proc_macro_derive(HelloMacro)]
pub fn hello_macro_derive(input: TokenStream) -> TokenStream {
    let ast = parse_macro_input!(input as DeriveInput);
    let name = &ast.ident;
    let gen = quote! {
        impl HelloMacro for #name {
            fn hello() {
                println!("Hello from {}!", stringify!(#name));
            }
        }
    };
    gen.into()
}

// 使用端
#[derive(HelloMacro)]
struct Pancakes;

fn main() {
    Pancakes::hello(); // 输出: Hello from Pancakes!
}
🌐 工程实践

实际案例：serde 的 #[derive(Serialize, Deserialize)]、tokio 的 #[tokio::main]、clap 的 #[derive(Parser)]。

开发过程宏：需要独立 crate（Cargo.toml 添加 proc-macro = true），依赖 syn + quote + proc-macro2。

✏️ 示例分析与易错点

注意：过程宏的错误信息通常很难读——syn 的解析错误会指向宏调用点而非宏定义内部。建议先写单元测试验证 token 转换逻辑。

⚡ 快速检测

05宏调试

🎯 为什么需要了解这个

宏出错时，错误信息往往指向宏调用处而非问题根源。掌握调试工具和技巧能大幅减少排错时间。

🔍 核心思路
  • cargo expand：展开所有宏，看到编译器实际编译的代码
  • trace_macros!：编译器内置（nightly），打印宏展开过程
  • 逐步简化：把宏调用简化到最小可复现 case
  • 编译错误定位：看错误的 span（源码位置），找到展开后的问题行
📐 规则与语法
BASHterminal
# 安装 cargo-expand
cargo install cargo-expand

# 展开当前 crate 的所有宏
cargo expand

# 只展开某个模块
cargo expand module_name

# 展开并格式化输出
cargo expand | rustfmt
RUSTdebug_macro.rs
macro_rules! debug_vec {
    ($($e:expr),*) => {{
        let mut v = Vec::new();
        $(
            // 调试技巧：展开时用 compile_error! 验证匹配
            // compile_error!(concat!("matched: ", stringify!($e)));
            v.push($e);
        )*
        v
    }};
}

fn main() {
    let v = debug_vec![1, 2, 3];
    println!("{:?}", v);
}
输出
[1, 2, 3]
🌐 工程实践

调试流程：① 看错误信息 → ② cargo expand 看展开结果 → ③ 把展开后的代码单独编译检查 → ④ 修复宏定义。

✏️ 示例分析与易错点

实用技巧：stringify!($expr) 把捕获的 token 变成字符串——可以在宏展开过程中打印调试信息。
