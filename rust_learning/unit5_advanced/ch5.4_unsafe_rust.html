Ch 5.4 Unsafe Rust

Rust 的安全保证来自编译器的严格检查。unsafe 不是"关闭安全"——它是告诉编译器"这部分我自己负责"。理解 unsafe 的边界，才能真正掌握 Rust 的全部能力。

01unsafe 的五种能力

🎯 为什么需要了解这个

很多人认为 "unsafe = 不安全"，但这是误解。unsafe 正确的理解是"编译器无法验证的安全性，由程序员承担证明责任"。

🔍 核心思路

unsafe 解锁且仅解锁以下 5 种操作：

  1. 解引用裸指针（*const T / *mut T）
  2. 调用 unsafe fn
  3. 访问或修改 static mut 变量
  4. 实现 unsafe trait
  5. 访问 union 字段

关键点：unsafe 不关闭借用检查、不关闭类型检查、不允许空指针取值。所有其他 Rust 规则照常。

📐 规则与语法
RUSTunsafe_basics.rs
fn main() {
    let mut x = 42;

    // 创建裸指针——不需要 unsafe
    let r1 = &x as *const i32;
    let r2 = &mut x as *mut i32;

    // 解引用裸指针——需要 unsafe
    unsafe {
        println!("r1 = {}", *r1);
        *r2 = 100;
        println!("r2 = {}", *r2);
    }
    println!("x = {}", x);
}
输出
r1 = 42
r2 = 100
x = 100
🌐 工程实践

最小化 unsafe 范围：unsafe 块尽量小，只包含必须的操作。这样审计时只需关注 unsafe 块。

注释说明安全性：在 unsafe 块上方写 // SAFETY: ... 注释说明为什么是安全的。

✏️ 示例分析与易错点

常见误解："unsafe 块里什么都可以做" → 错！unsafe 只解锁 5 种操作，其他规则不变。

02裸指针

🎯 为什么需要了解这个

裸指针是 unsafe 最常用的能力。FFI、底层数据结构（链表、树）、性能优化都可能需要裸指针。理解它和引用的区别至关重要。

🔍 核心思路

裸指针相比引用的"放松"：

  • 可以同时存在 *const T 和 *mut T 指向同一数据
  • 可以指向无效内存（悬垂指针）
  • 没有生命周期跟踪
  • 可以是空指针

创建是 safe 的，解引用才需要 unsafe。

📐 规则与语法
RUSTraw_pointers.rs
fn main() {
    let mut data = vec![1, 2, 3, 4, 5];

    // 从引用获取裸指针
    let ptr = data.as_mut_ptr();

    unsafe {
        // 指针算术
        let third = ptr.add(2);
        *third = 30;

        // 从裸指针重建切片
        let slice = std::slice::from_raw_parts(ptr, data.len());
        println!("{:?}", slice);
    }
}

// 空指针和有效性检查
fn check_null() {
    let p: *const i32 = std::ptr::null();
    println!("is null? {}", p.is_null());
    // unsafe { *p }  // ❌ UB! 解引用空指针
}
输出
[1, 2, 30, 4, 5]
🌐 工程实践

使用 NonNull<T>：如果你不需要空指针，用 NonNull<T> 代替裸指针——编译器可以利用"非空"信息优化。

ptr::read / ptr::write：操作未初始化内存时用这些而非普通解引用。

✏️ 示例分析与易错点

危险操作：指针算术越界、解引用已释放内存、对齐错误。Miri 可以检测这些问题。

⚡ 快速检测

03unsafe fn 与 unsafe trait

🎯 为什么需要了解这个

unsafe fn 和 unsafe trait 是 Rust 安全系统的"契约机制"——在类型层面表达"使用这个东西需要额外的安全保证"。

🔍 核心思路
  • unsafe fn：函数声明"调用者必须满足特定前置条件"。调用它必须在 unsafe 块中。
  • unsafe trait：trait 声明"实现它需要手动保证某些不变量"。unsafe impl 是实现者的承诺。
  • 经典例子：Send 和 Sync 是 unsafe trait——编译器自动推导大多数情况，但手动 impl 时你得保证线程安全。
📐 规则与语法
RUSTunsafe_fn_trait.rs
/// SAFETY: ptr 必须指向有效的、已初始化的 i32，
///         且在函数调用期间不被修改。
unsafe fn read_value(ptr: *const i32) -> i32 {
    *ptr  // 解引用裸指针
}

unsafe trait MySync {
    fn check(&self) -> bool;
}

struct SafeWrapper(i32);

// SAFETY: SafeWrapper 只包含 i32，天然线程安全
unsafe impl MySync for SafeWrapper {
    fn check(&self) -> bool { self.0 > 0 }
}

fn main() {
    let val = 42;
    let ptr = &val as *const i32;
    // 调用 unsafe fn 必须在 unsafe 块中
    let result = unsafe { read_value(ptr) };
    println!("read: {}", result);

    let w = SafeWrapper(10);
    println!("check: {}", w.check());
}
输出
read: 42
check: true
🌐 工程实践

SAFETY 注释：每个 unsafe fn 上方写 /// SAFETY: ... 说明调用者需要保证什么。每个 unsafe impl 写明实现者保证了什么不变量。

✏️ 示例分析与易错点

常见错误：在 unsafe fn 内部把所有代码都当作 unsafe 写——实际上只有确实需要 unsafe 的操作才应该在 unsafe 块中（Rust 2024 edition 可能默认要求）。

04安全抽象

🎯 为什么需要了解这个

标准库大量使用 unsafe——但你调用 Vec::push、String::from 时完全不需要 unsafe 块。秘密在于安全抽象：内部用 unsafe 实现高性能操作，对外暴露完全安全的 API。

🔍 核心思路

"安全边界"模式：

  1. 模块内部使用 unsafe 操作（如裸指针操作内存）
  2. 在 safe API 边界处验证所有前置条件
  3. 外部用户永远不需要写 unsafe

经典例子：Vec<T> 内部操作堆内存用裸指针，但 push、pop、[] 都是 safe 的。

📐 规则与语法
RUSTsafe_abstraction.rs
/// 安全版本的 split_at_mut 原理演示
fn my_split_at_mut(slice: &mut [i32], mid: usize) -> (&mut [i32], &mut [i32]) {
    let len = slice.len();
    assert!(mid <= len, "mid out of bounds");  // 安全边界：验证前置条件

    let ptr = slice.as_mut_ptr();
    unsafe {
        // SAFETY: mid <= len，两个切片不重叠
        (
            std::slice::from_raw_parts_mut(ptr, mid),
            std::slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}

fn main() {
    let mut data = vec![1, 2, 3, 4, 5];
    let (left, right) = my_split_at_mut(&mut data, 3);
    left[0] = 10;
    right[0] = 40;
    println!("left: {:?}", left);
    println!("right: {:?}", right);
}
输出
left: [10, 2, 3]
right: [40, 5]
🌐 工程实践

安全抽象是 Rust 的核心哲学：整个标准库就是这样建造的。你写的 crate 也应该遵循这个模式：unsafe 越少、越集中越好。

✏️ 示例分析与易错点

错误的安全抽象：如果 safe API 没有充分验证前置条件，unsafe 内部代码可能触发 UB。验证必须在 unsafe 块之前完成。

05FFI：与外部世界交互

🎯 为什么需要了解这个

Rust 不是孤岛。调用 C 库（如 OpenSSL、SQLite）、被 Python/Node.js 调用、编写操作系统内核——这些都需要 FFI。FFI 是 unsafe 的主要实际应用之一。

🔍 核心思路
  • 调用 C：extern "C" { fn abs(x: i32) -> i32; }
  • 被 C 调用：#[no_mangle] pub extern "C" fn my_func() {}
  • bindgen：自动从 C 头文件生成 Rust FFI 绑定
  • cbindgen：自动从 Rust 代码生成 C 头文件
📐 规则与语法
RUSTffi_example.rs
use std::os::raw::c_int;

// 声明 C 函数
extern "C" {
    fn abs(x: c_int) -> c_int;
}

// 导出给 C 使用
#[no_mangle]
pub extern "C" fn rust_add(a: c_int, b: c_int) -> c_int {
    a + b
}

fn main() {
    let result = unsafe { abs(-42) };
    println!("abs(-42) = {}", result);
}
输出
abs(-42) = 42
🌐 工程实践

-sys crate 惯例：Rust 生态中，foo-sys crate 提供 FFI 裸绑定，foo crate 提供安全抽象。例如 openssl-sys → openssl。

libc crate：提供 C 标准库类型的 Rust 映射（c_int、size_t 等）。

✏️ 示例分析与易错点

注意类型映射：C 的 int 在不同平台大小不同，始终用 c_int 而非 i32。字符串使用 CStr/CString 而非 &str/String。
