Ch 5.4 Unsafe Rust

unsafe 不是"关闭安全检查"——它是告诉编译器"这段代码的安全性由我负责"。理解 unsafe 解锁了什么、你需要维护哪些不变量、以及如何最小化 unsafe 的暴露面，是系统编程的核心能力。

🎯 学习目标

  • 明确 unsafe 块解锁的五种能力及其风险
  • 掌握裸指针（*const T / *mut T）的创建与解引用规则
  • 理解 unsafe fn 和 unsafe trait 的设计意图
  • 学会在 unsafe 之上构建安全的公共 API（Safe Abstraction）
  • 了解 FFI 基础：如何调用 C 函数和暴露 Rust 函数给 C

📋 前置知识

需要扎实理解所有权、借用规则和生命周期。unsafe 的核心挑战正是你手动保证那些编译器通常替你检查的不变量——如果对这些规则不清楚，写 unsafe 代码极易引入未定义行为。

5.4.1 unsafe 到底解锁了什么？

🤔

Rust 以安全著称，为什么还要提供 unsafe？它是不是意味着"放弃安全"？

unsafe 关键字让你使用五种在安全 Rust 中被禁止的操作：① 解引用裸指针；② 调用 unsafe 函数；③ 访问或修改可变静态变量；④ 实现 unsafe trait；⑤ 访问 union 的字段。这些操作并不是"坏的"——它们是构建底层抽象（如 Vec、Arc、操作系统接口）所必需的能力。

核心理解：unsafe 不关闭借用检查器，也不关闭类型检查。它只解锁上述五种操作，而你需要承诺：在这些操作中维护 Rust 安全模型的不变量。

Rustunsafe_five_powers.rs
fn main() {
    // ① 解引用裸指针
    let x = 42;
    let ptr = &x as *const i32;
    unsafe {
        println!("裸指针值: {}", *ptr);
    }

    // ③ 访问可变静态变量
    static mut COUNTER: i32 = 0;
    unsafe {
        COUNTER += 1;
        println!("静态计数器: {}", COUNTER);
    }
}
输出 
裸指针值: 42
静态计数器: 1

5.4.2 裸指针：创建安全，解引用危险

🎯

裸指针和引用有什么区别？为什么创建裸指针不需要 unsafe，但解引用需要？

裸指针（*const T 和 *mut T）不受借用规则约束——你可以同时拥有可变和不可变的裸指针，指针也可以指向无效内存。创建裸指针本身是安全的（只是一个数字），但解引用时可能触发未定义行为，所以必须在 unsafe 中进行。

裸指针 vs 引用：引用有借用规则保护，保证总是有效；裸指针没有这些保证。裸指针的典型用途是 FFI、手写数据结构、以及绕过借用检查器的极少数正当场景。

Rustraw_pointers.rs
fn main() {
    let mut value = 10;

    // 创建裸指针——安全操作
    let r1 = &value as *const i32;
    let r2 = &mut value as *mut i32;

    // 解引用裸指针——必须在 unsafe 中
    unsafe {
        println!("r1 = {}", *r1);
        *r2 = 20;
        println!("r2 修改后 = {}", *r2);
    }

    // 可以从任意地址创建裸指针（但解引用是 UB）
    let bad_ptr = 0x12345usize as *const i32;
    println!("裸指针地址: {:?}", bad_ptr);
    // unsafe { *bad_ptr } // 千万不要这样做！
}
输出 
r1 = 10
r2 修改后 = 20
裸指针地址: 0x12345

裸指针本质上就是一个内存地址（一个数字）。创建它不会读写任何内存，所以完全安全。危险发生在解引用时——你在告诉程序"去这个地址读/写数据"，而编译器无法保证那个地址有效。这就是解引用需要 unsafe 的原因。

5.4.3 unsafe 函数与 unsafe Trait

📜

为什么有些函数声明为 unsafe fn？调用者需要承担什么责任？

unsafe fn 的意思是"调用这个函数时，你必须保证满足某些前置条件——编译器无法替你检查"。类似地，unsafe trait 表示"实现这个 trait 时，你必须手动保证某些不变量"。Send 和 Sync 就是 unsafe trait——编译器自动推导时是安全的，但如果你要手动 impl，就需要保证线程安全。

不变量责任：当函数标记为 unsafe fn，它的文档应该清楚说明调用者需要保证哪些前置条件。这些条件通常包括：指针有效性、对齐要求、不会出现数据竞争等。

Rustunsafe_fn_trait.rs
/// # Safety
/// `ptr` 必须指向一个有效的、已初始化的 i32 值
unsafe fn read_value(ptr: *const i32) -> i32 {
    *ptr
}

// unsafe trait：实现者保证线程安全
unsafe trait ThreadSafe {}

struct MyType(i32);
// 手动实现 unsafe trait——我们保证 MyType 线程安全
unsafe impl ThreadSafe for MyType {}

fn main() {
    let x = 100;
    let ptr = &x as *const i32;

    // 调用 unsafe fn 需要 unsafe 块
    let val = unsafe { read_value(ptr) };
    println!("读取值: {val}");
}
输出 
读取值: 100

5.4.4 安全抽象：把 unsafe 包装成安全的 API

🛡️

标准库的 Vec、String、Arc 内部都用了 unsafe——为什么用户使用它们时不需要写 unsafe？

这就是"安全抽象"（Safe Abstraction）的核心思想：内部使用 unsafe 来实现高效操作，但通过精心设计的公共 API 保证外部用户无论如何调用都不会触发未定义行为。unsafe 的表面积被最小化到内部实现中，对外暴露的全是安全接口。

设计原则：① 把 unsafe 代码集中在最小范围内；② 用类型系统和 API 设计阻止外部违反不变量；③ 为每个 unsafe 块写注释说明为什么它是安全的；④ 用测试和 Miri 验证正确性。

Rustsafe_abstraction.rs
/// 一个安全的"分割可变切片"函数
/// 内部使用 unsafe，但 API 对外完全安全
fn split_at_mut(slice: &mut [i32], mid: usize) -> (&mut [i32], &mut [i32]) {
    let len = slice.len();
    assert!(mid <= len, "索引越界");

    let ptr = slice.as_mut_ptr();
    unsafe {
        // Safety: mid <= len 已检查，两个切片不重叠
        (
            std::slice::from_raw_parts_mut(ptr, mid),
            std::slice::from_raw_parts_mut(ptr.add(mid), len - mid),
        )
    }
}

fn main() {
    let mut data = vec![1, 2, 3, 4, 5];
    let (left, right) = split_at_mut(&mut data, 3);

    left[0] = 10;
    right[0] = 40;

    println!("left: {:?}", left);
    println!("right: {:?}", right);
}
输出 
left: [10, 2, 3]
right: [40, 5]

Miri 是 Rust 的实验性解释器，能在运行时检测出许多未定义行为（内存越界、使用未初始化值、违反借用规则等）。使用方式：rustup +nightly component add miri 然后 cargo +nightly miri test。在写包含 unsafe 的库时，用 Miri 跑测试是最佳实践。

5.4.5 FFI 基础：Rust 与 C 的桥梁

🌉

Rust 程序如何调用一个现有的 C 库函数？C 程序又如何调用 Rust 写的函数？

FFI（Foreign Function Interface）让 Rust 调用其他语言的函数，或把 Rust 函数暴露给其他语言。调用外部函数用 extern "C" 块声明；暴露 Rust 函数用 #[no_mangle] + extern "C" fn。所有 FFI 调用都是 unsafe 的——编译器无法验证外部代码的正确性。

FFI 要点：① 使用 extern "C" 指定 C ABI；② 类型必须兼容（用 libc crate 获取 C 类型）；③ 字符串需要用 CString / CStr 转换；④ 所有外部函数调用都在 unsafe 中。

Rustffi_basics.rs
// 声明外部 C 函数
extern "C" {
    fn abs(input: i32) -> i32;
    fn sqrt(input: f64) -> f64;
}

// 暴露 Rust 函数给 C
#[no_mangle]
pub extern "C" fn rust_add(a: i32, b: i32) -> i32 {
    a + b
}

fn main() {
    // 调用 C 标准库函数——需要 unsafe
    unsafe {
        println!("abs(-5) = {}", abs(-5));
        println!("sqrt(9.0) = {}", sqrt(9.0));
    }

    // Rust 函数也可以直接调用
    println!("rust_add(3, 4) = {}", rust_add(3, 4));
}
输出 
abs(-5) = 5
sqrt(9.0) = 3
rust_add(3, 4) = 7

Rust 编译器只能验证 Rust 代码的安全性。对于通过 FFI 调用的外部函数，编译器无法检查：参数是否有效、返回值是否正确、函数是否有副作用或内存安全问题。因此，所有 FFI 调用的安全性都由程序员负责，必须用 unsafe 标记。

为什么所有 FFI 函数调用都需要 unsafe？

📌 章节总结

  • unsafe 解锁五种操作：裸指针解引用、unsafe 函数、可变静态变量、unsafe trait、union 字段。
  • 裸指针创建是安全的，解引用是 unsafe 的——因为编译器无法保证地址有效。
  • unsafe fn 要求调用者保证前置条件；unsafe trait 要求实现者保证不变量。
  • 安全抽象是最佳实践：内部 unsafe 最小化，外部 API 完全安全。
  • FFI 是 unsafe 的典型场景——编译器无法验证外部代码的正确性。
