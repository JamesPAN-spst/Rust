字符串

Rust 的字符串设计围绕两个核心约束展开：UTF-8 正确性 和 所有权安全。理解 String 与 &str 的二元关系是高效处理文本数据的基础。

01String 与 &str

🎯 为什么需要了解这个

几乎每种语言都有字符串类型，但 Rust 把字符串分成了两个核心类型——String 和 &str。初学者经常困惑：为什么不能只用一种？

原因在于 Rust 的所有权系统。拥有数据和借用数据是两个根本不同的概念，分别对应 String 和 &str。理解这个区分，后续的 API 选择、函数签名设计才能水到渠成。

🔍 核心思路

String 是一个拥有所有权的、堆分配的、可增长的 UTF-8 字符串。内部结构等价于 Vec<u8>，由三部分组成：

  • ptr：指向堆内存的指针
  • len：当前已使用的字节数
  • cap：已分配的总容量

&str 是一个字符串切片引用（胖指针），包含指针和长度两个字段。它不拥有数据，只是对某段 UTF-8 字节的借用视图。

类比：String 就像你拥有的一本笔记本（可以写入、撕页、扩充），&str 就像别人借给你看的一页复印件（只读、不可修改大小）。

📐 规则与语法
RUSTstring_basics.rs
// 创建 String 的三种常见方式
let s1 = String::new();                    // 空字符串
let s2 = String::from("hello");            // 从字面值创建
let s3 = "hello".to_string();              // 通过 Display trait

// &str —— 字符串字面值默认就是 &'static str
let greeting: &str = "hello, world";

// String → &str：通过 Deref coercion 自动转换
let s: String = String::from("rust");
let slice: &str = &s;                      // 自动解引用
let slice2: &str = s.as_str();             // 显式方法

// &str → String：需要分配堆内存
let owned: String = slice.to_owned();
let owned2: String = String::from(slice);

🌐 工程实践
  • 函数参数：优先用 &str，因为它同时接受 &String 和字符串字面值
  • 结构体字段：需要拥有数据时用 String，临时引用用 &'a str（需生命周期标注）
  • 返回值：如果返回新创建的字符串用 String，返回输入的切片用 &str
  • String::with_capacity(n)：当已知大致长度时预分配，避免多次 realloc
✏️ 示例分析与易错点
RUSTcommon_mistake.rs
// ❌ 常见错误：把 &str 当作 String 使用
fn greeting_bad() -> &str {   // 编译错误！返回的引用指向谁？
    let s = String::from("hello");
    &s                         // s 在函数结束时 drop，引用悬垂
}

// ✅ 正确做法 1：返回 String（转移所有权）
fn greeting_owned() -> String {
    String::from("hello")      // 所有权转移给调用者
}

// ✅ 正确做法 2：返回 &'static str
fn greeting_static() -> &'static str {
    "hello"                    // 字面值嵌入二进制，永不释放
}

⚠ 核心易错点：str（无引用）是一个 unsized 类型，不能直接用作变量类型或函数参数。你几乎总是以 &str 的形式使用它。

02字符串操作

🎯 为什么需要了解这个

构建字符串是日常编程中最频繁的操作之一。Rust 提供了多种方式来拼接和修改字符串，但每种方式在所有权和性能上有不同的影响。选错方式会导致不必要的 clone、编译错误，甚至性能瓶颈。

🔍 核心思路

字符串拼接的三种主要方式各有特点：

方式所有权影响适用场景
push_str / push需要 &mut String逐步构建字符串
+ 运算符消耗左侧 String简单的二元拼接
format!只借用参数复杂格式化、保留所有权
📐 规则与语法
RUSTstring_ops.rs
// === push_str / push ===
let mut s = String::from("hello");
s.push_str(", world");    // 追加 &str
s.push('!');              // 追加单个 char
// s = "hello, world!"

// === + 运算符 ===
// 签名：fn add(self, s: &str) -> String
let s1 = String::from("hello");
let s2 = String::from(" world");
let s3 = s1 + &s2;       // s1 被移动！s2 被借用
// println!("{}", s1);    // ❌ s1 已经无效
println!("{}", s3);       // ✅ "hello world"

// === format! 宏 ===
let name = String::from("Rust");
let version = String::from("2024");
let msg = format!("{} edition {}", name, version);
// name 和 version 依然可用！format! 只借用
println!("{}", name);     // ✅ 还能用

容量增长策略：

RUSTcapacity.rs
let mut s = String::with_capacity(10);
println!("len={}, cap={}", s.len(), s.capacity());
// len=0, cap=10

s.push_str("hello");
println!("len={}, cap={}", s.len(), s.capacity());
// len=5, cap=10  —— 容量充足，无重分配

s.push_str(" world!!");
println!("len={}, cap={}", s.len(), s.capacity());
// len=13, cap=20 —— 超出容量，自动翻倍扩容
输出
len=0, cap=10
len=5, cap=10
len=13, cap=20
🌐 工程实践
  • 循环拼接：用 String::with_capacity 预分配 + push_str，避免反复 realloc
  • 多段拼接：format! 最清晰，编译器也能优化得不错
  • 大量字符串拼接：考虑先 collect 到 Vec<&str> 后 .join("")
  • 避免 + 链式调用：s1 + &s2 + &s3 虽合法但可读性差，且 s1 被消耗
RUSTjoin_pattern.rs
// 高效拼接多段字符串
let parts = vec!["hello", "beautiful", "world"];
let sentence = parts.join(" ");
// "hello beautiful world"

// 构建 CSV 行
let fields = vec!["name", "age", "city"];
let csv_line = fields.join(",");
// "name,age,city"
✏️ 示例分析与易错点
RUSTplus_trap.rs
let s1 = String::from("a");
let s2 = String::from("b");
let s3 = String::from("c");

// ❌ 这看起来对称，但实际上：
// s1 被移动，s2 被借用，临时结果再被移动，s3 被借用
let result = s1 + &s2 + &s3;

// ✅ 更清晰的替代方案
let s1 = String::from("a");
let result = format!("{}{}{}", s1, s2, s3);
// 所有变量都还能继续使用

⚠ + 运算符的左侧必须是 String，右侧必须是 &str。不能写 &str + &str 或 &str + String。

03字符串索引与切片

🎯 为什么需要了解这个

在 Python 或 JavaScript 中，s[0] 是获取第一个字符的标准方式。但在 Rust 中，s[0] 根本无法编译。这不是 Rust 的缺陷——而是对 Unicode 正确性的严肃对待。理解这个限制背后的原因，能帮你避免多语言文本处理中的经典 bug。

🔍 核心思路

一个"字符"在不同视角下有三种理解：

  • 字节（bytes）：UTF-8 编码的原始字节，每个固定 1 字节
  • Unicode 标量值（chars）：Rust 的 char 类型，UTF-8 编码下每个占 1-4 字节
  • 字形簇（grapheme clusters）：人类视觉上的"一个字符"，可能由多个标量值组成

Rust 拒绝用 s[i] 索引的根本原因：不清楚用户想要哪个层次，且按字节索引可能切到 UTF-8 多字节字符的中间，产生无效数据。

📐 规则与语法
RUSTindexing.rs
let hello = String::from("你好世界");

// ❌ 不可以直接索引
// let c = hello[0];  // 编译错误！String 没有实现 Index<usize>

// ✅ 按字节切片（必须在字符边界上）
let first_char = &hello[0..3];  // "你"（一个汉字 = 3 字节）
// let bad = &hello[0..2];      // ⚠ panic! 不在字符边界

// ✅ 按字符遍历
for ch in hello.chars() {
    print!("{} ", ch);  // 你 好 世 界
}

// ✅ 按字节遍历
for b in hello.bytes() {
    print!("{:02x} ", b);
    // e4 bd a0 e5 a5 bd e4 b8 96 e7 95 8c
}

// ✅ 获取第 n 个字符（注意：O(n) 复杂度）
let second = hello.chars().nth(1);  // Some('好')
输出
你 好 世 界
e4 bd a0 e5 a5 bd e4 b8 96 e7 95 8c
RUSTstring_methods.rs
let text = "Hello, 世界!";

// 各种度量
println!("字节数: {}", text.len());            // 13
println!("字符数: {}", text.chars().count());   // 9

// 常用方法
let trimmed = "  hello  ".trim();              // "hello"
let upper = "rust".to_uppercase();             // "RUST"
let replaced = "foo bar".replace("foo", "baz"); // "baz bar"

// 分割
let parts: Vec<&str> = "a,b,c".split(',').collect();
// ["a", "b", "c"]

// 检查
let starts = "hello".starts_with("hel");       // true
let contains = "hello".contains("ell");        // true
🌐 工程实践
  • 绝不假设 "1 字符 = 1 字节"：处理用户输入时始终用 .chars() 或专门的 Unicode 库
  • 范围切片要小心：&s[a..b] 中 a 和 b 必须是字符边界，否则 panic。用 .char_indices() 获取安全的切割位置
  • 性能敏感场景：如果确定只处理 ASCII，可用 .as_bytes() 直接操作字节切片，O(1) 索引
  • 字形簇处理：标准库不直接支持，需引入 unicode-segmentation crate
RUSTsafe_slice.rs
/// 安全地截取前 n 个字符
fn take_chars(s: &str, n: usize) -> &str {
    match s.char_indices().nth(n) {
        Some((idx, _)) => &s[..idx],
        None => s,  // 字符串不够长，返回全部
    }
}

let text = "你好世界";
let first_two = take_chars(text, 2);
assert_eq!(first_two, "你好");
✏️ 示例分析与易错点
RUSTchar_indices.rs
let s = "café";

// 使用 char_indices 安全获取位置
for (idx, ch) in s.char_indices() {
    println!("byte {} => '{}'", idx, ch);
}
// byte 0 => 'c'
// byte 1 => 'a'
// byte 2 => 'f'
// byte 3 => 'é'  (U+00E9 占 2 个 UTF-8 字节)

println!("len = {}", s.len());           // 5 字节
println!("chars = {}", s.chars().count()); // 4 字符
输出
byte 0 => 'c'
byte 1 => 'a'
byte 2 => 'f'
byte 3 => 'é'
len = 5
chars = 4

⚠ .chars().nth(n) 每次调用都从头遍历，时间复杂度 O(n)。如果需要频繁随机访问字符，先 .chars().collect::<Vec<char>>() 转为字符数组。

04字符串在所有权系统中的行为

🎯 为什么需要了解这个

字符串是最常在函数间传递的数据类型。每次传递都涉及一个选择：移动？借用？还是克隆？ 选错会导致编译错误或不必要的性能开销。掌握字符串的所有权模式，是写出高效 Rust 代码的关键一步。

🔍 核心思路

String 遵循标准的所有权规则——不实现 Copy（因为堆分配数据不能隐式浅拷贝），赋值和传参默认移动所有权。

核心决策树：

  • 只读访问 → 用 &str 借用
  • 需要修改 → 用 &mut String 可变借用
  • 需要独立拷贝 → 用 .clone() 或 .to_owned()
  • 交出所有权 → 直接传 String（移动语义）
📐 规则与语法
RUSTownership_patterns.rs
// === 移动语义 ===
let s1 = String::from("hello");
let s2 = s1;           // s1 → s2，s1 无效
// println!("{}", s1);  // ❌ 编译错误

// === Clone：深拷贝 ===
let s1 = String::from("hello");
let s2 = s1.clone();   // 分配新堆内存，复制全部字节
println!("{} {}", s1, s2);  // ✅ 两者独立

// === 函数参数的四种模式 ===

// 模式 1：借用只读（最常用）
fn print_len(s: &str) {
    println!("长度: {}", s.len());
}

// 模式 2：可变借用
fn append_bang(s: &mut String) {
    s.push('!');
}

// 模式 3：获取所有权
fn consume(s: String) {
    println!("consumed: {}", s);
}  // s 在这里被 drop

// 模式 4：获取所有权并返回新值
fn to_upper(s: String) -> String {
    s.to_uppercase()  // 所有权转移到返回值
}
RUSTstruct_ownership.rs
// 结构体持有 String：拥有所有权，无需生命周期
struct User {
    name: String,
    email: String,
}

// 结构体持有 &str：需要生命周期标注
struct UserRef<'a> {
    name: &'a str,
    email: &'a str,
}

// 实际使用
let user = User {
    name: String::from("Alice"),
    email: String::from("alice@example.com"),
};  // User 拥有两个字符串，可以安心存储

let name = String::from("Bob");
let user_ref = UserRef {
    name: &name,       // 借用，UserRef 不能活得比 name 久
    email: "bob@x.com", // &'static str，没问题
};
🌐 工程实践

参数类型选择指南：

场景推荐类型理由
只读函数参数&str最通用，接受 &String 和字面值
需要修改字符串&mut String原地修改，不分配新内存
存入结构体或容器String结构体需要拥有数据
灵活 API 入口impl Into<String>接受 String 和 &str，由调用者决定
RUSTinto_pattern.rs
// 灵活的 API 设计：接受 String 或 &str
struct Config {
    name: String,
}

impl Config {
    fn new(name: impl Into<String>) -> Self {
        Config { name: name.into() }
    }
}

// 两种调用方式都行
let c1 = Config::new("literal");              // &str → String
let c2 = Config::new(String::from("owned"));  // String 直接用
✏️ 示例分析与易错点
RUSTclone_vs_ref.rs
// ❌ 不必要的 clone——性能浪费
fn process_bad(data: &[String]) {
    for s in data {
        let copy = s.clone();  // 每次循环都深拷贝！
        println!("{}", copy.to_uppercase());
    }
}

// ✅ 只需要引用就够了
fn process_good(data: &[String]) {
    for s in data {
        println!("{}", s.to_uppercase());
        // to_uppercase 返回新 String，无需先 clone
    }
}

// ❌ 参数过于狭窄
fn first_word_bad(s: &String) -> &str {
    // 可以工作，但参数应该用 &str
    &s[..s.find(' ').unwrap_or(s.len())]
}

// ✅ 更好的签名
fn first_word(s: &str) -> &str {
    &s[..s.find(' ').unwrap_or(s.len())]
}

⚠ 经验法则：如果发现自己频繁写 .clone()，先停下来思考——是否可以用引用代替？不必要的 clone 是 Rust 新手最常见的性能陷阱。
