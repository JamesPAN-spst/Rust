Unit 3 · 标准库与项目实战

Ch 3.5 模块系统

这一章把模块系统从“文件怎么拆”提升到“边界怎么设计”：crate 和 module 的关系、`pub/use/crate/super` 到底在表达什么、何时应该重导出 API，以及项目目录应该如何按职责组织。

🎯 学习目标

  • 理解 crate、module、文件结构与命名空间之间的关系
  • 掌握 `pub`、`use`、`crate`、`super`、`self` 的基本语义
  • 学会通过 `pub use` 组织更友好的对外 API 入口
  • 建立按职责划分项目模块的工程直觉，避免无意义拆分

📋 前置知识

建议先掌握函数、struct、enum 和错误处理基础。模块系统本质上是“组织已有代码”，如果还没有基本构造块，就很难判断边界该落在哪里。

3.5.1 模块在解决什么问题：不是“多几个文件”，而是边界与命名空间

🏗️

项目一变大，如果所有函数都堆在 main.rs 里，读者要怎么知道哪些 API 是公开边界、哪些只是内部细节？

模块系统的核心作用是组织代码、隔离命名空间、表达可见性边界。它既帮助编译器定位文件，也帮助团队判断“哪些东西是稳定接口，哪些只是实现细节”。

正式规则：crate 是编译单元，module 是 crate 内部的命名空间与组织边界；文件只是模块的一种常见落地方式。

Rustmain.rs
mod parser {
    pub fn parse() {
        println!("parse module ready");
    }
}

mod report {
    pub fn render() {
        println!("report module ready");
    }
}

fn main() {
    parser::parse();
    report::render();
}
输出 
parse module ready
report module ready

3.5.2 `pub`、`crate`、`super`、`self`：路径是在表达“从哪里看这个名字”

🧭

同一个函数名，为什么有时写 `crate::parser::parse`，有时却写 `super::parse`？

路径不是语法负担，而是在描述当前视角：crate:: 从 crate 根出发，super:: 指向父模块，self:: 指向当前模块，pub 则决定外部是否看得到该项。

正式规则：默认一切私有；只有显式标记 pub 的项，才可能被模块边界之外访问。

Rustmodule_paths.rs
mod parser {
    pub fn parse_line() {
        println!("parse");
    }
}

fn main() {
    crate::parser::parse_line();
}
输出 
parse
默认私有能迫使你显式声明公开 API，减少“内部实现被外部偷偷依赖”的风险。模块边界因此更稳定，也更容易重构。

3.5.3 `use` 和重导出：让调用点更干净，让对外 API 更稳定

🪄

如果每次都写很长的全路径，代码会不会很吵？但如果随便 import，又会不会让边界变得模糊？

use 用于把常用路径引入当前作用域，减少重复；而 pub use 可以把内部层级重新整理成更友好的对外接口。

正式规则：use 只影响当前作用域内的名字书写方式，不改变原始模块结构；pub use 才会改变 crate 对外暴露的入口。

Rustpub_use.rs
mod parser {
    pub struct Config;
}

pub use parser::Config;

fn main() {
    let _cfg = Config;
    println!("re-exported");
}
输出 
re-exported

3.5.4 项目组织：按职责拆分，而不是按语法类别拆分

🗃️

模块到底该按“struct 放一组、函数放一组”来拆，还是按业务职责来拆？

更实用的方式通常是按职责或领域拆分：例如 config、parser、report、cli。这样文件结构能直接反映数据流和边界，而不是把相关逻辑打散到多个“语法类型桶”里。

正式规则：模块划分优先服务读者理解和依赖方向；如果两个函数总是围绕同一职责修改，通常应先考虑放在同一模块。

Rustproject_layout.rs
fn main() {
    let layout = ["config", "parser", "report", "main"];
    println!("modules = {}", layout.join(", "));
}
输出 
modules = config, parser, report, main

📌 章节总结

  • 模块系统首先是边界与命名空间工具，而不是单纯的文件拆分技巧。
  • 默认私有、显式 pub 能帮助你维护稳定的公开接口。
  • `use` 改善当前作用域可读性，`pub use` 则用于对外重导出更友好的入口。
  • 项目结构优先按职责或领域划分，让依赖方向和数据流更清晰。
