Ch 3.6 实战项目

这一章把 Unit 3 的能力串成一个迷你项目：日志统计小工具。目标不是做复杂功能，而是把模块拆分、字符串解析、Result 边界、集合计数和迭代器数据流真正拼成一个可落地的工程心智模型。

🎯 学习目标

  • 学会为一个小型 Rust 工具先定义输入、输出和错误边界
  • 掌握按职责拆分 reader / parser / report / main 的基本方式
  • 把字符串、Result、集合与迭代器组合成完整数据处理流程
  • 为进入下一单元前的真实工程练习建立清晰的实现路线

📋 前置知识

建议先完成 Unit 3 前五章。实战章默认你已经理解字符串借用、集合模式、迭代器链、错误传播和模块边界，否则很难把它们自然拼接起来。

3.6.1 先定义项目目标：做一个日志统计小工具

🎯

如果 Unit 3 的知识点都学过了，怎样把它们组合成一个“真的像项目”的练习？

一个很合适的迷你项目是“日志统计器”：读取文本行，解析日志级别和模块名，统计错误数量，再输出摘要报告。这个流程天然会用到字符串、集合、迭代器、错误处理和模块拆分。

正式规则：在实战章里，先定义输入、输出和错误边界，再决定代码如何拆模块；不要一上来先写工具函数。

Rustgoal.rs
fn main() {
    println!("goal: parse logs, count levels, print summary");
}
输出

3.6.2 模块拆分：把“读取、解析、统计、输出”拆成职责边界

🧩

一个小工具也需要模块吗？会不会太“工程化”？

恰恰相反。哪怕项目很小，只要同时存在输入处理、领域解析和结果输出，就值得拆模块。这样错误传播路径更清楚，后续加测试也更容易。

推荐拆分：reader 负责读入原始行，parser 负责把文本转成结构体，report 负责汇总输出，main 只负责组装流程。

Rustlayout.rs
mod reader {
    pub fn read_lines() {
        println!("read lines");
    }
}

mod parser {
    pub fn parse() {
        println!("parse lines");
    }
}

mod report {
    pub fn render() {
        println!("render report");
    }
}

fn main() {
    reader::read_lines();
    parser::parse();
    report::render();
}
输出 
read lines
parse lines
render report
因为 main 最适合表达应用装配顺序：读输入、调用解析、生成报告、打印错误。把细节塞进去会让流程和实现耦在一起，不利于测试和重构。

3.6.3 解析层：让 `Result` 成为输入不可信时的第一道边界

🧪

日志行格式不合法时，你想直接 panic，还是把错误交给调用方统计/跳过/展示？

解析层最适合返回 Result<LogEntry, ParseError>。这样你既能在上层决定“遇到坏行就终止”，也能选择“记录错误并继续处理其余输入”。

正式规则：只要输入来自用户、文件或网络，就默认它不可信；解析函数应该优先返回 Result 而不是 panic。

Rustparse_line.rs
#[derive(Debug)]
struct LogEntry {
    level: String,
    module: String,
}

fn parse_line(line: &str) -> Result<LogEntry, &'static str> {
    let mut parts = line.split(':');
    let level = parts.next().ok_or("missing level")?;
    let module = parts.next().ok_or("missing module")?;

    Ok(LogEntry {
        level: level.to_string(),
        module: module.to_string(),
    })
}

fn main() {
    println!("{:?}", parse_line("ERROR:auth"));
}
输出 
Ok(LogEntry { level: "ERROR", module: "auth" })

3.6.4 汇总层：用集合和迭代器把明细变成报告

📊

一旦解析出结构化日志，怎样把“很多条记录”变成可读摘要？

这正是集合和迭代器的主场：先过滤感兴趣的记录，再用 `HashMap` 计数，用 `Vec` 排序输出，最后格式化成展示字符串。数据流会从“行文本”逐步上升为“统计结果”。

正式规则：实战里最常见的流程是“读取 → 解析 Result → 过滤/映射 → 计数/分组 → 输出”，Unit 3 的核心能力几乎都串在这条线上。

Rustreport.rs
use std::collections::HashMap;

fn main() {
    let modules = ["auth", "db", "auth"];
    let mut counts = HashMap::new();

    for module in modules {
        *counts.entry(module).or_insert(0) += 1;
    }

    println!("auth => {}", counts["auth"]);
}
输出 
auth => 2

📌 章节总结

  • 先定义项目的输入、输出和失败边界，再决定代码结构，会比“先写函数再拼”稳定得多。
  • reader / parser / report / main 的职责划分能让小工具也保持可读性和可测试性。
  • 解析层优先返回 Result，把不可信输入留给上层策略处理。
  • 集合和迭代器负责把原始明细变成统计结果，模块系统则负责把流程组织清楚。
