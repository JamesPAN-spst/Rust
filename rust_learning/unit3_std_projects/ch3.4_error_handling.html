Ch 3.4 错误处理

这一章把 Rust 的错误处理拆成四个层次：什么时候应该 panic、什么时候返回 Result、何时只需要 Option，以及如何用 `?` 和组合器把错误传播、转换和恢复写得清楚而不混乱。

🎯 学习目标

  • 区分 panic、Result、Option 分别适合什么失败语义
  • 掌握 `?` 运算符的传播机制和使用前提
  • 学会用 `map_err`、`unwrap_or_else`、`match` 组织恢复逻辑
  • 避免把可恢复错误写成 panic，或把缺失值误建模成复杂错误类型

📋 前置知识

建议先熟悉 enum、模式匹配和基础函数签名。错误处理本质上是把失败纳入类型系统，因此如果还没习惯“返回类型表达分支”，会容易只想靠 panic 解决问题。

3.4.1 `panic!` vs `Result`：是程序无法继续，还是调用方可以决定？

🚨

同样是“出错了”，为什么 Rust 要把 panic 和 Result 明确区分开？

panic! 表示程序进入了无法继续的状态，通常用于违反内部不变量；Result<T, E> 表示失败是预期的一部分，调用方应该有机会选择恢复、重试或上报。

正式规则：边界输入、文件读取、网络请求等可预期失败优先建模为 Result；只有真正的不可恢复错误才应考虑 panic。

Rustresult_vs_panic.rs
fn divide(total: i32, count: i32) -> Result<i32, &'static str> {
    if count == 0 {
        return Err("count must not be zero");
    }
    Ok(total / count)
}

fn main() {
    println!("{:?}", divide(10, 2));
    println!("{:?}", divide(10, 0));
}
输出 
Ok(5)
Err("count must not be zero")

3.4.2 `Option`：不是所有“拿不到值”都该当成错误

❓

查找集合元素时没找到，是失败，还是一种正常分支？

Option<T> 用于“值可能存在，也可能不存在”的场景，例如可选配置、集合查找、用户可能没填字段。它不需要额外错误信息，因为“没有值”本身就是全部语义。

正式规则：当缺失本身就是合法业务状态时，用 Option；当失败原因需要传递给调用方时，用 Result。

Rustoption_lookup.rs
fn main() {
    let ports = vec![3000, 4000, 5000];
    let found = ports.iter().find(|&&port| port == 4000);

    println!("{:?}", found);
}
输出 
Some(4000)
可以。常见做法是用 `ok_or` / `ok_or_else` 把 `Option` 转成 `Result`，在“缺失值现在要被视为错误”时补上错误信息。

3.4.3 `?` 运算符：让错误传播变成显式但不啰嗦

⛓️

如果每一步都手写 match 处理错误，业务逻辑会不会被样板代码淹没？

? 的作用是：成功时继续解包值，失败时立刻把错误返回给外层调用者。它没有隐藏控制流，只是把常见的“早返回”写法浓缩成了一个符号。

正式规则：? 只能用于返回兼容 Result / Option 的函数或闭包中，且错误类型必须可转换。

Rustquestion_mark.rs
fn parse_port(text: &str) -> Result<u16, std::num::ParseIntError> {
    let port = text.parse::<u16>()?;
    Ok(port)
}

fn main() {
    println!("{:?}", parse_port("8080"));
    println!("{:?}", parse_port("abc"));
}
输出 
Ok(8080)
Err(ParseIntError { kind: InvalidDigit })

3.4.4 组合式处理：`map_err`、`unwrap_or_else` 和模式匹配

🧰

传播错误很重要，但有时候你也想补充上下文、提供默认值或只处理其中一部分分支，怎么办？

Rust 的错误处理不只有 `match`。你可以用 map_err 改写错误信息，用 unwrap_or_else 提供延迟计算的默认值，也可以继续回到 match 写最清楚的分支逻辑。

正式规则：优先选择最能表达意图的形式：简单转换用组合器，复杂分支用 match，不要把所有错误处理都写成链式“魔法”。

Rustmap_err.rs
fn load_retry_limit(text: &str) -> Result<u8, String> {
    text.parse::<u8>()
        .map_err(|_| format!("invalid retry limit: {text}"))
}

fn main() {
    println!("{:?}", load_retry_limit("3"));
    println!("{:?}", load_retry_limit("many"));
}
输出 
Ok(3)
Err("invalid retry limit: many")

📌 章节总结

  • panic 适合不可恢复的内部错误，Result 适合需要交给调用方处理的失败。
  • Option 表达“可能没有值”，不必硬造错误类型。
  • `?` 让常见的早返回传播变得简洁，但没有隐藏控制流。
  • 组合器和 match 都是工具；选择最能表达当前错误语义的写法即可。
