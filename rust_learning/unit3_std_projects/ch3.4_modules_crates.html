Unit 3 · 标准库与项目实践

模块与 Crate

Rust 的模块系统通过 mod、pub、use 三个关键字管理代码的组织和可见性。Cargo 作为构建工具和包管理器，将模块系统与外部生态无缝连接。掌握这套机制是编写可维护大型项目的基础。

01mod 与文件结构

🎯 为什么需要了解这个

随着项目增长，把所有代码放在一个文件里会变得极难维护。Rust 的模块系统让你把代码拆分到不同文件和目录，同时保持清晰的层级关系。关键是理解模块树和文件系统的映射规则。

🔍 核心思路

Rust 的模块系统有三个核心概念：

  • Package：一个 Cargo.toml 描述的项目，包含一个或多个 crate
  • Crate：编译单元（lib 或 bin），有一个根文件（crate root）
  • Module：crate 内部的代码组织单元，形成树形结构

mod 关键字有两种用法：

  • 内联定义：mod name { ... } 直接在当前文件中定义
  • 文件声明：mod name; 告诉编译器去查找同名文件
📐 规则与语法
TEXTproject_structure
my_project/
├── Cargo.toml
└── src/
    ├── main.rs          ← bin crate root
    ├── lib.rs           ← lib crate root（可选）
    ├── config.rs        ← mod config;
    ├── db/
    │   ├── mod.rs       ← mod db;（传统方式）
    │   └── pool.rs      ← mod pool; （在 db/mod.rs 中声明）
    └── handlers.rs      ← mod handlers;

2018 Edition 扁平化风格（推荐）：

TEXTflat_structure
my_project/
├── Cargo.toml
└── src/
    ├── main.rs
    ├── db.rs            ← mod db; 的内容（替代 db/mod.rs）
    └── db/
        └── pool.rs      ← db 子模块，在 db.rs 中 mod pool;
RUSTmain.rs
// 声明模块——编译器查找对应文件
mod config;      // → src/config.rs
mod db;          // → src/db.rs 或 src/db/mod.rs
mod handlers;    // → src/handlers.rs

fn main() {
    config::load();
    db::pool::connect();
    handlers::handle_request();
}
RUSTdb.rs
// db 模块的内容
pub mod pool;    // 声明子模块 → src/db/pool.rs

pub fn init() {
    println!("数据库初始化");
}
RUSTdb/pool.rs
// db::pool 子模块
pub fn connect() {
    println!("连接数据库");
}
🌐 工程实践
  • 推荐扁平化：2018 edition 的 module.rs + module/ 风格比满屏 mod.rs 更清晰
  • mod 只在 crate root 和父模块中声明：子文件不需要再次 mod 自己
  • 模块树 ≠ 文件树：内联 mod 可以在任何文件中定义子模块，不需要额外文件
  • 常见结构：main.rs 做入口和模块声明，lib.rs 做库 API，具体逻辑放在子模块
✏️ 示例分析与易错点
RUSTcommon_mistakes.rs
// ❌ 错误：在子模块文件中再次 mod 声明自己
// src/config.rs 中写：
// mod config { ... }  ← 错！这会创建 config::config

// ✅ 正确：config.rs 直接写内容
pub fn load() -> String {
    "prod".to_string()
}

// ❌ 错误：忘记在父模块中声明
// src/utils.rs 存在但 main.rs 中没有 mod utils;
// → 编译器不会自动发现文件！

// ❌ 错误：同时有 db.rs 和 db/mod.rs
// → 编译错误！两种风格不能同时存在于同一个模块

⚠ Rust 编译器不会自动扫描目录中的文件。每个模块都必须通过 mod name; 显式声明。

02pub 与可见性

🎯 为什么需要了解这个

可见性控制是 API 设计的核心。暴露太多内部细节会限制未来的重构空间；暴露太少则用户无法完成需求。Rust 提供了非常细粒度的可见性控制，让你精确决定每个函数、类型、字段的公开范围。

🔍 核心思路
可见性含义典型用途
（无修饰）私有——当前模块及子模块可见内部实现
pub完全公开公共 API
pub(crate)crate 内部公开跨模块共享但不暴露给外部
pub(super)父模块可见兄弟模块间共享
pub(in path)指定路径可见精确控制（少用）
📐 规则与语法
RUSTvisibility.rs
// === 模块级可见性 ===
mod outer {
    pub fn public_fn() {}        // 任何人都能调用
    fn private_fn() {}           // 只有 outer 及其子模块可调用
    pub(crate) fn crate_fn() {}  // 同 crate 内可调用

    pub mod inner {
        pub fn inner_public() {}
        pub(super) fn for_parent() {}  // 只有 outer 可调用

        fn demo() {
            super::private_fn();   // ✅ 子模块可访问父模块的私有项
            super::public_fn();    // ✅
        }
    }
}

fn main() {
    outer::public_fn();           // ✅
    // outer::private_fn();       // ❌ 私有
    outer::crate_fn();            // ✅ 同 crate
    outer::inner::inner_public(); // ✅
    // outer::inner::for_parent(); // ❌ pub(super) 只对 outer 可见
}

结构体字段的可见性：

RUSTstruct_visibility.rs
mod api {
    // 结构体是 pub 的，但字段各有不同
    pub struct User {
        pub name: String,         // 外部可访问
        pub(crate) email: String, // 仅 crate 内可访问
        password_hash: String,    // 私有！
    }

    impl User {
        // 构造函数——外部唯一创建 User 的方式
        pub fn new(name: String, email: String, password: &str) -> Self {
            User {
                name,
                email,
                password_hash: hash(password),
            }
        }

        pub fn verify(&self, password: &str) -> bool {
            hash(password) == self.password_hash
        }
    }

    fn hash(s: &str) -> String {
        format!("hashed_{}", s)  // 简化示例
    }
}

fn main() {
    let user = api::User::new(
        "Alice".into(), "alice@x.com".into(), "secret"
    );
    println!("{}", user.name);         // ✅ pub
    println!("{}", user.email);        // ✅ pub(crate)
    // println!("{}", user.password_hash); // ❌ 私有
}
🌐 工程实践
  • 最小暴露原则：默认私有，只在需要时提升可见性
  • pub 的结构体：如果有私有字段，必须提供 pub fn new() 构造函数
  • enum 变体：pub enum 的所有变体自动 pub（和 struct 不同）
  • pub(crate)：最实用的中间层级——跨模块共享内部 helper 而不污染公共 API
✏️ 示例分析与易错点
RUSTenum_vs_struct.rs
// pub enum：所有变体自动 pub
pub enum Color {
    Red,        // 自动 pub
    Green,      // 自动 pub
    Blue,       // 自动 pub
}

// pub struct：字段默认私有
pub struct Point {
    x: f64,     // 私有！
    y: f64,     // 私有！
}

// ❌ 外部不能这样创建
// let p = Point { x: 1.0, y: 2.0 }; // 编译错误

// ✅ 必须提供构造方法
impl Point {
    pub fn new(x: f64, y: f64) -> Self {
        Point { x, y }
    }
    pub fn x(&self) -> f64 { self.x }
    pub fn y(&self) -> f64 { self.y }
}

⚠ pub struct 有私有字段时，外部无法用字段语法 Struct { field: value } 构造。这是一个有意的设计——强制用户通过构造函数入口，方便后续修改内部表示。

03use 与路径

🎯 为什么需要了解这个

没有 use，每次调用函数都要写完整路径 std::collections::HashMap::new()。use 引入快捷方式减少冗余，而 pub use（re-export）更是库 API 设计的利器——它让用户面对简洁的接口，而非复杂的内部结构。

🔍 核心思路

Rust 的路径系统类比文件系统：

  • crate:: = 绝对路径的根（类似 /）
  • self:: = 当前模块（类似 ./）
  • super:: = 父模块（类似 ../）
  • 外部 crate 名直接作为路径起点（如 std::、serde::）
📐 规则与语法
RUSTuse_paths.rs
// === 绝对路径（从 crate 根开始） ===
use crate::config::Settings;

// === 外部 crate ===
use std::collections::HashMap;
use serde::Serialize;

// === 相对路径 ===
use self::helpers::validate;   // 当前模块的 helpers 子模块
use super::shared::Config;     // 父模块的 shared 子模块

// === 嵌套导入 ===
use std::io::{self, Read, Write};  // io 模块本身 + Read + Write
use std::collections::{HashMap, HashSet, BTreeMap};

// === 别名 ===
use std::io::Result as IoResult;
use std::fmt::Result as FmtResult;

// === glob（通配符）导入 ===
use std::prelude::v1::*;  // 通常只在 prelude 和测试中使用

pub use — 重新导出：

RUSTreexport.rs
// src/lib.rs
mod internal {
    pub mod deep {
        pub struct Config { pub name: String }
    }
}

// 没有 pub use 时，用户需要写：
// my_crate::internal::deep::Config

// 通过 pub use 简化公开 API
pub use internal::deep::Config;

// 现在用户只需要：
// my_crate::Config

// === 常见的 pub use 模式 ===
// 创建 prelude 模块，集中导出常用类型
pub mod prelude {
    pub use crate::Config;
    pub use crate::Error;
    pub use crate::Result;
}
// 用户：use my_crate::prelude::*;
🌐 工程实践
  • 惯用导入风格：函数用 use module::function;，类型/trait 直接导入名字
  • 同名冲突：用 as 别名或只导入到父模块级别
  • pub use 简化 API：让用户不必了解内部模块结构
  • prelude 模式：为常用类型创建 prelude 模块，方便 use crate::prelude::*;
✏️ 示例分析与易错点
RUSTuse_conventions.rs
// ✅ 推荐：函数导入到父模块
use std::io;
io::stdin();     // 明确知道 stdin 来自 io

// ✅ 推荐：类型直接导入
use std::collections::HashMap;
let map = HashMap::new();

// ❌ 不推荐：函数直接导入（来源不清）
use std::io::stdin;
stdin();  // 读者不知道 stdin 来自哪里

// ✅ 处理同名类型
use std::fmt::Result as FmtResult;
use std::io::Result as IoResult;

// 或者只导入到模块级别
use std::fmt;
use std::io;
fn example() -> fmt::Result { Ok(()) }
fn read() -> io::Result<()> { Ok(()) }

04Cargo 与依赖管理

🎯 为什么需要了解这个

Cargo 不只是编译工具——它是一个完整的项目和依赖管理系统。理解 Cargo.toml 的结构、workspace、features 等概念，是在真实项目中高效工作的前提。

🔍 核心思路

Cargo 的核心职责：

  • 构建：编译 crate 及其依赖（cargo build）
  • 依赖管理：从 crates.io 下载、解析版本、锁定依赖
  • 项目管理：创建项目、运行测试、生成文档、发布 crate
  • Workspace：管理多 crate 的 monorepo
📐 规则与语法
TOMLCargo.toml
[package]
name = "my-project"
version = "0.1.0"
edition = "2021"

[dependencies]
serde = { version = "1.0", features = ["derive"] }
tokio = { version = "1", features = ["full"] }
anyhow = "1"                    # 简写：只指定版本

[dev-dependencies]               # 仅测试时的依赖
assert_cmd = "2"
tempfile = "3"

[build-dependencies]             # 构建脚本的依赖
cc = "1"

[features]
default = ["json"]               # 默认启用的 features
json = ["serde/derive"]          # 自定义 feature
advanced = []

[[bin]]                          # 额外的二进制目标
name = "cli"
path = "src/bin/cli.rs"

Workspace 配置：

TOMLworkspace Cargo.toml
# 根目录 Cargo.toml
[workspace]
members = [
    "crates/core",
    "crates/cli",
    "crates/web",
]

# 共享依赖版本
[workspace.dependencies]
serde = { version = "1.0", features = ["derive"] }
tokio = { version = "1", features = ["full"] }

# 子 crate 的 Cargo.toml:
[dependencies]
serde = { workspace = true }     # 引用 workspace 级别的定义

常用 Cargo 命令：

BASHcargo_commands
cargo new my_project          # 创建新项目
cargo new my_lib --lib        # 创建库项目
cargo build                   # 编译（debug）
cargo build --release         # 编译（优化）
cargo run                     # 编译并运行
cargo test                    # 运行所有测试
cargo doc --open              # 生成并打开文档
cargo add serde               # 添加依赖
cargo add serde --features derive  # 带 features
cargo check                   # 快速检查编译错误（不生成二进制）
cargo clippy                  # 代码审查（lint）
cargo fmt                     # 格式化代码
🌐 工程实践
  • 版本约束："1.0" 等价于 ">=1.0.0, <2.0.0"（SemVer 兼容）
  • Cargo.lock：锁定精确版本——bin crate 应提交到 VCS，lib crate 通常不提交
  • features 设计：用 features 做条件编译，避免拉入不需要的依赖
  • cargo check：比 cargo build 快很多，日常开发首选
✏️ 示例分析与易错点
RUSTcfg_feature.rs
// 条件编译：根据 feature 引入不同的依赖
#[cfg(feature = "json")]
use serde_json;

#[cfg(feature = "json")]
pub fn parse_json(s: &str) -> Result<Value, serde_json::Error> {
    serde_json::from_str(s)
}

// 不启用 json feature 时，这个函数不存在
// 用户不会为不需要的功能付出编译时间和二进制大小

// Cargo.toml 中对应：
// [features]
// json = ["dep:serde_json"]
// [dependencies]
// serde_json = { version = "1", optional = true }

⚠ 依赖某个 crate 的 feature 被启用后是全局的——如果 crate A 和 crate B 都依赖 crate C 但启用不同的 features，最终 crate C 会启用两者的并集。
