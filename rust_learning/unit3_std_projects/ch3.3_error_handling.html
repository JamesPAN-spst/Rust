Unit 3 · 标准库与项目实践

错误处理

Rust 没有异常机制——它用类型系统表达错误。panic! 处理不可恢复的 bug，Result<T,E> 处理可恢复的失败。这种设计迫使你在编译期就面对每一个可能的错误，而非在运行时被意外击中。

01panic! 与 unwinding

🎯 为什么需要了解这个

在其他语言中，异常可以被悄悄忽略。Rust 选择了一种更激进的策略：不可恢复的错误直接 panic。理解 panic 的机制——它什么时候发生、default 行为是什么、如何配置——是编写健壮程序的第一步。

🔍 核心思路

panic 有两种行为模式（在 Cargo.toml 的 [profile] 中配置）：

  • Unwind（默认）：从 panic 点开始逆向遍历调用栈，逐帧调用析构函数（Drop）清理资源，然后终止当前线程。程序可以用 std::panic::catch_unwind 捕获。
  • Abort：直接终止整个进程，不做清理。二进制更小，适合嵌入式。

何时该 panic？ 违反了程序的内部不变量——通常是 bug，而非外部输入导致的失败。

📐 规则与语法
RUSTpanic_examples.rs
// 显式 panic
panic!("致命错误: 配置文件损坏");

// 带格式化的 panic
let file = "config.toml";
panic!("无法解析 {}: 格式错误", file);

// 隐式 panic 的常见来源
let v = vec![1, 2, 3];
let x = v[99];           // 越界 → panic!

let n: i32 = "abc".parse().unwrap();  // 解析失败 → panic!

// 设置 panic 策略
// Cargo.toml:
// [profile.release]
// panic = "abort"
RUSTcatch_unwind.rs
use std::panic;

// catch_unwind 可以捕获 panic（仅 unwind 模式下）
let result = panic::catch_unwind(|| {
    panic!("boom!");
});

match result {
    Ok(_) => println!("正常结束"),
    Err(_) => println!("捕获了一个 panic"),
}
// 输出: 捕获了一个 panic
// 注意：这不是常规错误处理方式！主要用于 FFI 边界等特殊场景
输出
捕获了一个 panic
🌐 工程实践
  • 适合 panic 的场景：测试中、原型验证、违反内部不变量（"this should never happen"）
  • 不适合 panic 的场景：用户输入无效、文件不存在、网络超时——这些应该用 Result
  • RUST_BACKTRACE=1：设置环境变量可以在 panic 时打印完整调用栈，方便调试
  • assert!/debug_assert!：前者在所有构建中检查，后者仅在 debug 构建中检查
✏️ 示例分析与易错点
RUSTassert_variants.rs
// assert! —— 条件断言
let age = 25;
assert!(age >= 0, "年龄不能为负: {}", age);

// assert_eq! / assert_ne! —— 相等性断言
let result = 2 + 2;
assert_eq!(result, 4);
assert_ne!(result, 5);

// debug_assert! —— 只在 debug 构建中生效
debug_assert!(expensive_check(), "仅在调试时检查");

// unreachable! —— 标记不可达代码
fn classify(n: i32) -> &'static str {
    match n.cmp(&0) {
        std::cmp::Ordering::Less    => "negative",
        std::cmp::Ordering::Equal   => "zero",
        std::cmp::Ordering::Greater => "positive",
    }
    // 如果有遗漏分支：unreachable!("不应到达这里")
}

⚠ 不要在库代码中随意 panic！库应该返回 Result 让调用者决定如何处理。只在真正不可恢复的情况下（如内部不变量被破坏）才 panic。

02Result<T,E> 详解

🎯 为什么需要了解这个

Result<T,E> 是 Rust 错误处理的核心。它把"这个操作可能失败"直接编码在类型中，编译器会强制你处理每一个可能的错误。这消除了"忘记检查返回值"这一类经典 bug。

🔍 核心思路

Result 的定义极其简单：

RUSTresult_def.rs
enum Result<T, E> {
    Ok(T),   // 操作成功，携带结果值
    Err(E),  // 操作失败，携带错误信息
}

围绕 Result 有一系列方法形成链式处理的惯用模式，避免深层嵌套的 match：

  • unwrap / expect：快速解包，失败时 panic
  • unwrap_or / unwrap_or_else：提供默认值
  • map / and_then：链式变换
  • map_err：变换错误类型
  • ok / err：转为 Option
📐 规则与语法
RUSTresult_methods.rs
use std::num::ParseIntError;

fn parse_number(s: &str) -> Result<i32, ParseIntError> {
    s.parse::<i32>()
}

fn main() {
    // === match 处理（最基础） ===
    match parse_number("42") {
        Ok(n) => println!("解析成功: {}", n),
        Err(e) => println!("解析失败: {}", e),
    }

    // === unwrap / expect ===
    let n = parse_number("42").unwrap();           // 42
    let n = parse_number("42").expect("必须是数字"); // 42
    // parse_number("abc").unwrap();                // panic!

    // === unwrap_or / unwrap_or_else ===
    let n = parse_number("abc").unwrap_or(0);        // 0
    let n = parse_number("abc").unwrap_or_else(|e| {
        eprintln!("警告: {}", e);
        -1
    });  // -1

    // === map：变换成功值 ===
    let doubled: Result<i32, _> = parse_number("21").map(|n| n * 2);
    // Ok(42)

    // === and_then：链式可能失败的操作 ===
    let result = parse_number("10")
        .and_then(|n| {
            if n > 0 { Ok(n) } else { Err("must be positive".parse().unwrap()) }
        });

    // === map_err：变换错误类型 ===
    let result: Result<i32, String> = parse_number("abc")
        .map_err(|e| format!("解析错误: {}", e));
    // Err("解析错误: invalid digit found in string")
}
RUSTresult_chain.rs
use std::fs;
use std::io;

// 链式操作：读文件 → 解析为数字 → 乘以 2
fn read_and_double(path: &str) -> Result<i64, String> {
    fs::read_to_string(path)
        .map_err(|e| format!("读取失败: {}", e))
        .and_then(|content| {
            content.trim().parse::<i64>()
                .map_err(|e| format!("解析失败: {}", e))
        })
        .map(|n| n * 2)
}

// 等价的用 ? 的写法（更常用，看 KP3）
🌐 工程实践
  • 原型阶段：unwrap() / expect() 快速推进，后续替换为正式处理
  • 生产代码：避免 unwrap，用 ? 传播或 match 精确处理
  • if let：只关心成功情况时用 if let Ok(v) = result { ... }
  • 链式调用：map + and_then 适合函数式风格，? 适合命令式风格
✏️ 示例分析与易错点
RUSTresult_pitfalls.rs
// ❌ 忽略 Result 的警告
fn bad_code() {
    let _ = std::fs::remove_file("temp.txt");
    // 用 let _ = 显式忽略。直接丢弃 Result 会触发 #[must_use] 警告
}

// ❌ 到处 unwrap
fn fragile(input: &str) -> i32 {
    input.parse().unwrap()  // 用户输入非数字 → panic!
}

// ✅ 优雅处理
fn robust(input: &str) -> Result<i32, String> {
    input.parse().map_err(|_| format!("'{}' 不是有效数字", input))
}

⚠ Result 标记了 #[must_use]——如果你调用一个返回 Result 的函数但不处理返回值，编译器会发出警告。不要忽视这个警告！

03? 运算符

🎯 为什么需要了解这个

手动 match 每个 Result 会让代码变得冗长。? 运算符是 Rust 的"错误传播语法糖"——它让你用一行代码完成"成功则继续，失败则返回"的逻辑，同时自动进行错误类型转换。

🔍 核心思路

expr? 等价于：

RUSTquestion_mark_desugar.rs
// expr?  展开为：
match expr {
    Ok(val) => val,
    Err(err) => return Err(From::from(err)),
}

关键点：

  • ? 只能在返回 Result（或 Option）的函数中使用
  • From::from(err) 自动进行错误类型转换
  • 多个 ? 可以链式使用，失败时在第一个错误处短路返回
📐 规则与语法
RUSTquestion_mark.rs
use std::fs;
use std::io;

// === 没有 ?：冗长的嵌套 match ===
fn read_username_v1() -> Result<String, io::Error> {
    let content = match fs::read_to_string("username.txt") {
        Ok(c) => c,
        Err(e) => return Err(e),
    };
    Ok(content.trim().to_string())
}

// === 用 ? 简化 ===
fn read_username_v2() -> Result<String, io::Error> {
    let content = fs::read_to_string("username.txt")?;
    Ok(content.trim().to_string())
}

// === 链式 ? ===
fn read_username_v3() -> Result<String, io::Error> {
    Ok(fs::read_to_string("username.txt")?.trim().to_string())
}

// === ? 也适用于 Option ===
fn first_even(nums: &[i32]) -> Option<i32> {
    let first = nums.first()?;     // None → 提前返回 None
    if first % 2 == 0 {
        Some(*first)
    } else {
        None
    }
}

错误类型自动转换：

RUSTfrom_conversion.rs
use std::io;
use std::num::ParseIntError;

// 定义统一的错误类型
#[derive(Debug)]
enum AppError {
    Io(io::Error),
    Parse(ParseIntError),
}

// 实现 From 转换
impl From<io::Error> for AppError {
    fn from(e: io::Error) -> Self { AppError::Io(e) }
}
impl From<ParseIntError> for AppError {
    fn from(e: ParseIntError) -> Self { AppError::Parse(e) }
}

// 现在 ? 能自动转换两种错误类型！
fn read_number(path: &str) -> Result<i32, AppError> {
    let content = std::fs::read_to_string(path)?;  // io::Error → AppError
    let num: i32 = content.trim().parse()?;         // ParseIntError → AppError
    Ok(num)
}
🌐 工程实践
  • main 中使用 ?：fn main() -> Result<(), Box<dyn std::error::Error>>
  • ? 适用于 Option：在返回 Option 的函数中，? 对 None 提前返回
  • 不要混用：不能在返回 Result 的函数中对 Option 用 ?（除非 .ok_or() 转换）
  • ? 在闭包中：? 从闭包返回，不是从外层函数返回。注意作用域！
RUSToption_to_result.rs
fn get_config(map: &std::collections::HashMap<&str, &str>)
    -> Result<&str, String>
{
    // Option → Result 用 .ok_or() 或 .ok_or_else()
    let host = map.get("host")
        .ok_or("缺少 host 配置".to_string())?;
    Ok(host)
}
✏️ 示例分析与易错点
RUSTquestion_mark_traps.rs
// ❌ ? 不能在返回 () 的函数中使用
fn do_stuff() {
    let content = std::fs::read_to_string("f.txt")?;
    //                                              ^ 编译错误
    // 函数返回 ()，不是 Result
}

// ❌ 闭包中的 ? 不会从外层函数返回
fn process(files: &[&str]) -> Result<(), std::io::Error> {
    // 这里的 ? 从闭包返回，不是从 process 返回
    // files.iter().for_each(|f| {
    //     let content = std::fs::read_to_string(f)?;  // 编译错误
    // });

    // ✅ 用 for 循环代替
    for f in files {
        let content = std::fs::read_to_string(f)?;  // 从 process 返回
        println!("{}", content);
    }
    Ok(())
}

⚠ ? 运算符 = 提前 return。它只能在函数签名匹配时使用。闭包内的 ? 从闭包自身返回，要特别注意。

04自定义错误类型

🎯 为什么需要了解这个

真实项目中的错误不止一种。文件 I/O 可能失败、JSON 可能解析出错、网络可能超时。你需要一个统一的错误类型来表达"所有可能的错误"，并让调用者可以根据错误类型做出不同的决策。

🔍 核心思路

自定义错误类型的标准套路：

  1. 定义一个 enum，每个变体代表一种错误原因
  2. 实现 std::fmt::Display（用户友好的错误消息）
  3. 实现 std::error::Error（进入标准错误类型层级）
  4. 为每种源错误实现 From（让 ? 自动转换）
📐 规则与语法
RUSTcustom_error.rs
use std::fmt;
use std::io;
use std::num::ParseIntError;

// 1. 定义错误枚举
#[derive(Debug)]
enum AppError {
    Io(io::Error),
    Parse(ParseIntError),
    Config(String),  // 自定义消息
}

// 2. 实现 Display
impl fmt::Display for AppError {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        match self {
            AppError::Io(e) => write!(f, "I/O 错误: {}", e),
            AppError::Parse(e) => write!(f, "解析错误: {}", e),
            AppError::Config(msg) => write!(f, "配置错误: {}", msg),
        }
    }
}

// 3. 实现 Error trait
impl std::error::Error for AppError {
    fn source(&self) -> Option<&(dyn std::error::Error + 'static)> {
        match self {
            AppError::Io(e) => Some(e),
            AppError::Parse(e) => Some(e),
            AppError::Config(_) => None,
        }
    }
}

// 4. 实现 From 转换
impl From<io::Error> for AppError {
    fn from(e: io::Error) -> Self { AppError::Io(e) }
}
impl From<ParseIntError> for AppError {
    fn from(e: ParseIntError) -> Self { AppError::Parse(e) }
}

// 现在可以潇洒地用 ?
fn load_config(path: &str) -> Result<i32, AppError> {
    let content = std::fs::read_to_string(path)?;  // io::Error → AppError
    let port: i32 = content.trim().parse()?;        // ParseIntError → AppError
    if port < 1 || port > 65535 {
        return Err(AppError::Config(format!("无效端口: {}", port)));
    }
    Ok(port)
}
🌐 工程实践
  • 库 vs 应用：库代码应该定义精确的错误类型（enum），应用代码可以用 Box<dyn Error> 或 anyhow
  • source() 方法：实现 source() 让错误链可追溯，调试时可以层层查看根因
  • type alias：type Result<T> = std::result::Result<T, AppError> 简化函数签名
  • 样板代码多？ → 用 thiserror crate 自动派生（见 KP5）
✏️ 示例分析与易错点
RUSTerror_matching.rs
// 调用者可以根据错误类型做精确处理
fn main() {
    match load_config("port.txt") {
        Ok(port) => println!("服务器端口: {}", port),
        Err(AppError::Io(e)) if e.kind() == io::ErrorKind::NotFound => {
            println!("配置文件不存在，使用默认端口 8080");
        }
        Err(AppError::Parse(_)) => {
            eprintln!("配置文件内容不是有效数字");
        }
        Err(e) => {
            eprintln!("错误: {}", e);
            // 追溯错误链
            let mut source = e.source();
            while let Some(cause) = source {
                eprintln!("  原因: {}", cause);
                source = cause.source();
            }
        }
    }
}

05anyhow 与 thiserror

🎯 为什么需要了解这个

手动为每个错误类型实现 Display、Error 和 From 是重复劳动。Rust 生态中有两个几乎"标配"的错误处理 crate：thiserror（用于库）和 anyhow（用于应用）。它们各自解决不同层面的问题。

🔍 核心思路
Crate适用场景核心思想
thiserror库开发用 derive 宏自动实现 Display/Error/From，减少样板代码
anyhow应用开发提供通用 anyhow::Error 类型，不需定义错误 enum

两者可以共存：库用 thiserror 定义精确错误，应用用 anyhow 包装一切。

📐 规则与语法

thiserror 示例：

RUSTwith_thiserror.rs
// Cargo.toml: thiserror = "1"
use thiserror::Error;

#[derive(Debug, Error)]
enum AppError {
    #[error("I/O 错误: {0}")]            // 自动实现 Display
    Io(#[from] std::io::Error),           // 自动实现 From

    #[error("解析错误: {0}")]
    Parse(#[from] std::num::ParseIntError),

    #[error("配置错误: {msg}")]
    Config { msg: String },
}

// 对比手动实现：上面 7 行 = 之前的 30+ 行！

fn load_config(path: &str) -> Result<i32, AppError> {
    let content = std::fs::read_to_string(path)?;
    let port: i32 = content.trim().parse()?;
    Ok(port)
}

anyhow 示例：

RUSTwith_anyhow.rs
// Cargo.toml: anyhow = "1"
use anyhow::{Context, Result};

// Result 是 anyhow::Result<T> = Result<T, anyhow::Error>
fn load_config(path: &str) -> Result<i32> {
    let content = std::fs::read_to_string(path)
        .context(format!("读取配置文件 {} 失败", path))?;
        // context() 添加人类可读的上下文信息

    let port: i32 = content.trim().parse()
        .context("配置文件内容不是有效端口号")?;

    Ok(port)
}

fn main() -> Result<()> {
    let port = load_config("port.txt")?;
    println!("端口: {}", port);
    Ok(())
}
// 错误输出（自动带链式上下文）：
// Error: 读取配置文件 port.txt 失败
//
// Caused by:
//     No such file or directory (os error 2)
🌐 工程实践
  • 库 → thiserror：保持精确的错误类型，让调用者可以 match 处理
  • 应用 → anyhow：快速开发，自动错误链，无需定义错误类型
  • anyhow + thiserror 混用：库 expose thiserror 错误，应用层用 anyhow 包装
  • bail! 和 ensure!：anyhow 提供的宏，用于快速返回错误
RUSTanyhow_macros.rs
use anyhow::{bail, ensure, Result};

fn validate_port(port: i32) -> Result<()> {
    // ensure! —— 条件不满足则返回错误
    ensure!(port > 0 && port < 65536, "端口 {} 超出范围", port);

    // bail! —— 无条件返回错误
    if port == 22 {
        bail!("端口 22 保留给 SSH");
    }

    Ok(())
}
