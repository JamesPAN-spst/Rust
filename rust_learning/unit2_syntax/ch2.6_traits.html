Ch 2.6 Trait 系统

Trait 是 Rust 抽象能力的核心：它让你描述“一个类型能做什么”，而不是只盯着“它是什么”。本章会把 impl、derive、trait bound，以及静态分发和动态分发的直觉串在一起。

🎯 学习目标

  • 理解 trait 如何表达共享行为，以及 impl 如何为类型实现行为
  • 掌握常见 derive 的用途，并知道它们为什么能减少样板代码
  • 理解 trait bound、impl Trait 的基本用法与设计意图
  • 建立对静态分发与动态分发的直觉，知道何时考虑 trait 对象

📋 前置知识

建议先完成函数、struct、enum 和模式匹配章节。本章默认你已经知道 impl 可以写普通方法，现在会进一步把它提升到“共享行为接口”的层面。

2.6.1 Trait：描述能力，而不是硬绑具体类型

🤔

如果日志、配置、网络消息都需要“格式化成一段文本”，为什么不直接给每个类型都写一个同名方法就好了？

Trait 的价值在于把“共享行为契约”提炼出来。这样调用方关心的是“你能不能做这件事”，而不是“你是不是某个具体类型”。

正式规则：trait 用来声明一组方法签名；使用 impl TraitName for Type 可以为具体类型实现这组行为。

Rusttrait_intro.rs
trait Summary {
    fn summary(&self) -> String;
}

struct News {
    title: String,
}

impl Summary for News {
    fn summary(&self) -> String {
        format!("news: {}", self.title)
    }
}

fn main() {
    let item = News { title: String::from("Rust 1.78") };
    println!("{}", item.summary());
}
输出 
news: Rust 1.78

抽象顺序：先从几个具体类型里看出共同动作，再抽 trait；不要为了“用上 trait”而过早抽象。

2.6.2 impl 与 derive：手写行为，或让编译器帮你生成常见行为

🤔

为什么有些 trait 需要自己 impl，而 Debug、Clone、Default 这类 trait 却经常只要一行 derive？

有些行为是业务语义相关的，例如如何生成摘要、如何比较领域对象，这通常需要你自己写 impl。另一些行为是通用且可机械推导的，例如打印调试信息、浅复制、默认值，编译器就能通过 derive 自动生成。

正式规则：derive 是为支持派生的 trait 自动生成实现；只有 trait 设计成可派生时，才能使用 #[derive(...)].

Rustderive.rs
#[derive(Debug, Clone, Default)]
struct Config {
    port: u16,
    debug: bool,
}

fn main() {
    let config = Config { port: 8080, debug: true };
    let copied = config.clone();
    let fallback = Config::default();

    println!("{:?}", copied);
    println!("{:?}", fallback);
}
输出 
Config { port: 8080, debug: true }
Config { port: 0, debug: false }

derive 不是“万能自动化”：当业务语义需要你自己决定时，例如排序规则、领域等价关系、展示文本，就不要机械依赖 derive。

因为它把大量重复样板从手写代码里拿掉，让你把精力放在真正有业务含义的实现上。

2.6.3 Trait bound 与 impl Trait：把“需要什么能力”写进函数签名

🤔

函数到底是该接收具体类型，还是接收“任何实现了某种能力的类型”？

当函数真正依赖的是某种行为，而不是某个具体实现时，用 trait bound 会更灵活。调用方只要提供实现了该 trait 的类型即可，函数本身不必知道更多细节。

正式规则：可以用 fn f(x: T) 或 fn f(x: impl Trait) 表示参数必须实现某个 trait；多个约束可通过 + 或 where 子句组合。

Rusttrait_bounds.rs
use std::fmt::Display;

fn announce(item: impl Display) {
    println!("announcement: {item}");
}

fn pair<T>(left: T, right: T)
where
    T: Display + PartialOrd,
{
    let bigger = if left >= right { left } else { right };
    println!("bigger = {bigger}");
}

fn main() {
    announce("Rust");
    pair(3, 5);
}
输出 
announcement: Rust
bigger = 5

签名思维：如果你的函数只需要“能打印”，那就别把参数写死成 String；把真正依赖的能力写成约束，会让 API 更可复用。

2.6.4 默认方法与组合：trait 也可以带来共享实现

🤔

Trait 只是“接口声明”吗？如果多个类型共享一部分默认行为，该怎么办？

trait 不只能声明必须实现的方法，也可以提供默认方法。这样类型只需实现最小必要能力，就能自动获得一部分通用行为。

这让 trait 成为“契约 + 可复用默认逻辑”的组合体，在库设计里尤其常见。它和传统面向对象继承不同，因为共享的是行为约束和默认实现，而不是字段层次。

Rustdefault_methods.rs
trait Summary {
    fn headline(&self) -> &str;

    fn preview(&self) -> String {
        format!("preview: {}", self.headline())
    }
}

struct Post {
    title: String,
}

impl Summary for Post {
    fn headline(&self) -> &str {
        &self.title
    }
}

fn main() {
    let post = Post { title: String::from("Traits") };
    println!("{}", post.preview());
}
输出 
preview: Traits

默认实现的边界：当默认逻辑只依赖 trait 自身已声明的方法时，它会非常自然；如果开始强依赖具体类型细节，就说明抽象层次可能不对。

它能让多个实现共享一部分通用行为，同时仍把关键差异留给各自类型去补足。

2.6.5 静态分发 vs 动态分发：编译期定形，还是运行时走 trait 对象？

🤔

当你听到“泛型更快、trait object 更灵活”时，真正该理解的到底是什么？

使用泛型 + trait bound 时，编译器会为具体类型生成对应代码，这通常称为静态分发；使用 &dyn Trait 或 Box<dyn Trait> 时，运行时通过虚表决定调用哪个实现，这就是动态分发。

直觉上，静态分发更适合性能敏感且类型在编译期已知的场景；动态分发更适合需要把不同具体类型放进同一容器、并通过统一接口处理的场景。

Rustdispatch.rs
trait Draw {
    fn draw(&self);
}

struct Button;
struct Label;

impl Draw for Button {
    fn draw(&self) { println!("button"); }
}

impl Draw for Label {
    fn draw(&self) { println!("label"); }
}

fn render_all(items: &[Box<dyn Draw>]) {
    for item in items {
        item.draw();
    }
}

fn main() {
    let items: Vec<Box<dyn Draw>> = vec![Box::new(Button), Box::new(Label)];
    render_all(&items);
}
输出 
button
label

不要把“动态分发”直接等同于“坏性能”：很多应用层场景里，灵活性和接口统一性更重要。关键是理解你为什么需要在同一位置装下多种实现。

当你要把多种实现同一 trait 的不同类型放进同一个集合，例如 UI 组件列表、插件列表、处理器链时，trait object 会很自然。

📌 章节总结

  • trait 用来描述共享行为契约，impl 让具体类型承诺并实现这些行为
  • derive 适合自动生成可机械推导的常见 trait，实现样板代码压缩
  • trait bound 与 impl Trait 让函数按“需要什么能力”而不是“接受什么具体类型”来设计
  • trait 可以提供默认方法，形成“契约 + 默认行为”的组合
  • 静态分发强调编译期已知类型，动态分发强调以 trait 对象统一处理不同实现
