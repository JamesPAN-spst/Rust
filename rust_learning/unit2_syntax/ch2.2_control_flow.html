Ch 2.2 控制流

Rust 的控制流不只是“让代码跑起来”，更是在训练你用表达式思维组织状态变化。学完这一章，你会知道 if、loop、while、for 各自解决什么问题，以及为什么 Rust 鼓励把结果从分支里直接产出。

🎯 学习目标

  • 理解 if 在 Rust 中既能分支也能产出值
  • 掌握 loop、while、for 的核心适用场景与差异
  • 学会使用 break 返回值组织循环结果
  • 建立“优先表达式、其次状态机、最后手写索引”的控制流直觉

📋 前置知识

建议先掌握变量绑定、基本类型和可变性。本章默认你已经知道布尔表达式、函数调用与最小程序结构，但还不要求掌握模式匹配细节。

2.2.1 if 不是语句装饰，而是表达式

🤔

为什么 Rust 允许把 if 直接写在 let 右边？这和“分支只是执行不同代码块”有什么本质区别？

Rust 把 if 设计成表达式，意味着分支不仅能控制流程，还能直接产出一个值。这样很多“先声明变量、再在不同分支里赋值”的写法可以合并成更紧凑、更不容易漏分支的表达。

正式规则：if / else 表达式的所有分支必须返回兼容的类型；条件本身必须是 bool，Rust 不会把整数自动当布尔值。

Rustif_expression.rs
fn main() {
    let is_debug = true;
    let level = if is_debug {
        "verbose"
    } else {
        "normal"
    };

    println!("log level = {level}");
}
输出 
log level = verbose

表达式心态：当两个分支都在“决定一个结果”时，优先考虑把 if 当作表达式，而不是先声明一个可变变量再回填。

因为 let level = if ... 最终要绑定成一个确定类型的值。如果一个分支返回 &str，另一个返回整数，编译器就无法为整个表达式定型。

2.2.2 loop：当你需要“直到我自己决定结束”

🤔

如果循环次数不固定，又不想把条件写在开头，Rust 为什么单独提供 loop，而不是只靠 while？

loop 表示“先进入，再由内部逻辑决定何时停下”。它很适合重试、读取直到成功、状态机推进等场景，因为退出条件往往散落在循环体内部。

正式规则：loop 会一直执行，直到遇到 break；break 可以携带一个值，把整个 loop 变成产出结果的表达式。

Rustloop_break_value.rs
fn main() {
    let mut attempts = 0;

    let token = loop {
        attempts += 1;
        if attempts == 3 {
            break format!("connected on attempt {attempts}");
        }
    };

    println!("{token}");
}
输出 
connected on attempt 3
它让“不断尝试直到拿到一个结果”这类逻辑更自然。你不必先声明一个外部可变变量，再在 break 前偷偷写进去。

2.2.3 while：适合条件驱动的状态推进

🤔

如果你每次迭代前都要先检查“还能不能继续”，为什么 while 比 loop + if 更清晰？

while 把“继续条件”放在入口，适合倒计时、轮询、缓冲区消费等场景。它的价值不是功能更强，而是让读代码的人在第一眼就知道“这个循环靠什么持续”。

正式规则：while 在每次进入循环体前先判断条件；条件为 false 时立即停止，不会像 loop 那样必须依赖内部 break。

Rustwhile_countdown.rs
fn main() {
    let mut remaining = 3;

    while remaining > 0 {
        println!("remaining = {remaining}");
        remaining -= 1;
    }

    println!("done");
}
输出 
remaining = 3
remaining = 2
remaining = 1
done

别把 while 当万能循环：如果你的循环目标其实是遍历集合，优先考虑 for；如果退出逻辑散落在中途，优先考虑 loop。

2.2.4 for：让遍历意图比索引更显眼

🤔

遍历数组时，为什么 Rust 更鼓励 for item in collection，而不是手写索引 i += 1？

for 把“逐个处理元素”的意图表达得非常直接，也能减少越界、索引同步和可变状态管理的负担。Rust 倾向于让你沿着迭代器思路写代码，而不是一上来就操作下标。

正式规则：for 会基于 IntoIterator 逐项取值；遍历数组、切片、Vec、范围都可以通过 for 完成。

Rustfor_iteration.rs
fn main() {
    let scores = [88, 92, 100];

    for score in scores {
        println!("score = {score}");
    }

    for step in 1..=3 {
        println!("step {step}");
    }
}
输出 
score = 88
score = 92
score = 100
step 1
step 2
step 3

工程直觉：只要你不是在实现某种算法细节，看到“遍历每个元素”这类需求时，先想 for，再想索引。

因为它把“怎么拿下一个元素”交给迭代协议处理，减少了手写 i += 1 和数组越界的机会。

2.2.5 如何在 if / loop / while / for 之间做选择？

🤔

写控制流时，真正该问的问题不是“我会不会语法”，而是“我要表达的状态变化是什么”。那该怎么选？

一个简单心智模型是：分支产出结果时想 if；无限重试靠 loop；入口条件明确时用 while；遍历序列时优先 for。 这能让你的代码更贴近意图，而不是只是“能跑”。

Rust 的表达式导向设计鼓励你少依赖外部可变状态，多让控制流直接返回结果。这样不仅更易读，也更容易在后续引入所有权与错误处理规则。

Rustchoose_flow.rs
fn main() {
    let user_input = Some("42");

    let parsed = if let Some(text) = user_input {
        text.parse::<i32>().ok()
    } else {
        None
    };

    println!("parsed = {:?}", parsed);
}
输出 
parsed = Some(42)

注意：本例里的 if let 只是预告下一章的模式匹配风格；当前你只需要记住 Rust 鼓励让控制流直接生成结果。

📌 章节总结

  • if 在 Rust 中是表达式，分支不仅控制流程，还可以直接产出值
  • loop 适合退出条件位于循环体内部的重试或状态机场景，break 可以返回值
  • while 适合入口条件明确的循环，读者能在第一眼看到持续条件
  • for 最适合遍历集合或范围，它让迭代意图比索引管理更突出
  • 选择控制流时优先表达意图，而不是先写可变变量再用语句把它们拼起来
