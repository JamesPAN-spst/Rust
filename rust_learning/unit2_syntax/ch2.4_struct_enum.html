Unit 2 · 基础语法

Ch 2.4 Struct 与 Enum

真正的 Rust 建模，不是把字段随手堆成几个变量，而是主动选择“这个状态是固定结构，还是多种可能之一”。本章会用 struct、enum、Option、Result 把这种建模意识建立起来。

🎯 学习目标

  • 理解 struct 如何表达稳定的数据形状以及为何比并排变量更清晰
  • 掌握 tuple struct、unit-like struct 和基础 impl 的使用直觉
  • 理解 enum 如何表达“多种互斥状态”以及每个变体可携带的数据
  • 建立对 Option / Result 的直觉，知道它们为什么优于空值和魔法错误码

📋 前置知识

建议先掌握函数、基本类型和控制流。本章会出现 match 风格的预告，但详细拆解将在下一章进行；当前重点是“如何用类型表达领域模型”。

2.4.1 Struct：把零散数据收束成一个清晰模型

🤔

用户资料、配置项、坐标信息为什么不该总是拆成一堆平行变量？

当几项数据总是一起出现时，把它们放进 struct 能让“这些字段属于同一个实体”变得显式。这样函数签名更清楚，后续扩展字段也不需要不断加参数。

正式规则：struct 定义一组命名字段；实例化时要为字段赋值，读取时使用点号语法。

Ruststruct_model.rs
struct User {
    name: String,
    active: bool,
    sign_in_count: u32,
}

fn main() {
    let user = User {
        name: String::from("Ferris"),
        active: true,
        sign_in_count: 3,
    };

    println!("{} / {} / {}", user.name, user.active, user.sign_in_count);
}
输出 
Ferris / true / 3

建模问题先于语法：当你发现函数参数开始变成 name, active, sign_in_count, ... 这种长串时，通常就在提醒你该引入 struct 了。

2.4.2 Tuple struct、unit struct 与 impl：语法不一样，建模语义也不一样

🤔

为什么 Rust 不只提供一种 struct 写法？难道普通命名字段还不够吗？

命名字段 struct 适合语义清晰的业务对象；tuple struct 适合“只想包装一个或几个值，但想让类型更有意义”的场景；unit-like struct 则常用作标记类型。

与此同时，impl 可以把与数据相关的行为放回类型身上，让“数据”和“对数据的操作”靠得更近。

Rusttuple_impl.rs
struct UserId(u64);
struct AlwaysOn;

struct Rectangle {
    width: u32,
    height: u32,
}

impl Rectangle {
    fn area(&self) -> u32 {
        self.width * self.height
    }
}

fn main() {
    let id = UserId(7);
    let rect = Rectangle { width: 3, height: 4 };
    let _marker = AlwaysOn;

    println!("id = {}, area = {}", id.0, rect.area());
}
输出 
id = 7, area = 12

别把 impl 想成“必须等到 trait 章节才能用”：即使还没学 trait，impl 也已经能用来组织普通方法和构造逻辑。

当你想把一个原始值包成更有业务意义的类型时，例如 UserId(u64)、Port(u16)。这样能减少把不同整数类型混用的风险。

2.4.3 Enum：让“可能状态”进入类型系统

🤔

如果一个网络请求可能处于 Loading、Success、Failed 三种状态，用几个布尔字段拼出来真的靠谱吗？

enum 最大的价值是把“互斥状态”直接变成类型的一部分。你不再需要靠多个字段手动维护“哪些组合合法”，而是让编译器知道某个值此刻只能是这些变体之一。

正式规则：enum 可以定义多个变体；每个变体既可以只是标签，也可以携带各自不同的数据。

Rustenum_states.rs
enum RequestState {
    Loading,
    Success(String),
    Failed(u16),
}

fn main() {
    let state = RequestState::Success(String::from("OK"));

    match state {
        RequestState::Loading => println!("loading"),
        RequestState::Success(body) => println!("success: {body}"),
        RequestState::Failed(code) => println!("failed: {code}"),
    }
}
输出 
success: OK

建模收益：和多个布尔值相比，enum 更容易保证状态组合合法，也更方便在后续新增一个变体时让编译器帮你补全处理逻辑。

2.4.4 Option 与 Result：把“可能没有值”和“可能失败”说清楚

🤔

很多语言喜欢用 null、-1、空字符串来表示异常情况。Rust 为什么宁愿让你多写一层 Option / Result？

Option 表示“这里可能有一个 T，也可能没有”；Result 表示“这里可能成功拿到 T，也可能带着 E 失败”。Rust 把这些不确定性显式放进类型里，避免调用者假装问题不存在。

这种设计会让 API 更啰嗦一点，但也让缺失值和错误处理更难被忽略，尤其是在大型代码库里价值很高。

Rustoption_result.rs
fn find_port(name: &str) -> Option<u16> {
    if name == "http" {
        Some(80)
    } else {
        None
    }
}

fn parse_port(text: &str) -> Result<u16, std::num::ParseIntError> {
    text.parse::<u16>()
}

fn main() {
    println!("{:?}", find_port("http"));
    println!("{:?}", parse_port("8080"));
}
输出 
Some(80)
Ok(8080)

直觉提醒：Option 不是“简化版 Result”；它表达的是“缺席本身是正常分支”，而不是“这里发生了具体错误”。

如果“没有值”本身就是预期状态，例如查找不到缓存项，用 Option；如果你需要告诉调用者为什么失败，用 Result。

2.4.5 Struct 还是 Enum？先问“这是一个对象，还是一种状态集合？”

🤔

很多建模纠结，其实不是语法不会，而是没先想清楚“我要表达的东西到底是什么”。

如果你描述的是一个实体稳定拥有的一组属性，通常选 struct；如果你描述的是“这个值可能处于几种不同形态之一”，通常选 enum。很多真实模型会把两者组合起来使用。

例如一个 HTTP 响应可以是 struct，其中 body 字段又可能是 enum；一个订单状态是 enum，而每个状态里又可以携带 struct 数据。类型不是互斥工具，而是建模积木。

Rustmodel_choice.rs
struct ServerConfig {
    host: String,
    port: u16,
}

enum StartupState {
    Idle,
    Binding(ServerConfig),
    Failed(String),
}

fn main() {
    let state = StartupState::Binding(ServerConfig {
        host: String::from("127.0.0.1"),
        port: 8080,
    });

    println!("{}", matches!(state, StartupState::Binding(_)));
}
输出 
true

一句判断法：问自己“这个类型的值，是一直有这些字段，还是只会在几种形态里选一种？”答案通常会指向 struct 或 enum。

如何快速区分 struct 和 enum 的使用倾向？

📌 章节总结

  • struct 适合表达稳定的实体形状，让相关字段以一个清晰模型出现
  • tuple struct 和 unit-like struct 提供了不同层次的语义包装能力，impl 用来组织与类型相关的行为
  • enum 适合表达多种互斥状态，每个变体都可以携带自己需要的数据
  • Option 与 Result 把“可能缺席”和“可能失败”显式编码进类型，减少空值与错误码带来的歧义
  • 建模时先分清“稳定对象”还是“状态集合”，再决定 struct、enum 或二者组合
