Ch 2.3 函数与闭包

函数让逻辑有边界，闭包让逻辑可以连同环境一起被传递。Rust 在这两者上既强调表达式风格，也强调捕获语义的明确性。本章会把 return、闭包捕获和 Fn / FnMut / FnOnce 的直觉串在一起。

🎯 学习目标

  • 掌握 Rust 函数签名、参数和返回值的基本规则
  • 理解语句与表达式的区别，以及它们如何影响返回值
  • 理解闭包如何捕获环境以及何时发生借用、可变借用与移动
  • 建立对 fn、Fn、FnMut、FnOnce 的使用直觉

📋 前置知识

默认你已经掌握变量、基本控制流和最小 Rust 程序结构。本章会碰到所有权术语，但重点不是借用检查细节，而是“函数/闭包怎样组织代码与环境”。

2.3.1 函数签名是在声明边界，不只是写语法

🤔

为什么 Rust 要你在函数参数和返回值上写得很明确？这是不是只是“类型语言的啰嗦”？

函数签名决定了调用者能给什么、会拿回什么，也是编译器推断 API 意图的重要依据。越清晰的签名，越容易看出所有权是否转移、结果是否可失败。

正式规则：使用 fn 定义函数，参数必须写类型；返回值类型在 -> 后声明，如果没有返回值则默认为单位类型 ()。

Rustfunction_signature.rs
fn add_tax(price: f64, rate: f64) -> f64 {
    price * (1.0 + rate)
}

fn main() {
    let final_price = add_tax(100.0, 0.13);
    println!("{final_price:.2}");
}
输出 
113.00

读签名的习惯：看到函数时，先读参数和返回值，再读函数体。这样你会更快进入“它承诺了什么”的视角。

2.3.2 表达式 vs 语句：最后一行为什么能当返回值？

🤔

为什么 Rust 函数里最后一行不写分号就能返回，而一旦加分号，类型就可能完全变掉？

Rust 区分“会产出值的表达式”和“只执行动作的语句”。函数体、if、match、块表达式都可以靠最后一个表达式产出结果，这让很多逻辑不需要显式 return。

正式规则：块中最后一个无分号表达式会成为该块的值；加上分号后，它就变成语句，值变为 ()。

Rustexpression_return.rs
fn label(score: i32) -> &'static str {
    if score >= 60 {
        "pass"
    } else {
        "retry"
    }
}

fn main() {
    println!("{}", label(75));
}
输出 
pass

高频报错来源：很多“expected i32, found ()”都来自多写了一个分号，让本该返回值的表达式变成了语句。

return 适合提前退出，例如参数校验失败、遇到错误就立刻返回。平稳路径则常用最后一行表达式，让结构更紧凑。

2.3.3 闭包：把“逻辑 + 环境”打包在一起

🤔

为什么迭代器、回调和排序 API 喜欢接收闭包，而不是只接收普通函数名？

普通函数只能依赖参数；闭包还能顺手捕获外部环境中的值，所以它特别适合“当场写一段小逻辑，同时利用当前上下文”的场景。

正式规则：闭包语法通常写作 |args| expr 或 |args| { ... }；编译器会根据使用场景为参数和返回值推断类型，并自动决定捕获方式。

Rustclosure_capture.rs
fn main() {
    let prefix = String::from("item");
    let render = |id: u32| format!("{prefix}-{id}");

    println!("{}", render(7));
}
输出 
item-7

直觉：闭包最适合“只在本地用一次或几次的小逻辑”；如果一段逻辑值得命名、复用和测试，仍然优先考虑普通函数。

因为闭包往往直接在调用点被消费，编译器能从上下文推断参数和返回类型。但一旦上下文不清晰，仍然要补注解。

2.3.4 捕获模式：借用、可变借用、move

🤔

同样是闭包，为什么有时调用完外部变量还能用，有时却提示“value moved into closure”？

闭包会根据它对环境值的使用方式自动选择捕获模式：只读时倾向借用，修改时倾向可变借用，需要拿走所有权时则会 move。理解这一点，比死记某个报错更重要。

正式规则：闭包会尽量以最小权限捕获环境；加上 move 关键字会强制把捕获值的所有权移入闭包。

Rustcapture_modes.rs
fn main() {
    let mut count = 0;
    {
        let mut inc = || {
            count += 1;
            println!("count = {count}");
        };
        inc();
        inc();
    }

    let label = String::from("job");
    let take = move || println!("{label}");
    take();
}
输出 
count = 1
count = 2
job

不要把 move 理解成“闭包专用关键字”：它本质上仍然是所有权转移。在线程、异步任务或需要延长生命周期的场景里，move 闭包尤为常见。

先问闭包对外部值做了什么：只读、修改，还是要在闭包内部长期持有/转移出去。行为决定捕获模式，语法只是表现。

2.3.5 fn / Fn / FnMut / FnOnce：把“怎么调用”看成能力等级

🤔

面对这些看似相似的名字，最容易掉坑的是去死背定义。有没有更直观的理解方式？

可以把它们理解成闭包可调用能力的三个层级：Fn 表示只读调用、可反复调用；FnMut 表示调用时可能修改内部状态；FnOnce 表示调用一次就可能消耗掉捕获值。fn 则是函数指针类型，本身不捕获环境。

一个闭包如果只需要只读捕获，它通常既满足 Fn，也自动满足 FnMut / FnOnce 的较宽接口需求；反过来则不成立。理解“能力越强，约束越少”会比记表格更有用。

Rustfn_traits.rs
fn call_twice<F>(mut f: F)
where
    F: FnMut(),
{
    f();
    f();
}

fn main() {
    let mut count = 0;
    call_twice(|| {
        count += 1;
        println!("count = {count}");
    });
}
输出 
count = 1
count = 2

选接口的经验法则：如果你只需要读环境，用 Fn；需要多次调用且允许改状态，用 FnMut；要接管资源并允许只调用一次，用 FnOnce。

📌 章节总结

  • 函数签名描述的是 API 边界：参数是什么、返回什么，以及结果如何被理解
  • Rust 以表达式为核心，块的最后一行无分号时可以直接成为返回值
  • 闭包是在本地把逻辑与环境一起打包的工具，特别适合回调和迭代器 API
  • 闭包捕获模式由使用方式决定，move 会显式把所有权转入闭包
  • Fn / FnMut / FnOnce 可以理解成闭包“怎么调用、会不会消耗状态”的能力分级
