Ch 2.5 模式匹配

到了 Rust，match 不是可有可无的语法糖，而是理解 enum、Option、Result 乃至数据解构的核心工具。本章会把 match、if let、while let、守卫和穷尽性放进同一个心智模型。

🎯 学习目标

  • 理解 match 如何把“按形状拆值”与“按情况分支”结合起来
  • 掌握对 enum、tuple、struct 的常见解构方式
  • 知道 if let / while let 适合解决什么问题，以及它们和 match 的取舍
  • 建立对穷尽性检查、match guard 与 _ 占位的直觉

📋 前置知识

建议先完成 struct、enum、Option、Result 的学习。本章假设你已经知道这些类型长什么样，但还没有系统建立“按模式拆解它们”的习惯。

2.5.1 match：按值的形状来分支，而不是按 if 条件硬猜

🤔

面对 enum 或 Option，为什么 Rust 更推荐 match，而不是一层层 if else？

match 的优势不只是分支多，而是它把“我期待这个值长成什么样”写成了模式。读代码的人不仅知道走哪条路，还知道每条路拿到了哪些绑定值。

正式规则：match 会从上到下尝试各个分支模式，命中后执行对应分支；整个 match 必须覆盖所有可能情况。

Rustmatch_option.rs
fn main() {
    let port = Some(8080);

    match port {
        Some(value) => println!("port = {value}"),
        None => println!("no port"),
    }
}
输出 
port = 8080

先想模式，再想逻辑：当值本身有明确形状时，match 往往比写若干布尔条件更直观，也更不容易漏掉某个分支。

2.5.2 解构：tuple、struct、enum 都能按模式拆开

🤔

模式匹配为什么常常让人有“像在拆包”的感觉？

因为模式本质上就是对数据形状的描述。你可以在 match 里直接把 tuple、struct、enum 里的内容拆出来，而不是先拿整个值、再点字段、再判断。

这让“数据长什么样”和“我需要哪些部分”紧密靠在一起，非常适合处理复杂返回值和嵌套状态。

Rustdestructure.rs
struct Point {
    x: i32,
    y: i32,
}

fn main() {
    let point = Point { x: 3, y: 4 };

    match point {
        Point { x, y: 0 } => println!("on x-axis: {x}"),
        Point { x, y } => println!("point = ({x}, {y})"),
    }
}
输出 
point = (3, 4)

不要把模式理解成“只能配合 enum”：tuple、数组、struct 甚至字面量范围都能参与模式匹配；这也是 Rust 解构能力强大的原因。

可以。你可以只绑定自己关心的字段，其余字段用 .. 忽略。模式匹配强调“我要的形状”，不是“必须手动取出所有内容”。

2.5.3 if let 与 while let：为“只关心一种模式”而生

🤔

如果你只想处理 Some(value) 这一种情况，完整写一个 match 会不会太重？

if let 适合“我只关心某一种模式，其余情况略过”；while let 则适合“只要某种模式持续成立就继续循环”。它们本质上是针对常见场景的 match 精简写法。

正式规则：if let / while let 只处理一个成功匹配的模式，其余情况会走 else 或结束循环，因此不具备完整的穷尽性检查。

Rustif_while_let.rs
fn main() {
    let mut stack = vec![1, 2, 3];

    if let Some(top) = stack.last() {
        println!("top = {top}");
    }

    while let Some(value) = stack.pop() {
        println!("pop {value}");
    }
}
输出 
top = 3
pop 3
pop 2
pop 1

取舍原则：如果你只关心一种模式，if let / while let 更简洁；如果每种情况都很重要，或者你想要穷尽性反馈，回到 match。

2.5.4 守卫、范围与 _：模式不只会“等于某个值”

🤔

如果你既想匹配模式，又想加一个额外条件，例如“Some(x) 且 x > 10”，该怎么写才不别扭？

Rust 允许在模式后加守卫 if，让你在成功匹配形状后再补充更细的条件。你也可以用字面量、范围、_ 占位和绑定名，把分支写得既精确又不冗长。

正式规则：match guard 写作 pattern if condition；下划线 _ 表示忽略某部分或整个剩余情况。

Rustmatch_guards.rs
fn main() {
    let code = Some(404);

    match code {
        Some(value) if value >= 500 => println!("server error"),
        Some(404) => println!("not found"),
        Some(value @ 400..=499) => println!("client error: {value}"),
        Some(value) => println!("status: {value}"),
        None => println!("missing"),
    }
}
输出 
not found

注意顺序：match 会按分支顺序匹配。把宽泛模式写在前面，可能会让后面更具体的分支永远到不了。

因为真实世界里并不是每个分支你都关心细节。_ 能明确表示“其余情况我知道存在，但这里不需要逐一命名”。

2.5.5 穷尽性：编译器在逼你把状态空间想完整

🤔

为什么 Rust 对 match 那么严格，甚至少写一个分支都不让过？这不是在增加心智负担吗？

恰恰相反，穷尽性检查是在帮你把“可能状态”一次性想清楚。尤其当 enum 后续新增变体时，编译器会把所有未更新的匹配点都找出来，这是一种非常强的重构保护。

把它理解成“类型驱动的待办清单”会比把它看成限制更有帮助：模型变了，编译器会提醒你哪些业务逻辑还没跟上。

Rustexhaustive.rs
enum Mode {
    Dev,
    Prod,
}

fn describe(mode: Mode) -> &'static str {
    match mode {
        Mode::Dev => "developer mode",
        Mode::Prod => "production mode",
    }
}

fn main() {
    println!("{}", describe(Mode::Dev));
}
输出 
developer mode

把穷尽性当朋友：当你新增一个枚举变体时，编译错误往往正是在告诉你“还有哪些业务路径没设计完成”。

因为它天生就只承诺处理一个模式，其余情况可直接忽略；而 match 的职责是对整个状态空间做显式分派。

Rust 强调 match 穷尽性的主要收益是什么？

📌 章节总结

  • match 按值的形状来分支，非常适合处理 enum、Option、Result 和结构化数据
  • 模式可以直接解构 tuple、struct、enum，把“匹配”和“取值”合并在一起
  • if let / while let 是只关心单一模式时的简化工具，但不提供完整穷尽性检查
  • match guard、范围和 _ 让模式既精确又灵活，但分支顺序仍然重要
  • 穷尽性检查是 Rust 的设计反馈机制，能在模型变化时主动暴露遗漏逻辑

🧪 课后测试

  1. ⭐ 基础： 写一个 match，输入 Option，分别打印 Some(value) 和 None。
  2. ⭐⭐ 进阶： 定义一个 Point struct，用 match 解构并区分“在 x 轴上”和“普通点”两种情况。
  3. ⭐⭐⭐ 挑战： 把一个 while 循环改写成 while let：不断从 Vec 弹出元素直到为空，并说明这样写为什么更贴合数据形状。
  4. 🏆 附加： 设想你在写一个解析器：输入可能是数字、标识符或非法 token。请用 enum + match 设计处理流程，并说明穷尽性检查能帮到什么。
