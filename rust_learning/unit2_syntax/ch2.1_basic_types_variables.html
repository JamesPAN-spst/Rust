Ch 2.1 基本类型与变量

这一章解决 Rust 初学者最常见的第一批问题：变量到底绑定了什么、什么时候需要 `mut`、推断何时足够、 元组和数组适合放什么数据，以及为什么 `const` 和 `let` 不能混用。学完后，你会更容易读懂编译器提示。

🎯 学习目标

  • 理解 Rust 中“绑定”与“值”的关系，区分可变绑定与不可变绑定
  • 掌握常见标量类型与复合类型的使用方式和适用场景
  • 理解 shadowing、类型推断和显式注解分别在解决什么问题
  • 避免 `const`、数组长度、越界访问、错误类型假设等入门阶段常见错误

📋 前置知识

默认你已经能运行一个最小的 Rust 程序，并知道 `fn main()`、`println!` 和基本编译流程。本章不讨论所有权细节，但会偶尔提到“绑定”和“值”不是一回事。

2.1.1 变量绑定为什么默认不可变？

🤔

为什么 Rust 不像很多语言那样默认允许变量随时改值，而是要求你显式写 `mut`？

Rust 里的 `let` 更像“给一个值起名字并绑定到当前作用域”，而不是“创建一个随时可改的盒子”。默认不可变能让编译器更早发现意外修改，也让阅读代码时更容易判断状态是否会变化。

正式规则：使用 `let` 创建的绑定默认不可变；只有 `let mut` 才允许通过该绑定修改值。

Rustbindings.rs
fn main() {
    let language = "Rust";
    let mut version = 1.77;

    println!("{language}");
    version = 1.78;
    println!("version = {version}");

    // language = "Rustacean"; // 编译错误：cannot assign twice to immutable variable
}
输出 
Rust
version = 1.78

实践建议：先写不可变绑定，只有确实需要更新状态时再补 `mut`。这通常会让后续借用和重构更轻松。

2.1.2 标量类型：推断够用，但别忘了边界

🧠

为什么 `let x = 42;` 常常没问题，但一旦调用 `parse()` 或做数值运算，编译器又会要求你写类型？

Rust 会尽量根据上下文推断整数、浮点、布尔和字符类型，但当信息不够时，编译器不会猜测到可能出错的程度。你需要用注解告诉它“这里应该是哪一种标量”。

正式规则：Rust 的标量类型包括整数、浮点数、布尔和字符；当上下文不足以唯一确定类型时，必须添加显式注解。

Rustscalar_types.rs
fn main() {
    let retries: u8 = 3;
    let ratio = 0.5f32;
    let is_ready = true;
    let initial = 'R';
    let port: u16 = "8080".parse().expect("端口必须是数字");

    println!("{retries}, {ratio}, {is_ready}, {initial}, {port}");
}
输出 
3, 0.5, true, R, 8080
`'R'` 是 `char`，表示单个 Unicode 标量值；`"R"` 是字符串切片 `&str`。初学者经常把字符和字符串字面量混为一谈，导致类型不匹配。

2.1.3 Shadowing 不是可变性替代品

🔁

为什么 Rust 既有 `mut`，又允许你再次写 `let spaces = spaces.len();`？它们看起来都像“改值”。

`mut` 是在同一个绑定上修改值；shadowing 则是创建一个同名的新绑定，把旧绑定遮蔽掉。它的好处是你可以在保留名称的同时改变类型、缩小生命周期或者分阶段清洗数据。

正式规则：同一作用域内可以使用新的 `let` 对已有名称进行 shadowing；新绑定与旧绑定是两个不同的绑定。

Rustshadowing.rs
fn main() {
    let input = " 42 ";
    let input = input.trim();
    let input: i32 = input.parse().expect("必须是整数");

    println!("parsed = {input}");

    let mut counter = 1;
    counter += 1;
    println!("counter = {counter}");
}
输出 
parsed = 42
counter = 2

常见误区：想把 `String` 处理成长度时，很多人会先写 `let mut s = String::from("hi"); s = s.len();`，这会失败，因为同一绑定不能在赋值时换类型；此时应该用 shadowing。

2.1.4 复合类型：元组适合打包，数组强调固定长度

📦

如果你想把一个函数的多个返回值放在一起，或者需要一个长度固定的配置表，应该用同一种类型吗？

Rust 的复合类型里，元组可以把不同类型组合在一起，常用于返回多个结果；数组要求所有元素类型相同且长度固定，适合已知大小的数据块、缓冲区和查表场景。

正式规则：元组元素类型可不同，通过解构或 `.0`、`.1` 访问；数组写作 `[T; N]`，其中长度 `N` 是类型的一部分。

Rusttuples_arrays.rs
fn main() {
    let http_status: (u16, &str) = (200, "OK");
    let (code, message) = http_status;

    let weekdays = ["Mon", "Tue", "Wed", "Thu", "Fri"];
    let zeros = [0; 4];

    println!("{code} {message}");
    println!("first = {}, len = {}", weekdays[0], weekdays.len());
    println!("zeros = {:?}", zeros);
}
输出 
200 OK
first = Mon, len = 5
zeros = [0, 0, 0, 0]
运行时索引越界会 panic，因为 Rust 不允许悄悄读到非法内存。编译器无法在所有情况下静态证明索引安全时，会把越界检查保留到运行时。

2.1.5 `const` vs `let`：编译期常量不是“永远不变的变量”

📌

配置端口、重试上限、数组长度这些值，什么时候应该写成 `const`，什么时候普通 `let` 就够了？

`let` 绑定的是运行时值；`const` 定义的是编译期常量，必须显式写类型，而且值必须能在编译期求出。二者都可以不可变，但它们服务的阶段不同。

正式规则：`const` 需要类型注解，不能依赖运行时计算；`let` 可以配合推断在运行时绑定值，并且可以被 shadowing。

Rustconst_vs_let.rs
const MAX_RETRIES: u8 = 5;
const BUFFER_SIZE: usize = 1024;

fn main() {
    let filename = "config.toml";
    let retry_delay_ms = 200;

    println!("file = {filename}");
    println!("retries = {MAX_RETRIES}, buffer = {BUFFER_SIZE}, delay = {retry_delay_ms}");
}
输出 
file = config.toml
retries = 5, buffer = 1024, delay = 200

入门高频错误清单：把 `const` 当作“全局 `let`”、忘记给 `const` 写类型、认为 `mut` 可以改变变量类型、把 `char` 写成字符串、以为数组和 `Vec` 一样可伸缩。

📌 章节总结

  • `let` 创建的是绑定，默认不可变；`let mut` 才允许通过该绑定修改值
  • 标量类型通常可由上下文推断，但在 `parse()`、边界敏感数值和公共 API 中应主动注解
  • shadowing 适合分阶段转换数据或改变类型，`mut` 适合同一类型上的状态更新
  • 元组适合组合异构返回值，数组适合同构且固定长度的数据
  • `const` 是编译期常量，和普通不可变 `let` 的使用场景不同
