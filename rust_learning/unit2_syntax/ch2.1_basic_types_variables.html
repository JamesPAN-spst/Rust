Unit 2 · 基础语法

基本类型与变量

Rust 的类型系统从第一行 let 开始就在引导你写出安全的代码。本章系统梳理四种绑定方式、标量与复合类型、shadowing 的设计意图，以及类型转换的最佳实践。

01let / mut / const / static — 四种绑定

🎯 为什么需要了解这个

在 C/C++ 中，变量默认可变，程序员需要自律添加 const。Rust 反转了默认——let 绑定默认不可变。这不是语法糖，而是"安全第一"哲学在最底层的体现：修改数据的地方越少，bug 就越少。

🔍 核心思路
关键字可变性作用域值的性质
let不可变局部运行时求值
let mut可变局部运行时求值
const不可变任意（编译期内联）编译期常量表达式
static不可变*全局（固定内存地址）程序全生命周期

*static mut 存在但需要 unsafe，实际开发中几乎不用。

📐 规则与语法
RUSTbindings.rs
fn main() {
    // 1. let：不可变绑定
    let x = 10;
    // x = 20; // ❌ 编译错误：cannot assign twice to immutable variable

    // 2. let mut：可变绑定
    let mut y = 10;
    y = 20; // ✅ OK

    // 3. const：编译期常量，必须标注类型
    const MAX_POINTS: u32 = 100_000;

    // 4. static：全局变量，有固定内存地址
    static LANGUAGE: &str = "Rust";

    println!("x={}, y={}, MAX={}, LANG={}", x, y, MAX_POINTS, LANGUAGE);
}
输出
x=10, y=20, MAX=100000, LANG=Rust

关键区别：const 的值会被编译器内联到每个使用处（类似 C 的 #define，但有类型检查）；static 有固定的内存地址，可以取引用 &LANGUAGE。

🌐 工程实践
  • 默认用 let，只在需要修改时才加 mut——代码审查时一眼看出哪些变量会变。
  • 用 const 做配置常量：const TIMEOUT_MS: u64 = 5000;，替代魔法数字。
  • 用 static 做全局单例：配合 std::sync::OnceLock 或 lazy_static! 做延迟初始化。
  • static mut 几乎永远不要用——用 Mutex/RwLock/AtomicXxx 替代。
✏️ 示例分析与易错点
RUSTconst_vs_static.rs
const PI: f64 = 3.14159265358979;
static GREETING: &str = "hello";

fn main() {
    // const 每次使用可能产生不同的内联副本
    // static 有唯一固定的内存地址
    let g: &'static str = GREETING;
    println!("{}, PI={}", g, PI);

    // static 可以取地址比较
    let ptr1 = &GREETING as *const _;
    let ptr2 = &GREETING as *const _;
    println!("same address: {}", std::ptr::eq(ptr1, ptr2));
}
输出
hello, PI=3.14159265358979
same address: true
易错：const 不能用于非 const fn 的函数调用。例如 const V: Vec<i32> = Vec::new(); 可以（Vec::new 是 const fn），但 const S: String = String::from("hi"); 不行。
⚡ 快速检测

02标量类型 — 整数、浮点、布尔、字符

🎯 为什么需要了解这个

Rust 是强类型语言，不会像 C 一样隐式转换 int + float。选错类型不只是"跑不动"——可能导致溢出 panic 或截断丢数据。掌握每种标量的宽度和行为是避免运行时 bug 的基础。

🔍 核心思路

Rust 有四类标量：

类别类型说明
有符号整数i8 / i16 / i32 / i64 / i128 / isize补码表示
无符号整数u8 / u16 / u32 / u64 / u128 / usize非负
浮点数f32, f64IEEE 754；默认 f64
布尔booltrue / false，1 字节
字符charUnicode 标量值，4 字节

默认推断：整数字面量 → i32，浮点字面量 → f64。

📐 规则与语法
RUSTscalars.rs
fn main() {
    // 整数字面量的多种写法
    let decimal = 98_222;        // 下划线增加可读性
    let hex     = 0xff;          // 十六进制
    let octal   = 0o77;          // 八进制
    let binary  = 0b1111_0000;   // 二进制
    let byte    = b'A';          // u8 字节字面量 = 65

    // 类型后缀
    let x = 42u8;                // 明确指定 u8
    let y = 3.14f32;             // 明确指定 f32

    // bool
    let t = true;
    let f: bool = false;

    // char — 注意是单引号，支持 Unicode
    let c = 'z';
    let emoji = '🦀';
    println!("decimal={}, hex={}, byte={}", decimal, hex, byte);
    println!("char size: {} bytes", std::mem::size_of::<char>());
}
输出
decimal=98222, hex=255, byte=65
char size: 4 bytes
🌐 工程实践
  • 索引用 usize——数组索引、.len() 返回值都是 usize。
  • 网络/文件用固定宽度——协议解析中用 u16/u32 精确匹配字段宽度。
  • 溢出处理四兄弟——checked_add（返回 Option）/ wrapping_add（回绕）/ saturating_add（饱和）/ overflowing_add（返回元组）。debug 模式下直接 + 溢出会 panic。
  • 避免 f32 做精度敏感计算——金融场景用整数（分为单位）或专用 decimal 库。
✏️ 示例分析与易错点
RUSToverflow.rs
fn main() {
    let a: u8 = 255;

    // debug 模式：直接 a + 1 会 panic!
    // let b = a + 1; // thread 'main' panicked at 'attempt to add with overflow'

    // 四种安全替代方案
    let b = a.checked_add(1);     // None（溢出）
    let c = a.wrapping_add(1);    // 0（回绕）
    let d = a.saturating_add(1);  // 255（饱和到最大值）
    let (e, overflowed) = a.overflowing_add(1); // (0, true)

    println!("checked:     {:?}", b);
    println!("wrapping:    {}", c);
    println!("saturating:  {}", d);
    println!("overflowing: {} (overflow={})", e, overflowed);
}
输出
checked:     None
wrapping:    0
saturating:  255
overflowing: 0 (overflow=true)
易错：不要混用 usize 和 i32——Rust 不会隐式转换，需要显式 as usize 或 try_into()。
⚡ 快速检测
在 64 位系统上，usize 等价于哪个类型？

03复合类型 — 元组与数组

🎯 为什么需要了解这个

标量只存单个值。需要把多个值打包时，Rust 提供两种编译期固定大小的复合类型：元组（不同类型的组合）和数组（相同类型的序列）。它们在栈上分配，不涉及堆，是最轻量的组合结构。

🔍 核心思路
特征元组 (T1, T2, ...)数组 [T; N]
元素类型可以不同必须相同
长度编译期固定编译期固定
访问方式.0 .1 .2 或解构[index] 或切片
存储位置栈栈

切片 &[T] 是数组/Vec 的动态视图——不拥有数据，只包含 (指针, 长度)。

📐 规则与语法
RUSTcompound.rs
fn main() {
    // ---- 元组 ----
    let tup: (i32, f64, char) = (42, 6.28, '🦀');
    let (x, y, z) = tup;           // 解构
    println!("x={}, y={}, z={}", x, y, z);
    println!("tup.0 = {}", tup.0); // 索引访问

    // 单元素元组需要逗号区分
    let single = (42,);    // 元组 (i32,)
    let not_tuple = (42);  // 只是 i32

    // 空元组 = unit type ()
    let _unit: () = ();

    // ---- 数组 ----
    let arr = [1, 2, 3, 4, 5];        // [i32; 5]
    let zeros = [0; 10];              // [i32; 10]，10 个 0
    let first = arr[0];
    println!("first={}, len={}", first, arr.len());

    // ---- 切片 ----
    let slice: &[i32] = &arr[1..4];   // [2, 3, 4]
    println!("slice={:?}, len={}", slice, slice.len());
}
输出
x=42, y=6.28, z=🦀
tup.0 = 42
first=1, len=5
slice=[2, 3, 4], len=3
🌐 工程实践
  • 函数返回多个值——用元组：fn divide(a: i32, b: i32) -> (i32, i32) { (a / b, a % b) }
  • 固定大小缓冲区——let buf: [u8; 1024] = [0; 1024]; 栈分配，零开销。
  • 参数传切片——fn sum(data: &[i32]) 比 fn sum(data: [i32; 5]) 更通用，能接受任意长度。
  • 超过 3 个字段——用命名 struct 而非元组，提高可读性。
✏️ 示例分析与易错点
RUSTslice_fn.rs
fn sum(data: &[i32]) -> i32 {
    let mut total = 0;
    for val in data {
        total += val;
    }
    total
}

fn main() {
    let arr = [10, 20, 30, 40, 50];
    println!("sum all:  {}", sum(&arr));        // 传整个数组的切片
    println!("sum 1..3: {}", sum(&arr[1..3])); // 传子切片 [20, 30]
}
输出
sum all:  150
sum 1..3: 50
易错：[T; N] 和 [T; M] 是不同类型（N ≠ M 时）。不能把 [i32; 3] 传给期望 [i32; 5] 的函数。用切片 &[i32] 消除长度约束。
⚡ 快速检测
let a = (42); 的类型是什么？

04Shadowing — 重新绑定的艺术

🎯 为什么需要了解这个

很多语言禁止同名变量重复声明，但 Rust 有意允许 shadowing（遮蔽）。这不是设计缺陷——它让你能在数据转换管道中复用变量名，同时保持每一步都是不可变绑定，兼具可读性和安全性。

🔍 核心思路

Shadowing 的本质：用 let 重新声明同名变量，创建全新的绑定，旧绑定被遮蔽。

对比Shadowingmut
类型改变✅ 可以❌ 不行
本质创建新绑定修改现有绑定的值
旧值被遮蔽（但仍在内存中直到作用域结束）被覆盖
适用场景类型转换管道原地累加/计数器
📐 规则与语法
RUSTshadowing.rs
fn main() {
    // 1. 值变化 + 类型不变
    let x = 5;
    let x = x + 1;   // 6
    let x = x * 2;   // 12
    println!("x = {}", x);

    // 2. 类型变化——mut 做不到！
    let spaces = "   ";         // &str
    let spaces = spaces.len();  // usize
    println!("spaces = {}", spaces);

    // 3. mut 不能改变类型
    // let mut s = "hello";
    // s = s.len(); // ❌ expected &str, found usize

    // 4. 内部作用域 shadowing
    let y = 10;
    {
        let y = y + 5;  // 内部 y = 15
        println!("inner y = {}", y);
    }
    println!("outer y = {}", y); // 外部 y = 10
}
输出
x = 12
spaces = 3
inner y = 15
outer y = 10
🌐 工程实践
  • 输入解析管道——let input = read_line(); let input = input.trim(); let input: i32 = input.parse().unwrap(); 每步都用 input，类型渐变。
  • 避免命名焦虑——不用想 input_str、input_trimmed、input_num。
  • 保持不可变——每步都是不可变绑定，编译器能更好地分析和优化。
✏️ 示例分析与易错点
RUSTparsing_pipeline.rs
use std::io;

fn main() {
    println!("请输入一个数字：");
    let mut input = String::new();
    io::stdin().read_line(&mut input).unwrap();

    // shadowing 管道：每步类型不同
    let input = input.trim();               // &str
    let input: f64 = input.parse()          // f64
        .expect("请输入有效数字");
    let input = (input * 100.0) as i64;     // i64（转为分）

    println!("转换为分：{}", input);
}
易错：Shadowing 遮蔽旧变量但不立即释放其内存（等到作用域结束）。一般来说编译器会优化掉不再使用的旧绑定，不需要担心。
⚡ 快速检测
以下代码能编译吗？
let mut x = "hello"; x = x.len();

05类型转换 — as、From/Into 与 TryFrom

🎯 为什么需要了解这个

Rust 没有隐式类型转换。每次类型不匹配都是编译错误。你需要显式选择转换方式——这看起来啰嗦，但它防止了整数溢出、精度丢失等隐蔽 bug。理解三种转换方式的安全性级别，是写出健壮代码的关键。

🔍 核心思路
方式安全性适用场景
as⚠️ 可能截断/溢出已知安全的 primitive 转换
From / Into✅ 编译期保证无损无损转换（i32 → i64）
TryFrom / TryInto✅ 返回 Result可能失败的转换（i64 → i32）
.parse()✅ 返回 Result字符串 → 数值
📐 规则与语法
RUSTconversion.rs
fn main() {
    // 1. as — 原始类型强制转换
    let x: i32 = 42;
    let y: i64 = x as i64;        // 安全：小→大
    let z: i8  = x as i8;         // 42 在 i8 范围内 OK，但更大值会截断

    let big: i32 = 300;
    let truncated: u8 = big as u8; // 300 % 256 = 44
    println!("300 as u8 = {}", truncated);

    // 2. From / Into — 安全无损
    let n: i64 = i64::from(42i32);   // From
    let m: i64 = 42i32.into();       // Into（编译器推断目标类型）
    println!("from={}, into={}", n, m);

    // 3. TryFrom / TryInto — 可能失败
    use std::convert::TryFrom;
    let big_val: i64 = 300;
    let result = i8::try_from(big_val);   // Err：超出范围
    println!("try_from 300→i8: {:?}", result);

    let small_val: i64 = 42;
    let ok = i8::try_from(small_val);     // Ok(42)
    println!("try_from 42→i8:  {:?}", ok);

    // 4. parse — 字符串转数值
    let pi: f64 = "3.14".parse().unwrap();
    println!("parsed: {}", pi);
}
输出
300 as u8 = 44
from=42, into=42
try_from 300→i8: Err(TryFromIntError(()))
try_from 42→i8:  Ok(42)
parsed: 3.14
🌐 工程实践
  • 优先用 From/Into——编译期保证无损。自定义类型也应该实现 From。
  • 外部输入用 TryFrom——网络、文件、用户输入等不可信数据。
  • as 仅在确定安全时——如 u8 as u32（小→大）。避免大→小的 as。
  • 实现 From 自动获得 Into——只需实现 From，标准库 blanket impl 自动提供 Into。
✏️ 示例分析与易错点
RUSTcustom_from.rs
struct Celsius(f64);
struct Fahrenheit(f64);

impl From<Celsius> for Fahrenheit {
    fn from(c: Celsius) -> Self {
        Fahrenheit(c.0 * 9.0 / 5.0 + 32.0)
    }
}

fn main() {
    let boiling = Celsius(100.0);
    let f: Fahrenheit = boiling.into(); // 自动获得 Into
    println!("100°C = {}°F", f.0);

    // 反向：Fahrenheit → Celsius 需要另外实现
    // let c: Celsius = f.into(); // ❌ 没有实现
}
输出
100°C = 212°F
易错：as 转换不会报错也不会 panic，悄悄截断。let x: i32 = 256; let y = x as u8; 得到 0，不是 256。在不确定范围时用 TryFrom。
