Unit 1 · 所有权系统

Ch 1.2 Borrowing

Ownership 解决了“谁负责资源”，Borrowing 则解决“我暂时想用它，但不想拿走它”。 这一章会把不可变借用、可变借用、借用范围和函数参数设计串成一个实用的 Rust 心智模型。

🎯 学习目标

  • 理解为什么 Rust 用借用而不是默认复制或共享可变状态
  • 区分不可变借用与可变借用的规则和使用场景
  • 读懂借用作用域与“最后一次使用”对借用检查的影响
  • 能为函数参数选择按值、&T 或 &mut T 的合适形式
  • 用“多个读者或一个写者”的模型解释常见借用报错

📋 前置知识

默认你已经知道 Ch 1.1 中的 owner、move、scope 和 Drop。 如果你还在混淆“值被 move 走了”和“值只是被借出去看一下”，先回顾 Ownership，再来看借用会更顺。

1.2.1 为什么 Rust 需要 Borrowing？先从不可变借用开始

🤔

你只是想读取一个 String 的长度，为什么还要把所有权整个交出去？函数读完以后，原变量难道不该继续可用吗？

Borrowing 的核心价值是：允许你临时使用某个值，而不接管它的所有权。 最常见的是不可变借用 &T，它表示“我只读，不改，也不负责释放”。

正式规则： 同一时刻可以有多个不可变借用，因为多个只读观察者不会改变资源状态。

Rustimmutable_borrow.rs
fn print_len(text: &String) {
    println!("len = {}", text.len());
}

fn main() {
    let name = String::from("Ferris");

    print_len(&name);
    println!("owner still usable: {name}");
}
输出 
len = 6
owner still usable: Ferris

和按值传参不同，这里 name 没有 move 进函数；函数只是暂时拿到对同一份数据的只读访问权。

1.2.2 可变借用：能改，但必须独占

✍️

如果两段代码同时都能修改同一个 String，你怎么保证修改顺序、状态一致性和引用有效性？

Rust 用 &mut T 表示可变借用。它给你写权限，但交换条件是独占访问： 同一时刻只能有一个可变借用，而且不能同时存在不可变借用。

记忆法： 借用检查器最常见的直觉不是“编译器很严格”，而是“同一份数据要么很多读者，要么一个写者”。

Rustmutable_borrow.rs
fn add_suffix(text: &mut String) {
    text.push_str(" language");
}

fn main() {
    let mut topic = String::from("Rust");
    add_suffix(&mut topic);
    println!("{topic}");
}
输出 
Rust language
因为两个可写入口意味着状态可能被交错修改，还可能让一个引用观察到另一个引用导致的重分配或失效。Rust 选择在编译期直接禁止这种别名写入。

1.2.3 借用作用域：看最后一次使用，不只看花括号

⏱️

很多初学者会以为“引用变量写在上面，就会一直借用到函数结束”。可为什么有些代码在后面又能重新拿到 &mut？

借用的有效范围要看它最后一次真正被使用的位置。这也是很多人第一次接触 borrow scope 时最容易忽略的点。 现代 Rust 的非词法生命周期（NLL）会尽量把借用结束点缩到最后一次使用处，而不是死板地拖到整段作用域末尾。

正式规则： 借用必须在后续冲突借用发生前结束；实际结束点通常由最后一次使用决定，而不只是变量声明位置。

Rustborrow_scope.rs
fn main() {
    let mut word = String::from("borrow");

    let view = &word;
    println!("view = {view}");

    let editable = &mut word;
    editable.push_str(" checker");

    println!("{word}");
}
输出 
view = borrow
borrow checker

这里先有不可变借用 view，但它在第一次 println! 后就不再被使用，所以后面的可变借用可以成立。

1.2.4 函数参数中的引用：API 在声明“要读、要改，还是要接管”

🧩

当你设计函数签名时，为什么 Rust 社区经常强调“能借用就先借用”，而不是一上来就收下整个 String？

函数签名本质上是在公开你的 ownership 决策： 按值接收表示你要拿走它，&T 表示只读借用， &mut T 表示临时修改。文本类参数如果只需要读取，优先用 &str 往往更灵活。

Rustfunction_refs.rs
fn shout(text: &str) {
    println!("{}", text.to_uppercase());
}

fn add_exclamation(text: &mut String) {
    text.push('!');
}

fn main() {
    let mut message = String::from("rust");

    shout(&message);
    add_exclamation(&mut message);

    println!("{message}");
}
输出 
RUST
rust!

实战建议： 如果函数只看字符串内容，不关心它是 String、字符串字面量还是切片，签名写成 &str 通常最通用。

1.2.5 借用检查器的心智模型：多个读者或一个写者

🛡️

面对“cannot borrow as mutable because it is also borrowed as immutable”时，真正该想的是哪条规则，而不是“编译器为什么又不让我写”？

一个实用心智模型是： owner 持有资源；引用只是临时许可证；许可证分为共享读和独占写；这些许可证不能形成会破坏一致性的重叠。 这样你看到报错时，就能先找“哪个读借用还活着、哪个写借用又想进来”。

常见错误： 先拿了不可变引用，又在它还会被使用时去做 push、clear、push_str 等可变操作。

Rustcommon_mistake.rs
fn main() {
    let mut numbers = vec![10, 20, 30];
    let first = &numbers[0];

    // numbers.push(40);
    // ^ 如果在这里修改，可能导致底层缓冲区重分配，
    //   那么 first 就可能指向失效位置，所以编译器会阻止。

    println!("first = {first}");
}
说明 
把 push 放在 first 最后一次使用之后，或者改成先修改再取引用，通常就能通过借用检查。
通常不是立刻 clone()，而是先重排代码顺序：让只读借用先结束，再做修改；或者先修改，再重新取得引用。只有在确实需要独立所有权时才考虑 clone。

📌 章节总结

  • Borrowing 让你临时使用值而不接管所有权，是 Ownership 的自然延伸。
  • &T 适合共享只读访问，&mut T 适合独占修改。
  • 借用冲突是否成立，要看借用是否在时间上真正重叠，而不只是变量声明位置。
  • 函数签名是在表达 ownership 决策：读用借用，改用可变借用，接管才按值。
  • 面对借用报错时，优先用“多个读者或一个写者”的模型定位冲突。
