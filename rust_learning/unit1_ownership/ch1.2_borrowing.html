Chapter 1.2

借用 · Borrowing

如果每次使用数据都要 move 所有权，编程会极其不便。借用（Borrowing）允许你临时访问数据而不获取所有权——就像图书馆借书：你可以阅读，但书还是图书馆的。Rust 的借用规则在编译期消除了数据竞争和悬垂引用。

01不可变引用 &T

🎯 为什么需要了解这个

上一章我们看到：把 String 传给函数后，调用者就不能再用了。但大多数时候，函数只需要读数据，不需要拥有它。不可变引用 &T 就是为此设计的——让函数借来看看，但不拿走。

🔍 核心思路

&T（不可变引用/共享引用）的两个关键属性：

  1. 不转移所有权：借用结束后，所有者仍然拥有数据
  2. 可以同时存在多个：多人同时阅读一本书没问题

代价：通过 &T 不能修改数据。这就是 "不可变" 的含义。

📐 规则与语法

基本语法：

RUSTshared_ref.rs
fn calculate_length(s: &String) -> usize {
    s.len()   // 只读访问，不获取所有权
}

fn main() {
    let greeting = String::from("hello");
    let len = calculate_length(&greeting); // & 创建引用
    // greeting 仍然有效！
    println!("'{}' 的长度是 {}", greeting, len);
}
输出
'hello' 的长度是 5

多个不可变引用可以共存：

RUSTmultiple_refs.rs
fn main() {
    let data = String::from("shared data");
    let r1 = &data;
    let r2 = &data;
    let r3 = &data;
    println!("{}, {}, {}", r1, r2, r3); // 全部有效
}
输出
shared data, shared data, shared data

不能通过 &T 修改数据：

RUSTimmutable_ref_error.rs
fn try_modify(s: &String) {
    // s.push_str(" world");  // ❌ 编译错误！
}
编译错误
error[E0596]: cannot borrow `*s` as mutable, as it is behind a `&` reference
 --> src/main.rs:2:5
  |
1 | fn try_modify(s: &String) {
  |                  ------- help: consider changing this to be a mutable reference: `&mut String`
2 |     s.push_str(" world");
  |     ^ `s` is a `&` reference, so the data it refers to cannot be borrowed as mutable
🌐 工程实践

惯用写法：函数参数优先使用引用而非获取所有权。

  • 只需要读 → &T
  • 需要修改 → &mut T（下节讲）
  • 需要拥有 → T（较少见）

更好的参数类型：&str 优于 &String（更通用）；&[T] 优于 &Vec<T>。

✏️ 示例分析与易错点

&str vs &String：

RUSTstr_vs_string.rs
// ❌ 过于限制
fn greet_bad(name: &String) {
    println!("Hello, {}!", name);
}

// ✅ 更通用：String, &str, 字面值都能传
fn greet_good(name: &str) {
    println!("Hello, {}!", name);
}

fn main() {
    let owned = String::from("Alice");
    greet_good(&owned);     // String → &str (自动 Deref)
    greet_good("Bob");      // &str 直接传

    // greet_bad("Bob");    // ❌ &str 不能传给 &String
}
输出
Hello, Alice!
Hello, Bob!
⚡ 快速检测
函数签名 fn foo(s: &String) 中，foo 能修改 s 吗？foo 返回后调用者还能用原 String 吗？

02可变引用 &mut T

🎯 为什么需要了解这个

只读引用不够——有时函数需要修改传入的数据。&mut T（可变引用）让你借用并修改数据，但 Rust 对它施加了严格约束：同一时刻只能有一个 &mut T。这个 "独占性" 是防数据竞争的关键。

🔍 核心思路

&mut T 的核心特性：

  1. 可以修改数据
  2. 独占性：同一时刻，对同一数据只能有一个 &mut T
  3. 不能和 &T 共存：有人在写，就不能有人在读

为什么这么设计？想象一个线程通过 &mut 在修改 Vec，另一个线程通过 & 在遍历——Vec 扩容可能导致内存重新分配，读者的指针瞬间悬垂。

📐 规则与语法

基本用法：

RUSTmut_ref.rs
fn add_exclaim(s: &mut String) {
    s.push_str("!");
}

fn main() {
    let mut greeting = String::from("hello");
    add_exclaim(&mut greeting);   // 可变借用
    println!("{}", greeting);     // 修改已生效，greeting 仍属于 main
}
输出
hello!

注意三个 mut 的位置：

  • let mut greeting — 变量本身可变
  • &mut greeting — 创建可变引用
  • s: &mut String — 函数参数接受可变引用

独占性约束：

RUSTtwo_mut_error.rs
fn main() {
    let mut s = String::from("hello");
    let r1 = &mut s;
    let r2 = &mut s;   // ❌
    println!("{}, {}", r1, r2);
}
编译错误
error[E0499]: cannot borrow `s` as mutable more than once at a time
 --> src/main.rs:4:14
  |
3 |     let r1 = &mut s;
  |              ------ first mutable borrow occurs here
4 |     let r2 = &mut s;
  |              ^^^^^^ second mutable borrow occurs here
5 |     println!("{}, {}", r1, r2);
  |                        -- first borrow later used here
🌐 工程实践

独占性的好处：

  • 线程安全：同一时刻只有一个写者，不可能有数据竞争
  • 迭代器安全：遍历 Vec 时不能同时修改它，避免迭代器失效
  • 推理简化：看到 &T 就知道数据不会变；看到 &mut T 就知道没有其他人在看

这些在 C/C++ 需要程序员靠自律维护的不变量，Rust 在编译期自动保证。

✏️ 示例分析与易错点

通过作用域或 NLL 化解冲突：

RUSTnll_rescue.rs
fn main() {
    let mut s = String::from("hello");

    // 方法1：使用作用域隔离
    {
        let r1 = &mut s;
        r1.push_str(" world");
    } // r1 在这里结束

    let r2 = &mut s;  // ✅ 没有冲突
    r2.push_str("!");
    println!("{}", s);

    // 方法2：NLL — 不需要显式作用域
    let mut v = vec![1, 2, 3];
    let first = &v[0];
    println!("first = {}", first);  // first 最后一次使用
    // NLL: first 的生命周期到此结束
    v.push(4);  // ✅ 可以修改了
    println!("{:?}", v);
}
输出
hello world!
first = 1
[1, 2, 3, 4]
⚡ 快速检测
为什么 Rust 禁止同时存在两个 &mut T？

03借用规则：Aliasing XOR Mutability

🎯 为什么需要了解这个

前两节讲了 &T 和 &mut T 的各自规则。这一节把它们统一成一条：在任意时刻，要么有多个 &T，要么有一个 &mut T，不能两者共存。理解这条规则，你就理解了 Rust 借用检查器 90% 的行为。

🔍 核心思路

正式规则：

在任意时刻，对同一数据：
  ✅ 任意数量的 &T（shared / aliased）
  ✅ 恰好一个 &mut T（exclusive）
  ❌ &T 和 &mut T 同时存在

用一个术语总结：Aliasing XOR Mutability——别名和可变性互斥。

这条规则的威力在于：它适用于单线程和多线程。即使在单线程中，aliasing + mutation 也能导致 bug（如迭代器失效）。

📐 规则与语法

经典的迭代器失效问题：

RUSTiterator_invalidation.rs
fn main() {
    let mut v = vec![1, 2, 3];

    // ❌ 在遍历时修改 — C++ 中是未定义行为，Rust 编译期拒绝
    // for &item in &v {
    //     if item == 2 {
    //         v.push(4);  // 需要 &mut v，但 &v 仍活跃
    //     }
    // }

    // ✅ 正确做法：先收集需要的信息，再修改
    let needs_push = v.iter().any(|&x| x == 2);
    if needs_push {
        v.push(4);
    }
    println!("{:?}", v);
}
输出
[1, 2, 3, 4]

编译器如何检查——理解 "活跃范围"：

RUSTactive_range.rs
fn main() {
    let mut s = String::from("hello");

    let r = &s;           // -- r 的生命周期开始
    println!("{}", r);    // -- r 最后一次使用
                           // -- NLL: r 的生命周期到此结束

    let m = &mut s;       // ✅ r 已经结束，不冲突
    m.push_str(" world");
    println!("{}", m);
}
输出
hello
hello world
🌐 工程实践

处理借用冲突的常见模式：

  • 重排代码：把读操作集中到修改之前，利用 NLL
  • 使用索引代替引用：v[i] 不保持借用
  • Clone：在性能不敏感的地方 clone 一份数据，避免借用冲突
  • 拆分 struct：如果 &mut self 的方法只改一个字段，可以拆成多个字段分别借用
✏️ 示例分析与易错点

HashMap 的借用冲突经典案例：

RUSThashmap_conflict.rs
use std::collections::HashMap;

fn main() {
    let mut map = HashMap::new();
    map.insert("a", 1);
    map.insert("b", 2);

    // ❌ 不能同时读和写
    // let val = map.get("a").unwrap();  // &map
    // map.insert("c", 3);               // &mut map
    // println!("{}", val);

    // ✅ 使用 entry API — 一次调用完成读+写
    map.entry("c").or_insert(3);
    println!("{:?}", map);

    // ✅ 或者先 clone 需要的值
    let val = map.get("a").copied().unwrap();
    map.insert("c", val + 10);
    println!("{:?}", map);
}
输出
{"a": 1, "b": 2, "c": 3}
{"a": 1, "b": 2, "c": 11}
⚡ 快速检测
以下哪个组合在同一时刻是合法的？

04引用必须始终有效

🎯 为什么需要了解这个

C/C++ 中最危险的 bug 之一就是 悬垂指针（dangling pointer）：指针指向的内存已经被释放。Rust 通过 borrow checker 在编译期保证：引用永远指向有效数据。

🔍 核心思路

引用有效性的核心原则：引用的生命周期不能超过被引用数据的生命周期。

用更直白的话说：数据必须活得比引用更久。如果数据先死了，引用就悬垂了——Rust 不允许这种情况存在。

📐 规则与语法

典型的悬垂引用——函数返回局部变量的引用：

RUSTdangling_ref.rs
// ❌ 不能返回局部变量的引用
fn dangle() -> &String {
    let s = String::from("hello");
    &s   // s 在函数结束时被 drop，引用指向无效内存
}

// ✅ 正确做法：返回 String（转移所有权）
fn no_dangle() -> String {
    let s = String::from("hello");
    s    // 所有权转移给调用者
}
编译错误（dangle 函数）
error[E0106]: missing lifetime specifier
 --> src/main.rs:1:16
  |
1 | fn dangle() -> &String {
  |                ^ expected named lifetime parameter
  |
  = help: this function's return type contains a borrowed value,
          but there is no value for it to be borrowed from

NLL (Non-Lexical Lifetimes)：

在 Rust 2018+ 中，引用的生命周期不再严格绑定到 {} 作用域，而是在最后一次使用处结束：

RUSTnll_demo.rs
fn main() {
    let mut data = vec![1, 2, 3];

    // 旧式生命周期 (Rust 2015)：r 活到 } → 和 push 冲突
    // NLL (Rust 2018+)：r 在最后使用处结束
    let r = &data[0];
    println!("first: {}", r);   // r 最后一次使用
    // NLL: r 到这里就结束了

    data.push(4);               // ✅ 没有活跃的 &data
    println!("{:?}", data);
}
输出
first: 1
[1, 2, 3, 4]
🌐 工程实践

常见的引用有效性问题及解决：

  • 函数返回引用：需要生命周期标注（下一章），或改为返回拥有所有权的类型
  • 闭包捕获引用：需要确保闭包不会活得比数据长。最简单的方案是 move 闭包
  • 结构体存储引用：需要生命周期参数（下一章详细讲）

经验法则：如果你在与 borrow checker 战斗，考虑是否应该用所有权（String 而非 &str）代替引用。在学习阶段，先让代码跑起来比 "零拷贝" 更重要。

✏️ 示例分析与易错点

闭包中的悬垂引用：

RUSTclosure_dangle.rs
fn make_greeter() -> impl Fn() {
    let name = String::from("Alice");
    // move 闭包获取 name 的所有权，避免悬垂引用
    move || {
        println!("Hello, {}!", name);
    }
    // 如果不用 move，name 在函数结束时被 drop，闭包持有悬垂引用
}

fn main() {
    let greet = make_greeter();
    greet();
    greet();
}
输出
Hello, Alice!
Hello, Alice!

条件分支中的引用生命周期：

RUSTbranch_lifetime.rs
fn main() {
    let s1 = String::from("long string");
    let result;
    {
        let s2 = String::from("short");
        // result = longest(&s1, &s2);  // ❌ s2 在 } 后被 drop
        //                                // result 可能指向已释放的 s2
        result = &s1;                    // ✅ s1 活得够久
    }
    println!("{}", result);
}
输出
long string
