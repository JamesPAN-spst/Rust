## 第二步：章节规划
### Ch 1.1 Ownership
- 目标：建立 Unit 1 的总心智模型，先讲清“为什么有所有权”，再解释 move / copy / clone / drop。
- 知识点规划：
  1. Ownership 的工程动机与唯一 owner 规则
  2. owner 与作用域的关系
  3. move 在赋值 / 传参中的语义
  4. Copy / Clone / Move 的区别
  5. Drop 与 RAII 式资源释放
- 快速测试方向：围绕“为什么不是隐式深拷贝、谁负责释放、什么时候还能继续使用旧变量”。

### Ch 1.2 Borrowing
- 目标：把 Borrowing 讲成 Ownership 的延伸，而不是零散限制。
- 知识点规划：
  1. 不可变借用的只读语义
  2. 可变借用的独占规则
  3. 借用作用域与最后一次使用
  4. 函数参数中的引用设计（`&str`、`&T`、`&mut T`）
  5. 借用检查器心智模型与常见冲突
- 快速测试方向：围绕“是否 move、何时可变、借用何时结束、签名怎么选、如何修复冲突报错”。

### Ch 1.3 Lifetimes
- 目标：把生命周期从抽象符号讲成“描述引用关系”的工具，解释为什么它能阻止悬垂引用。
- 知识点规划：
  1. lifetime 不是延寿魔法，而是关系标注
  2. dangling references 与编译期证明
  3. 函数签名中的生命周期约束
  4. lifetime elision 的适用边界
  5. struct 持有引用与常见错误
- 快速测试方向：围绕“返回引用来自谁、哪些场景能省略、何时该改成拥有型字段”。

### Ch 1.4 所有权的实际影响
- 目标：把所有权放回 API 设计、集合、迭代、错误处理和并发基础的工程场景中。
- 知识点规划：
  1. API 签名中的 ownership 决策
  2. 集合借用与重分配风险
  3. `iter` / `iter_mut` / `into_iter` 的语义区别
  4. 错误处理中的拥有型与借用型返回值取舍
  5. 线程边界、`move` 与重构优先级
- 快速测试方向：围绕“是否该接管数据、什么时候会消费容器、clone 该不该出现、线程为什么要 move”。
