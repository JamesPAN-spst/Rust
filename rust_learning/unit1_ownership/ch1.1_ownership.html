Unit 1 · 所有权系统

Ch 1.1 Ownership

Ownership 是 Rust 的底层叙事：值归谁管、谁能把它交给别人、谁最终负责释放资源。 本章不只是背规则，而是建立“为什么 Rust 宁可在编译期多问几句，也不把资源错误留到运行期”的直觉。

🎯 学习目标

  • 理解 Rust 为什么用 Ownership 解决内存安全与资源管理问题
  • 掌握 owner、scope、move、copy、clone、drop 之间的关系
  • 能解释赋值和传参为什么会触发所有权转移
  • 区分“值还能不能用”和“值有没有被复制”这两个不同问题
  • 把 Ownership 看成 API 设计和后续 Borrowing / Lifetimes 的前提

📋 前置知识

默认你已经能读懂变量绑定、函数调用、String 与整数的基本写法。 如果你还没有把“栈上值”和“堆上资源”的区别建立起来，本章会顺带给出直觉，但重点仍然是 Rust 如何规定资源归属。

1.1.1 为什么 Rust 要有 Ownership？

🤔

如果两个变量都以为自己拥有同一块堆内存，程序结束时谁来释放？如果两边都释放一次，会发生什么？

很多语言把资源释放交给垃圾回收器，或者交给程序员手动管理。Rust 选择第三条路： 在编译期把资源归属写进语言规则。这样做的目标不是“显得严格”，而是让双重释放、悬垂引用和数据竞争这类问题尽量在运行前暴露。

正式规则：每个值在任意时刻只有一个 owner；owner 离开作用域时，值会被清理。

Rustwhy_ownership.rs
fn main() {
    let language = String::from("Rust");
    println!("{language}");
}
说明 
这里的 String 由 language 拥有；main 结束时，Rust 会自动释放它占用的资源。

1.1.2 owner、scope 与“谁负责收尾”

📦

Rust 里的变量只是名字，还是在承担更具体的职责？为什么离开花括号就会触发资源释放？

在 Rust 里，绑定不仅是“给值起个名字”，还是“当前谁拥有它”的声明。 这个 owner 和作用域绑定在一起：当 owner 离开作用域，编译器就知道资源生命周期到头了。

Rustscope.rs
fn main() {
    {
        let message = String::from("scoped");
        println!("inside: {message}");
    }

    // println!("{message}");
    // 编译错误：message 已经离开作用域
}
输出 
inside: scoped

这里不是“变量名看不见了”这么简单，而是它拥有的资源也已经按规则被释放。

因为它给编译器一个稳定、可静态分析的边界：只要 owner 不再可达，资源就能被安全回收，不需要额外的运行时追踪。

1.1.3 move：为什么赋值不是偷偷深拷贝？

➡️

把一个 String 赋值给另一个变量后，如果两个变量都还能继续用，谁才应该负责释放底层堆内存？

对拥有资源的类型，Rust 默认用 move 处理赋值和按值传参。 它的含义不是“复制了一份”，而是“拥有权从旧绑定转移到新绑定”。这样语言就能继续维持“同一时刻只有一个 owner”的规则。

设计动机： 如果默认深拷贝，很多赋值都会隐式触发堆分配和复制成本；Rust 选择把昂贵复制变成显式行为。

Rustmove.rs
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;

    println!("s2 = {s2}");
    // println!("s1 = {s1}");
    // 编译错误：borrow of moved value
}
输出 
s2 = hello

1.1.4 Copy、Clone、Move：三个词在回答三个不同问题

🧠

为什么整数赋值后两个变量都能继续用，而 String 却不行？clone() 又是在补救什么？

Move 解决“拥有权转给谁”；Copy 解决“这个值是否足够轻量，可以按位复制且原值继续有效”； Clone 解决“我确实要复制一份独立资源，而且愿意显式承担成本”。

Rustcopy_clone.rs
fn main() {
    let x = 42;
    let y = x;
    println!("x = {x}, y = {y}");

    let a = String::from("rust");
    let b = a.clone();
    println!("a = {a}, b = {b}");
}
输出 
x = 42, y = 42
a = rust, b = rust

常见误区：clone() 不是“看到报错就按一下”的按钮。先问自己：我要的是独立所有权，还是只想借用读取？

1.1.5 Drop：所有权规则最终怎样释放资源？

🧹

语言知道了 owner 是谁之后，具体在什么时候、通过什么机制把文件句柄、锁或堆内存释放掉？

Rust 把清理逻辑放进 Drop。当 owner 离开作用域时，值的 drop 逻辑会自动运行。 这让资源释放遵循同一套作用域规则，而不是靠程序员记住每一条释放路径。

Rustdrop.rs
struct LogOnDrop(&'static str);

impl Drop for LogOnDrop {
    fn drop(&mut self) {
        println!("dropping {}", self.0);
    }
}

fn main() {
    let _guard = LogOnDrop("resource");
    println!("working...");
}
输出 
working...
dropping resource
RAII 的核心是“资源生命周期绑定到对象生命周期”。Rust 借助 Ownership 和 Drop，把 RAII 做成默认语义：对象离开作用域，资源就自动清理。

📌 章节总结

  • Ownership 的本质是用语言规则明确“谁拥有资源、何时释放资源”。
  • owner 与作用域绑定，因此值的可用性和清理时机都可静态判断。
  • 对拥有堆资源的类型，赋值和按值传参通常意味着 move，而不是隐式深拷贝。
  • Copy 适合轻量值，Clone 是显式复制资源，Drop 负责清理收尾。
  • 后续 Borrowing 和 Lifetimes，都是在 Ownership 这个大前提下讨论“如何临时使用而不接管”。
