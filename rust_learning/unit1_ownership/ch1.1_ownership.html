Chapter 1.1

所有权 · Ownership

所有权是 Rust 最核心的创新：每个值有且仅有一个所有者，所有者离开作用域时值被自动释放。 这条简单规则在编译期消除了 use-after-free、double free 等经典内存错误——不需要 GC，零运行时开销。

01为什么 Rust 要有 Ownership

🎯 为什么需要了解这个

内存管理是系统编程的头号杀手。C/C++ 的手动管理导致 use-after-free（微软安全漏洞的 70%）、double free、内存泄漏；GC 语言（Java/Go/Python）解决了安全问题，但带来 运行时暂停 和 额外内存开销。

Rust 想要两全其美：编译期保证安全 + 运行时零开销。所有权就是实现这个目标的核心机制。

🔍 核心思路

三条所有权规则（记住这三条，其余全是推论）：

  1. 每个值有且仅有一个所有者（owner）
  2. 值在任一时刻只能有一个所有者（ownership 可以转移，但不能共享）
  3. 当所有者离开作用域，值被自动释放（drop）

这三条规则让编译器在编译期就能确定每块内存何时分配、何时释放——不需要 GC，不需要手动 free。

📐 规则与语法

问题场景：C 的 double free

Cdouble_free.c
char *s = malloc(6);
strcpy(s, "hello");
char *t = s;      // t 和 s 指向同一块内存
free(s);           // 释放一次
free(t);           // 💥 double free — 未定义行为！

Rust 的解法：赋值 = 转移所有权

RUSTownership_basic.rs
fn main() {
    let s = String::from("hello");   // s 是 "hello" 的所有者
    let t = s;                        // 所有权从 s 转移到 t
    // println!("{}", s);             // ❌ 编译错误！s 已经不拥有数据
    println!("{}", t);                // ✅ t 是合法的所有者
}   // t 离开作用域 → drop → 释放堆内存（只释放一次）
输出
hello

作用域与自动释放：编译器在 } 处插入 drop(t) 调用，释放堆内存。由于 s 已经失效，不会释放两次。

🌐 工程实践

对比三种内存管理策略：

策略代表语言安全性性能心智负担
手动管理C/C++❌ 依赖程序员✅ 零开销极高
垃圾回收Java/Go/Python✅ 自动安全⚠️ GC 暂停低
所有权系统Rust✅ 编译期保证✅ 零开销中等（需学习）

所有权的心智负担集中在学习阶段。一旦形成直觉，编译器会帮你防错——这正是 Rust "如果能编译，就不会有内存错误" 的核心保证。

✏️ 示例分析与易错点

函数传参也会 move：

RUSTmove_to_function.rs
fn take_ownership(s: String) {
    println!("got: {}", s);
}   // s 在这里被 drop

fn main() {
    let greeting = String::from("hi");
    take_ownership(greeting);
    // println!("{}", greeting);   // ❌ 编译错误 E0382
}
编译错误
error[E0382]: borrow of moved value: `greeting`
 --> src/main.rs:8:20
  |
6 |     let greeting = String::from("hi");
  |         -------- move occurs because `greeting` has type `String`
7 |     take_ownership(greeting);
  |                    -------- value moved here
8 |     println!("{}", greeting);
  |                    ^^^^^^^^ value borrowed here after move

解决方案：如果后续还要用，传引用 &greeting；或让函数返回 String 归还所有权；或 .clone()。

02Move 语义

🎯 为什么需要了解这个

在其他语言中，b = a 要么是复制值（C struct），要么是复制引用（Java/Python）。Rust 的默认行为是 move：b = a 将所有权从 a 转移到 b，a 不再可用。理解 move 是理解 Rust 一切行为的基础。

🔍 核心思路

Move 的本质：在底层，move 实际上是 memcpy 栈帧上的字节 + 使原变量失效。对于 String，栈上存储的是 (ptr, len, capacity) 三个 usize——move 只复制这 24 字节，堆上的实际数据没有被复制。

这就是 Rust 的关键洞察：只转移「持有者元数据」，不复制数据本身。

📐 规则与语法

栈上数据 vs 堆上数据：

RUSTstack_vs_heap.rs
fn main() {
    // ── 栈上数据：Copy ──
    let x: i32 = 42;
    let y = x;          // 按位复制（i32 实现了 Copy）
    println!("x={}, y={}", x, y); // 两个都能用

    // ── 堆上数据：Move ──
    let s1 = String::from("hello");  // s1 拥有堆上的 "hello"
    let s2 = s1;                      // s1 的 (ptr, len, cap) 复制到 s2
                                      // s1 被标记为无效
    // println!("{}", s1);            // ❌ 编译错误
    println!("{}", s2);               // ✅ 只有 s2 是合法所有者
}
输出
x=42, y=42
hello

String 的内存模型：

move 前：                        move 后：
  s1 [ptr|5|5] ──→ [h,e,l,l,o]    s1 [无效]
                                    s2 [ptr|5|5] ──→ [h,e,l,l,o]

堆上数据没有复制——这就是 move 高效的原因。

🌐 工程实践

Move 发生的场景（牢记这几个）：

  • let b = a; — 变量赋值
  • foo(a) — 函数传参（函数签名 fn foo(s: String)）
  • vec.push(a) — 放入容器
  • return a; — 函数返回（所有权转移给调用者）

避免意外 move 的方式：传引用 &a；对需要多处使用的数据 .clone()。

✏️ 示例分析与易错点

返回值转移所有权：

RUSTreturn_ownership.rs
fn create_greeting(name: &str) -> String {
    let s = format!("Hello, {}!", name);
    s   // 所有权从函数内转移到调用者
}

fn main() {
    let msg = create_greeting("Rustacean");
    println!("{}", msg);
}
输出
Hello, Rustacean!

常见错误：在循环中意外 move

RUSTloop_move.rs
fn consume(s: String) { println!("{}", s); }

fn main() {
    let name = String::from("Alice");
    for _ in 0..3 {
        // consume(name);  // ❌ 第二次迭代时 name 已被 move
        consume(name.clone()); // ✅ 每次迭代 clone 一份
    }
    println!("原始: {}", name); // name 仍然有效
}
输出
Alice
Alice
Alice
原始: Alice

03Copy Trait

🎯 为什么需要了解这个

如果所有赋值都 move，连 let y = x;（x 是 i32）之后都不能用 x，写代码会极其痛苦。Copy trait 是 Rust 给简单类型的特殊通行证：赋值时自动按位复制，原变量仍然有效。

🔍 核心思路

何时 Copy，何时 Move？

  • 实现了 Copy trait 的类型：赋值 = 按位复制，两个副本都有效
  • 没有实现 Copy 的类型：赋值 = move，原变量失效

Copy 的判定标准：类型的 "复制" 只需要 memcpy 就够了——不涉及堆分配、文件句柄等需要析构逻辑的资源。

📐 规则与语法

默认 Copy 的类型：

  • 所有整数类型：i8, i16, i32, i64, i128, isize, u8 ... usize
  • 浮点数：f32, f64
  • bool、char
  • 元组（仅当所有元素都 Copy）：(i32, f64) ✅、(i32, String) ❌
  • 数组（仅当元素 Copy）：[i32; 3] ✅
  • 共享引用 &T（不管 T 是否 Copy）

不 Copy 的类型：String、Vec<T>、Box<T>、HashMap 等——它们管理堆上资源。

为自定义类型实现 Copy：

RUSTderive_copy.rs
#[derive(Debug, Copy, Clone)]
struct Point {
    x: f64,
    y: f64,
}

fn main() {
    let p1 = Point { x: 1.0, y: 2.0 };
    let p2 = p1;     // Copy! p1 和 p2 都有效
    println!("p1={:?}, p2={:?}", p1, p2);
}
输出
p1=Point { x: 1.0, y: 2.0 }, p2=Point { x: 1.0, y: 2.0 }

Copy vs Clone：

RUSTcopy_vs_clone.rs
fn main() {
    // Copy：隐式，赋值时自动发生
    let a: i32 = 10;
    let b = a;        // 隐式 copy

    // Clone：显式，需要调用 .clone()
    let s1 = String::from("hello");
    let s2 = s1.clone();   // 显式深拷贝
    println!("s1={}, s2={}", s1, s2); // 两个都有效
}
输出
s1=hello, s2=hello
🌐 工程实践

何时该 derive(Copy)？

  • ✅ 小型值类型（坐标、颜色、配置标志）：复制成本极低
  • ❌ 不要给含有 String、Vec 等字段的类型加 Copy（编译器也不允许）
  • ⚠️ 大 struct（例如 [f64; 1000]）虽然可以 Copy，但复制成本高，考虑传引用

经验法则：如果不确定，先不加 Copy。后期从 non-Copy 改成 Copy 是向后兼容的；反过来则是破坏性变更。

✏️ 示例分析与易错点

包含 String 字段不能 Copy：

RUSTcannot_copy.rs
#[derive(Copy, Clone)]  // ❌ 编译错误！
struct User {
    name: String,       // String 不是 Copy
    age: u32,
}
编译错误
error[E0204]: the trait `Copy` may not be implemented for this type
 --> src/main.rs:1:10
  |
1 | #[derive(Copy, Clone)]
  |          ^^^^
3 |     name: String,
  |     ------------ this field does not implement `Copy`

解决方案：只 derive(Clone)，需要时用 .clone() 显式复制。

04Drop Trait 与 RAII

🎯 为什么需要了解这个

如果 Rust 只有 "所有者离开作用域自动释放"，那 i32 和 String 的释放逻辑一样吗？当然不一样——String 需要释放堆内存，File 需要关闭文件句柄。Drop trait 就是告诉 Rust "这个类型被丢弃时需要执行什么清理逻辑" 的机制。

🔍 核心思路

RAII（Resource Acquisition Is Initialization）：

  • 资源（内存、文件、锁……）的获取绑定到变量的创建
  • 资源的释放绑定到变量的销毁（离开作用域）

C++ 发明了 RAII，但 C++ 的拷贝语义让它容易出错。Rust 通过所有权 + Drop 实现了更安全的 RAII：每个资源恰好被释放一次，在编译期保证。

📐 规则与语法

实现 Drop trait：

RUSTcustom_drop.rs
struct DatabaseConn {
    url: String,
}

impl Drop for DatabaseConn {
    fn drop(&mut self) {
        println!("🔌 关闭数据库连接: {}", self.url);
    }
}

fn main() {
    let conn = DatabaseConn {
        url: String::from("postgres://localhost/mydb"),
    };
    println!("连接已建立");
    // conn 在这里离开作用域 → 自动调用 drop
}
输出
连接已建立
🔌 关闭数据库连接: postgres://localhost/mydb

Drop 的执行顺序：

RUSTdrop_order.rs
struct Guard(i32);

impl Drop for Guard {
    fn drop(&mut self) {
        println!("dropping Guard({})", self.0);
    }
}

fn main() {
    let a = Guard(1);
    let b = Guard(2);
    let c = Guard(3);
    println!("所有 Guard 已创建");
    // drop 顺序：c → b → a（后创建的先销毁，像栈一样 LIFO）
}
输出
所有 Guard 已创建
dropping Guard(3)
dropping Guard(2)
dropping Guard(1)

提前释放：std::mem::drop

RUSTearly_drop.rs
fn main() {
    let data = String::from("重要数据");
    println!("使用数据: {}", data);
    drop(data);  // 提前释放
    println!("data 已释放，后续可做其他事");
    // 这里不能再用 data
}
输出
使用数据: 重要数据
data 已释放，后续可做其他事
🌐 工程实践

Rust 标准库中的 RAII 示例：

  • String、Vec：drop 时释放堆内存
  • File：drop 时关闭文件描述符
  • MutexGuard：drop 时自动释放锁——不会忘记解锁！
  • Box<T>：drop 时释放堆上的 T

与 C++ 的对比：C++ 也有 RAII（析构函数），但 C++ 允许拷贝构造产生多个 "所有者"，容易出问题。Rust 通过 move 语义确保只有一个所有者，RAII 更加可靠。

✏️ 示例分析与易错点

不能显式调用 .drop()：

RUSTexplicit_drop_error.rs
fn main() {
    let s = String::from("hello");
    // s.drop();   // ❌ 编译错误！不能显式调用 drop
    drop(s);       // ✅ 使用 std::mem::drop
}
编译错误（如果用 s.drop()）
error[E0040]: explicit use of destructor method
 --> src/main.rs:3:7
  |
3 |     s.drop();
  |       ^^^^ explicit destructor calls not allowed

为什么禁止 .drop()？ 如果允许，编译器在作用域结束时还会再调一次 drop——双重释放。std::mem::drop(s) 通过吞掉 s 的所有权来安全地触发析构。

Copy 和 Drop 互斥：

RUSTcopy_drop_conflict.rs
#[derive(Copy, Clone)]
struct Oops(i32);

impl Drop for Oops {   // ❌ 编译错误！
    fn drop(&mut self) {
        println!("dropping {}", self.0);
    }
}
编译错误
error[E0184]: the trait `Copy` may not be implemented for this type;
              the type has a destructor

如果一个值可以 Copy，那每个副本都会 drop——对有析构逻辑的类型这是危险的。所以 Rust 禁止同时实现 Copy 和 Drop。
