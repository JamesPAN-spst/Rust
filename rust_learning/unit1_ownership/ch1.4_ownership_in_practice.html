Chapter 1.4

所有权的实际影响

前三章建立了所有权、借用、生命周期的理论基础。这一章拉回现实：这些规则在实际项目中消除了哪些经典 bug？ 与 C++ 智能指针相比有何优势？在日常开发中应该使用什么模式？

01为什么没有双重释放

🎯 为什么需要了解这个

Double free 是 C/C++ 中最臭名昭著的漏洞之一——释放同一块内存两次会导致堆损坏、security exploit 甚至 remote code execution。让我们精确理解 Rust 是如何在编译期消除这个问题的。

🔍 核心思路

Double free 的前提条件：两个变量/指针指向同一块内存，各自尝试释放一次。

Rust 的解法：所有权唯一性（每个值只有一个所有者）+ move 语义（赋值后原变量失效）= 不可能有两个 "所有者" 各自释放同一块内存。

📐 规则与语法

逐步推演：

RUSTno_double_free.rs
fn main() {
    let s1 = String::from("hello");  // s1 是所有者
    let s2 = s1;                      // 所有权 move 到 s2；s1 失效

    // 编译器在此处插入的 drop:
    // drop(s2);  ← 只有 s2 被 drop（一次）
    // s1 已经不是所有者，没有 drop

    // 如果 s1 没被 move：
    // drop(s1);  ← s1 是所有者，drop 一次
    // 无论如何，每块内存恰好被 drop 一次
}

对比 C++：

C++double_free_cpp.cpp
void danger() {
    auto p1 = new std::string("hello");
    auto p2 = p1;   // C++ 不阻止：两个裸指针指向同一块内存
    delete p1;        // 释放一次
    delete p2;        // 💥 double free — 未定义行为
}
// C++11 的 unique_ptr 解决了一部分，但 p.get() 仍可产生裸指针

// Rust 等价代码不可能编译通过：
// let s1 = String::from("hello");
// let s2 = s1;
// drop(s1);  // ❌ 编译错误：s1 已 moved
🌐 工程实践

现实中的影响：微软安全团队报告，约 70% 的 CVE 漏洞源于内存安全问题（use-after-free, double free, buffer overflow）。Rust 在编译期消除了其中的大部分。

这就是为什么 Linux 内核、Windows 驱动、Android 的部分核心组件开始用 Rust 重写。

✏️ 示例分析与易错点

即使用了 drop() 也不会 double free：

RUSTdrop_safety.rs
fn main() {
    let s = String::from("hello");
    drop(s);           // 手动释放
    // println!("{}", s);  // ❌ 编译错误：s 已被 move 进 drop()
    // 不可能再释放一次
}
编译错误
error[E0382]: borrow of moved value: `s`
 --> src/main.rs:4:20
  |
2 |     let s = String::from("hello");
  |         - move occurs because `s` has type `String`
3 |     drop(s);
  |          - value moved here
4 |     println!("{}", s);
  |                    ^ value borrowed here after move

02为什么没有数据竞争

🎯 为什么需要了解这个

数据竞争（data race）是并发编程中最难调试的 bug——两个线程同时访问同一数据，至少一个在写。它导致不可预测的结果，且几乎不可复现。Rust 在编译期就消灭了数据竞争。

🔍 核心思路

数据竞争的三个必要条件（全部满足才会发生）：

  1. 两个或更多线程同时访问同一内存
  2. 至少一个线程在写
  3. 没有同步机制

Rust 如何打破条件 1+2：借用规则（aliasing XOR mutability）在编译期保证——如果有人在写，就不可能有人在读或写。这在单线程中由编译器直接检查；在多线程中通过 Send/Sync trait 和类型系统保证。

📐 规则与语法

线程间共享数据需要 Arc + Mutex：

RUSTthread_safe.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    // Arc: 线程安全的引用计数（多个线程共享所有权）
    // Mutex: 互斥锁（同一时刻只有一个线程能访问内部数据）
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter); // 增加引用计数
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap(); // 获取锁
            *num += 1;
            // MutexGuard drop → 自动释放锁
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("结果: {}", *counter.lock().unwrap());
}
输出
结果: 10

编译器拒绝不安全的共享：

RUSTunsafe_share.rs
use std::thread;

fn main() {
    let mut data = vec![1, 2, 3];

    // ❌ 编译器拒绝：闭包捕获了 &mut data，但线程可能比 data 活得更久
    // thread::spawn(|| {
    //     data.push(4);
    // });

    // ✅ 用 move 转移所有权
    let handle = thread::spawn(move || {
        data.push(4);
        println!("{:?}", data);
    });
    // data 已经 move，main 不能再用
    handle.join().unwrap();
}
输出
[1, 2, 3, 4]
🌐 工程实践

Send 和 Sync 的角色：

  • Send：类型可以安全地 move 到另一个线程（大部分类型 Send，Rc<T> 不 Send）
  • Sync：&T 可以安全地在多个线程间共享（大部分类型 Sync，Cell<T> 不 Sync）
  • 这两个 trait 是自动实现的（auto trait），编译器根据字段类型自动判断

如果你的类型不满足 Send/Sync，编译器不允许你跨线程传递——这就是 "无畏并发" 的核心机制。

✏️ 示例分析与易错点

Rc 不能跨线程：

RUSTrc_not_send.rs
use std::rc::Rc;
use std::thread;

fn main() {
    let data = Rc::new(vec![1, 2, 3]);

    // ❌ 编译错误：Rc> 不是 Send
    // let data_clone = Rc::clone(&data);
    // thread::spawn(move || {
    //     println!("{:?}", data_clone);
    // });

    // ✅ 改用 Arc
    use std::sync::Arc;
    let data = Arc::new(vec![1, 2, 3]);
    let data_clone = Arc::clone(&data);
    thread::spawn(move || {
        println!("{:?}", data_clone); // ✅ Arc 是 Send
    }).join().unwrap();
}
输出
[1, 2, 3]

03与 C++ 智能指针的对比

🎯 为什么需要了解这个

C++11 引入了 unique_ptr、shared_ptr、weak_ptr 来管理内存。Rust 的 Box、Rc/Arc 看起来类似——但 Rust 的编译器保证了 C++ 做不到的安全性。理解差异有助于 C++ 开发者迅速建立 Rust 心智模型。

🔍 核心思路
C++Rust关键区别
unique_ptr<T>Box<T>C++ 的 move 后 unique_ptr 变成 null（可以使用但是 UB）；Rust 的 Box move 后原变量不可用（编译错误）
shared_ptr<T>Rc<T> / Arc<T>C++ 不区分单/多线程版本；Rust 编译期强制区分
weak_ptr<T>Weak<T>功能类似，Rust 的 upgrade 返回 Option（无 null）
T*（裸指针）*const T / *mut TRust 裸指针只能在 unsafe 中解引用
📐 规则与语法

Box vs unique_ptr：

RUSTbox_demo.rs
fn main() {
    // Box: 唯一所有权的堆分配
    let b1 = Box::new(42);
    let b2 = b1;          // move! b1 失效
    // println!("{}", b1); // ❌ 编译错误

    println!("{}", b2);    // ✅
    // b2 离开作用域 → 堆内存被释放
}

// C++ 对比：
// auto p1 = std::make_unique(42);
// auto p2 = std::move(p1);
// // p1 现在是 nullptr — 但编译器不阻止你用 *p1（UB！）
输出
42

Rc vs shared_ptr：

RUSTrc_demo.rs
use std::rc::Rc;

fn main() {
    let a = Rc::new(String::from("shared data"));
    println!("引用计数: {}", Rc::strong_count(&a)); // 1

    let b = Rc::clone(&a);  // 增加引用计数，不复制数据
    println!("引用计数: {}", Rc::strong_count(&a)); // 2

    {
        let c = Rc::clone(&a);
        println!("引用计数: {}", Rc::strong_count(&a)); // 3
    } // c drop → 计数 -1

    println!("引用计数: {}", Rc::strong_count(&a)); // 2
    println!("数据: {}", a);
}
输出
引用计数: 1
引用计数: 2
引用计数: 3
引用计数: 2
数据: shared data
🌐 工程实践

选择指南：

  • Box<T>：单一所有者、堆分配。用于递归类型、大数据上栈、trait 对象（Box<dyn Trait>）
  • Rc<T>：多个所有者、单线程。用于图结构、共享配置数据
  • Arc<T>：多个所有者、多线程。通常搭配 Mutex 或 RwLock
  • 直接值或引用：优先考虑。只有在所有权关系复杂时才用智能指针
✏️ 示例分析与易错点

Rc 的循环引用问题（和 shared_ptr 一样）：

RUSTrc_cycle.rs
use std::cell::RefCell;
use std::rc::Rc;

#[derive(Debug)]
struct Node {
    next: Option>>,
}

fn main() {
    let a = Rc::new(RefCell::new(Node { next: None }));
    let b = Rc::new(RefCell::new(Node { next: Some(Rc::clone(&a)) }));

    // 创建循环引用：a → b → a
    a.borrow_mut().next = Some(Rc::clone(&b));

    // ⚠️ 内存泄漏！引用计数永远不会归零
    // 解决方案：在适当的地方使用 Weak 代替 Rc

    println!("a count: {}", Rc::strong_count(&a)); // 2
    println!("b count: {}", Rc::strong_count(&b)); // 2
}
输出
a count: 2
b count: 2

注意：Rust 的所有权系统不能防止 Rc 循环引用导致的内存泄漏（和 C++ shared_ptr 一样）。解决方案是使用 Weak<T>。

⚡ 快速检测

04所有权模式在实际项目中的体现

🎯 为什么需要了解这个

所有权不仅是内存管理工具，它塑造了 Rust 独特的 API 设计风格和编程模式。理解这些模式能帮你写出更地道、更安全的 Rust 代码。

🔍 核心思路

所有权影响的核心设计抉择：

  • 函数接口：参数是 T、&T 还是 &mut T？
  • 数据结构：存 owned 数据还是引用？
  • 何时 clone：安全性 vs 性能的权衡
  • 设计模式：Builder、Typestate、RAII Guard
📐 规则与语法

模式 1: Builder 模式（所有权链式调用）

RUSTbuilder_pattern.rs
struct RequestBuilder {
    url: String,
    method: String,
    headers: Vec<(String, String)>,
}

impl RequestBuilder {
    fn new(url: &str) -> Self {
        RequestBuilder {
            url: url.to_string(),
            method: "GET".to_string(),
            headers: vec![],
        }
    }

    // self（不是 &self）→ 获取所有权 → 返回修改后的 self
    fn method(mut self, m: &str) -> Self {
        self.method = m.to_string();
        self
    }

    fn header(mut self, key: &str, value: &str) -> Self {
        self.headers.push((key.to_string(), value.to_string()));
        self
    }

    fn build(self) -> String {
        format!("{} {} {:?}", self.method, self.url, self.headers)
    }
}

fn main() {
    let request = RequestBuilder::new("https://api.example.com")
        .method("POST")
        .header("Content-Type", "application/json")
        .header("Authorization", "Bearer token123")
        .build();

    println!("{}", request);
}
输出
POST https://api.example.com [("Content-Type", "application/json"), ("Authorization", "Bearer token123")]

模式 2: 类型状态模式（Typestate）

RUSTtypestate.rs
// 不同状态用不同类型表示——编译期保证状态机正确性
struct Locked;
struct Unlocked;

struct Door {
    _state: std::marker::PhantomData,
}

impl Door {
    fn unlock(self) -> Door {
        println!("🔓 Door unlocked");
        Door { _state: std::marker::PhantomData }
    }
    // 不能 open Locked 门！方法根本不存在
}

impl Door {
    fn open(&self) {
        println!("🚪 Door opened");
    }

    fn lock(self) -> Door {
        println!("🔒 Door locked");
        Door { _state: std::marker::PhantomData }
    }
}

fn main() {
    let door: Door = Door { _state: std::marker::PhantomData };
    // door.open();        // ❌ 编译错误！Locked 状态没有 open 方法
    let door = door.unlock(); // move: 旧的 Locked 门消失
    door.open();              // ✅ Unlocked 状态有 open 方法
    let _door = door.lock();  // move: 回到 Locked 状态
}
输出
🔓 Door unlocked
🚪 Door opened
🔒 Door locked
🌐 工程实践

何时使用 clone：

  • ✅ 数据小（i32、小 struct）→ 直接 Copy/Clone
  • ✅ 原型开发阶段 → 先 clone 让代码跑起来，后面再优化
  • ✅ 生命周期关系过于复杂 → clone 比引入 'a 标注更清晰
  • ⚠️ 大数据（长 String、大 Vec）→ 考虑引用、Rc/Arc 或 Cow
  • ❌ 热循环中 clone 大数据 → 性能杀手，用引用代替

clippy 的帮助：cargo clippy 会标记不必要的 .clone()，帮你在代码稳定后优化。

✏️ 示例分析与易错点

RAII Guard 模式（实际项目中最常见）：

RUSTraii_guard.rs
use std::time::Instant;

struct Timer {
    label: String,
    start: Instant,
}

impl Timer {
    fn new(label: &str) -> Self {
        println!("⏱️ [{}] started", label);
        Timer {
            label: label.to_string(),
            start: Instant::now(),
        }
    }
}

impl Drop for Timer {
    fn drop(&mut self) {
        let elapsed = self.start.elapsed();
        println!("⏱️ [{}] finished in {:?}", self.label, elapsed);
    }
}

fn main() {
    {
        let _t = Timer::new("heavy computation");
        // 模拟一些工作
        let sum: u64 = (0..1_000_000).sum();
        println!("sum = {}", sum);
    } // _t drop → 自动打印耗时

    println!("timer already cleaned up");
}
输出
⏱️ [heavy computation] started
sum = 499999500000
⏱️ [heavy computation] finished in 1.23ms
timer already cleaned up

函数签名就是文档：

RUSTsignature_as_doc.rs
// 只需要读 → &str
fn word_count(text: &str) -> usize {
    text.split_whitespace().count()
}

// 需要修改 → &mut Vec
fn deduplicate(items: &mut Vec) {
    items.sort();
    items.dedup();
}

// 需要拥有 → String（消耗输入）
fn into_uppercase(s: String) -> String {
    s.to_uppercase()
}

fn main() {
    let text = String::from("hello world hello");
    println!("words: {}", word_count(&text)); // text 仍然有效

    let mut items = vec!["b".into(), "a".into(), "b".into()];
    deduplicate(&mut items);
    println!("deduped: {:?}", items);

    let upper = into_uppercase(text); // text 被 move
    // println!("{}", text);  // ❌ text 已被消耗
    println!("{}", upper);
}
输出
words: 3
deduped: ["a", "b"]
HELLO WORLD HELLO
