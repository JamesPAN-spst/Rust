Unit 1 · 所有权系统

Ch 1.4 所有权的实际影响

所有权不只是“编译会不会报错”的基础规则，它会持续影响你怎样设计 API、怎样遍历集合、怎样组织错误处理， 以及为什么 Rust 的并发模型能把很多数据竞争问题提前拦住。本章把 Ownership 放回真实工程语境里看。

🎯 学习目标

  • 理解所有权如何影响函数签名和 API 设计选择
  • 掌握集合操作、重分配和迭代方式与借用规则的联系
  • 理解错误处理里何时返回拥有权、何时借用上下文
  • 建立并发基础直觉：move、线程边界与数据共享限制
  • 遇到所有权摩擦时，能在 clone、借用、重排代码和重构数据结构之间做实际取舍

📋 前置知识

建议你已经完成 Unit 1 前三章，能够读懂 move、借用冲突和生命周期签名。 本章默认你接受一个前提：所有权并不是语法惩罚，而是工程设计约束；因此讨论重点会落在接口选择和代码结构上。

1.4.1 API 设计：签名就是 ownership 决策

🧩

同样是“打印用户名”，为什么有的函数收 String，有的收 &str，有的还会收 impl AsRef<str>？

API 设计的第一个问题不是“类型能不能工作”，而是“我到底要不要接管数据”。 如果函数只是读取内容，优先借用；如果函数要保存、跨线程发送或重组所有权，按值接收可能更合适。 所有权模型逼你把这个选择提前写进签名，因此接口语义通常更清晰。

Rustapi_design.rs
fn greet(name: &str) {
    println!("hello, {name}");
}

fn store_name(name: String) -> String {
    name
}

fn main() {
    let user = String::from("Ferris");
    greet(&user);
    let saved = store_name(user);
    println!("saved = {saved}");
}
输出 
hello, Ferris
saved = Ferris

经验法则：只读优先借用；要修改就借可变引用；确实要长期持有或转移给别的所有者时，再按值接收。

1.4.2 集合与重分配：为什么拿了元素引用后常常不能继续 push？

📚

为什么你只是从 Vec 里借了一个元素，编译器却不让你紧接着 push 新元素？

因为集合操作可能触发底层缓冲区重分配。若你手里还拿着旧缓冲区上的引用，后续修改就可能让这些引用失效。 Rust 通过借用规则在编译期阻止这类“拿着旧地址继续写新数据”的风险。

Rustcollections.rs
fn main() {
    let mut numbers = vec![10, 20, 30];
    let first = &numbers[0];

    println!("first = {first}");
    numbers.push(40);
    println!("len = {}", numbers.len());
}
输出 
first = 10
len = 4

这里能通过，是因为 first 在 push 之前已经完成最后一次使用；如果把打印放到后面，借用冲突就会出现。

先检查是否能把“读取元素”和“修改集合”分成两个不重叠阶段。很多时候不需要 clone，只需要重排语句顺序或改成先记录索引、后重新取引用。

1.4.3 迭代方式会改变 ownership：iter、iter_mut、into_iter

🔁

为什么只是换一个迭代方法，容器后面还能不能继续用就变了？

因为三种迭代方式在语义上完全不同：iter() 借用只读元素，iter_mut() 借用可变元素， into_iter() 则会接管容器，把元素所有权逐个交出来。理解这一点，比死记方法名更重要。

Rustiteration.rs
fn main() {
    let languages = vec![String::from("Rust"), String::from("Go")];

    for lang in languages.iter() {
        println!("view: {lang}");
    }

    let consumed = languages.into_iter().map(|s| s.to_uppercase()).collect::>();
    println!("{:?}", consumed);
}
输出 
view: Rust
view: Go
["RUST", "GO"]

常见误区：看到 for x in vec 时，先问自己这是借用迭代还是消费迭代。若是消费迭代，循环后原容器通常已不可再用。

1.4.4 错误处理：返回值里的 ownership 要跟语义一致

🧯

设计 Result 时，错误信息应该借用输入文本，还是直接拥有一份消息？为什么很多错误类型更喜欢拥有数据？

错误处理常常需要跨函数边界传播。若错误值里塞满短生命周期借用，签名会迅速变复杂。 因此很多场景宁可让错误类型拥有自己的上下文字符串，也不把生命周期压力继续往上传。 同理，成功分支若需要把数据交给调用方长期使用，也常常直接返回拥有权。

Rusterror_handling.rs
fn parse_port(text: &str) -> Result {
    text.trim()
        .parse::()
        .map_err(|_| format!("invalid port: {text}"))
}

fn main() {
    match parse_port("8080") {
        Ok(port) => println!("port = {port}"),
        Err(err) => println!("error: {err}"),
    }
}
输出 
port = 8080

实战判断：如果返回值需要离开当前上下文长期存在，拥有型返回值通常比带 lifetime 的借用型错误更容易维护。

1.4.5 并发基础与重构取舍：什么时候该 move，什么时候该改结构？

🧵

为什么 thread::spawn 经常要求你写 move？又为什么遇到 ownership 摩擦时，不一定该第一时间加 clone()？

线程可能在当前作用域结束后继续运行，所以传入线程闭包的数据必须有明确所有权；这就是 move 常出现的原因。 但在单线程代码里，ownership 摩擦更常见的修复手段其实是：重排借用范围、拆分函数职责、改成返回拥有权、或把共享状态放进合适的同步原语，而不是无脑复制。

Rustconcurrency.rs
use std::thread;

fn main() {
    let task = String::from("index files");

    let handle = thread::spawn(move || {
        println!("worker: {task}");
    });

    handle.join().unwrap();
}
输出 
worker: index files
一般先看能否借用；借用冲突时看能否缩短借用范围；再看是否应该把结果改成拥有型返回值；最后才考虑 clone。到了跨线程共享场景，再考虑 Arc、Mutex 等同步原语，而不是绕过所有权模型。

📌 章节总结

  • Ownership 会直接塑造 API：按值、借用、可变借用各自表达不同职责。
  • 集合修改与元素借用的冲突，通常来自重分配和引用失效风险。
  • 迭代方式不只是语法差异，而是在选择借用遍历还是消费所有权。
  • 错误处理里，拥有型返回值常常能降低生命周期复杂度并提升边界清晰度。
  • 并发与重构都离不开所有权：线程边界需要明确 owner，日常修复则应优先重排数据流，再考虑 clone。
