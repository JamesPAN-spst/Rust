Unit 1 · 所有权系统

Ch 1.3 Lifetimes

Lifetimes 不是“让引用活得更久”的魔法，而是告诉编译器“这个引用至少会和谁一样久”。 本章要把生命周期从抽象符号，讲成一个关于借用关系、返回值约束和 API 边界的可读模型。

🎯 学习目标

  • 理解生命周期标注是在描述引用关系，而不是延长对象寿命
  • 能用悬垂引用案例解释编译器为什么要检查 lifetime
  • 掌握常见函数签名中的生命周期标注与返回值约束
  • 理解生命周期省略规则（elision）何时够用，何时必须手写
  • 识别结构体中持有引用时的常见误区与修复方向

📋 前置知识

建议你已经完成 Ch 1.1 和 Ch 1.2，尤其要先理解 owner、borrow、借用作用域和“多个读者或一个写者”的模型。 Lifetimes 解决的是“引用之间的持续关系”，不是重新讲一遍 move 或 mut。

1.3.1 生命周期的直觉：它描述的是关系，不是时长魔法

🧭

看到 'a 时，很多人第一反应是“这里把变量延长到了多久”。可编译器真的允许你靠标注强行延寿吗？

生命周期标注的真正作用是：描述引用之间至少要保持多久的有效关系。 它不创造新寿命，只是把已有借用关系说清楚。也就是说，'a 是约束，不是增益 buff。

正式理解： lifetime parameter 用来告诉编译器“这些引用的有效期如何关联”，而不是“请把它们活久一点”。

Rustlifetime_intuition.rs
fn identity<'a>(text: &'a str) -> &'a str {
    text
}

fn main() {
    let name = String::from("Ferris");
    let view = identity(&name);
    println!("{view}");
}
说明 
这里的 'a 表示：返回引用不会比传入引用活得更久，而不是让 name 自动延长生命周期。

1.3.2 悬垂引用：为什么编译器宁可拒绝，也不愿赌一把

⚠️

如果函数返回了一个指向局部变量的引用，而局部变量在函数结束时已经被销毁，这个引用还剩下什么？

这就是悬垂引用（dangling reference）：引用还在，但它指向的数据已经不存在。 Rust 的生命周期检查，本质上就是在编译期证明“返回的引用不会比被引用的数据活得更久”。证明不了，就拒绝编译。

Rustdangling.rs
fn dangling_ref() -> &str {
    let text = String::from("oops");
    &text
}

fn main() {
    // let r = dangling_ref();
}
说明 
这段代码不会通过编译，因为 text 在函数结束时被释放，返回引用会悬空。
常见修复方向有两个：要么返回拥有权（例如直接返回 String），要么让调用方提供数据并只返回基于输入借用出来的引用。

1.3.3 函数签名中的生命周期：返回谁，就约束谁

🔗

如果函数接收两个字符串切片并返回其中一个，编译器怎么知道返回值和哪个输入相关？

当返回引用可能来自多个输入时，编译器需要你把关系写出来。 最经典的例子是 longest：返回值的生命周期必须不长于两个输入中较短的那个，所以签名要把它们关联到同一个 lifetime parameter。

Rustlongest.rs
fn longest<'a>(left: &'a str, right: &'a str) -> &'a str {
    if left.len() >= right.len() {
        left
    } else {
        right
    }
}

fn main() {
    let a = String::from("rust");
    let b = String::from("ownership");
    println!("{}", longest(&a, &b));
}
输出 
ownership

判断口诀： 如果返回引用来自输入引用，就要问自己：返回值和哪个输入绑定？如果来自多个输入，关系通常需要写进签名。

1.3.4 Elision 规则：有些 lifetime 可以省，但不是总能省

✂️

为什么有些函数明明用了引用却不用写 'a，而另一些一省略就报错？

Rust 有一组生命周期省略规则（lifetime elision rules），会在常见简单场景里自动补全。 例如只有一个输入引用时，输出引用通常默认继承它的生命周期；方法里的 &self 也有特殊规则。 但一旦返回值可能和多个输入有关，省略就不够了。

Rustelision.rs
fn first_word(text: &str) -> &str {
    text.split_whitespace().next().unwrap_or("")
}

struct Label {
    name: String,
}

impl Label {
    fn as_str(&self) -> &str {
        &self.name
    }
}
说明 
上面两个例子都能省略 lifetime，因为返回值和唯一输入引用（或 &self）之间的关系足够明确。

1.3.5 struct 中的引用与常见错误：你是在借住，不是在拥有

🏷️

为什么一个持有 &str 字段的结构体必须写生命周期，而持有 String 的结构体却不用？

因为持有引用的结构体并不拥有数据，它只是“借住”在别人的数据上。 所以结构体类型本身必须声明：这个借住关系至少要和哪段生命周期一样久。 这也是很多初学者在把局部字符串切片塞进 struct 时最容易遇到的报错来源。

Ruststruct_ref.rs
struct Excerpt<'a> {
    part: &'a str,
}

fn main() {
    let article = String::from("Rust lifetimes feel abstract at first.");
    let first = article.split('.').next().unwrap();
    let excerpt = Excerpt { part: first };

    println!("{}", excerpt.part);
}
输出 
Rust lifetimes feel abstract at first

常见错误：在函数内部创建一个 String，再把它的切片塞进返回的 struct。问题不在 struct 语法，而在底层数据已经离开作用域。

当数据需要跨函数返回、跨层保存或避免复杂生命周期传播时，直接用 String、Vec 等拥有型字段通常更简单。引用型字段更适合短期视图、零拷贝解析或明确受控的借用场景。

📌 章节总结

  • Lifetime 标注描述的是引用之间的有效期关系，不会强行延长对象寿命。
  • 悬垂引用是生命周期检查想要阻止的核心风险之一。
  • 当函数返回引用且它可能来自多个输入时，生命周期关系通常必须手写。
  • Elision 规则能覆盖简单场景，但不是所有签名都能省略。
  • 结构体中持有引用时，要么清楚声明借用关系，要么考虑改成直接拥有数据以降低复杂度。

🧪 课后测试

  1. ⭐ 基础： 用自己的话解释“生命周期标注不会延长值的寿命，只会描述引用关系”这句话。
  2. ⭐⭐ 进阶： 写一个 pick_first 函数，接收两个 &str 并返回第一个；然后解释为什么签名需要或不需要显式 lifetime。
  3. ⭐⭐⭐ 挑战： 构造一个返回局部引用的错误示例，把它分别改写成“返回拥有权”和“借用调用方输入”两种合法方案。
  4. 🏆 附加： 设计一个日志解析结构体：比较“字段直接用 String 存储”和“字段用 &str 借用原始缓冲区”两种方案在性能、复杂度和可维护性上的取舍。
