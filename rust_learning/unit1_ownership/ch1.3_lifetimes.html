Chapter 1.3

生命周期 · Lifetimes

生命周期是 Rust 最令初学者困惑的概念——但它的本质很简单：告诉编译器"这个引用的数据至少活多久"。 生命周期标注不改变任何东西的实际存活时间，它只是你和编译器之间的一份合约。

01生命周期参数 'a 语法

🎯 为什么需要了解这个

上一章末尾我们看到：fn longest(a: &str, b: &str) -> &str 不能编译。编译器不知道返回的引用来自 a 还是 b，也就无法判断返回值的有效期。

生命周期参数 'a 就是你给编译器的线索："返回值的有效期与输入参数的有效期之间有这样的关系"。

🔍 核心思路

生命周期参数不是在说"活多久"，而是在说"谁活得比谁久"。

把它想象成数学中的约束：'a 不是一个具体的时间长度，而是一个变量。编译器会检查调用处是否满足这个约束。

核心公式：返回引用的生命周期 ≤ 所有标注了同一个 'a 的输入引用的生命周期的最小值。

📐 规则与语法

语法：

RUSTlifetime_syntax.rs
// ❌ 编译器不知道返回的引用寿命跟谁
// fn longest(a: &str, b: &str) -> &str { ... }

// ✅ 用 'a 告诉编译器：返回值的生命周期 ≤ min(a 的生命周期, b 的生命周期)
fn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() >= b.len() { a } else { b }
}

fn main() {
    let s1 = String::from("long string");
    let result;
    {
        let s2 = String::from("hi");
        result = longest(s1.as_str(), s2.as_str());
        println!("longest: {}", result); // ✅ s2 还活着
    }
    // println!("{}", result); // ❌ s2 已经 drop，result 可能指向它
}
输出
longest: long string

解读：'a 被实例化为 s1 和 s2 生命周期的交集（即较短那个 = s2 的生命周期）。所以 result 只在 s2 活着的范围内有效。

如果返回值只来自一个参数：

RUSTone_param_lifetime.rs
// 返回值只来自 a，b 的生命周期无关
fn first_only<'a>(a: &'a str, b: &str) -> &'a str {
    a   // 只返回 a，所以只需要 a 的生命周期约束
}

fn main() {
    let s1 = String::from("hello");
    let result;
    {
        let s2 = String::from("world");
        result = first_only(&s1, &s2);
    } // s2 drop，但 result 来自 s1，没问题
    println!("{}", result); // ✅
}
输出
hello
🌐 工程实践

何时需要手动标注：

  • 函数有多个引用参数且返回引用→ 大概率需要
  • 函数只有一个引用参数且返回引用 → 省略规则搞定
  • 方法有 &self 且返回引用 → 省略规则搞定
  • 不返回引用 → 不需要

经验：先不标注，编译器会告诉你哪里需要。然后思考"返回值来自哪个输入？"来决定标注。

✏️ 示例分析与易错点

常见错误：返回局部变量的引用

RUSTreturn_local_ref.rs
// ❌ 返回值不能来自函数内部创建的数据
fn bad_longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    let result = String::from("really long");
    // result.as_str()  // ❌ result 在函数结束时 drop
    a  // ✅ 只能返回来自输入参数的引用
}
编译错误（如果返回 result.as_str()）
error[E0515]: cannot return reference to local variable `result`
 --> src/main.rs:3:5
  |
3 |     result.as_str()
  |     ^^^^^^^^^^^^^^^ returns a reference to data owned by the current function

解决方案：如果需要返回新构造的数据，返回 String（所有权）而非 &str（引用）。

⚡ 快速检测
fn foo<'a>(x: &'a str, y: &'a str) -> &'a str 中 'a 的值是？

02生命周期省略规则

🎯 为什么需要了解这个

如果每个函数都要手动标注生命周期，代码会满是 'a 标注。Rust 的省略规则让编译器自动推断大部分情况的生命周期，你只在 "不明确" 时才需要手动标注。

🔍 核心思路

编译器按顺序尝试三条规则。如果应用完三条规则后仍有未确定的输出生命周期，才要求你手动标注。

  1. 规则 1：每个引用参数各自获得一个独立的生命周期参数
  2. 规则 2：如果只有一个输入生命周期参数，它被赋给所有输出引用
  3. 规则 3：如果参数中有 &self 或 &mut self，self 的生命周期被赋给所有输出引用
📐 规则与语法

规则推演示例：

RUSTelision_examples.rs
// ── 示例1：一个输入引用 → 规则 1+2 搞定 ──
fn first_word(s: &str) -> &str { ... }
// 推断过程：
//   规则1: fn first_word<'a>(s: &'a str) -> &str
//   规则2: fn first_word<'a>(s: &'a str) -> &'a str  ✅ 完成

// ── 示例2：方法有 &self → 规则 1+3 搞定 ──
impl MyStruct {
    fn name(&self) -> &str { ... }
}
// 推断过程：
//   规则1: fn name<'a>(&'a self) -> &str
//   规则3: fn name<'a>(&'a self) -> &'a str  ✅ 完成

// ── 示例3：两个输入引用 → 规则 1+2+3 都无法确定 → 需要手动标注 ──
fn longest(a: &str, b: &str) -> &str { ... }
// 推断过程：
//   规则1: fn longest<'a, 'b>(a: &'a str, b: &'b str) -> &str
//   规则2: 不适用（有两个输入生命周期）
//   规则3: 不适用（没有 &self）
//   → 输出生命周期未确定 → ❌ 编译错误，要求手动标注

实际代码验证：

RUSTelision_in_action.rs
// 不需要标注（省略规则搞定）
fn first_char(s: &str) -> Option {
    s.chars().next()
}

// 不需要标注（&self 规则）
struct Config {
    name: String,
}
impl Config {
    fn name(&self) -> &str {
        &self.name
    }
}

fn main() {
    let c = Config { name: String::from("myapp") };
    println!("{}: {}", c.name(), first_char("hello").unwrap());
}
输出
myapp: h
🌐 工程实践

经验法则：

  • 永远先不标注，让编译器告诉你
  • 绝大多数函数不需要生命周期标注（省略规则覆盖 ~95% 的场景）
  • 最常手写标注的场景：接受两个 &str 返回 &str
  • 如果你发现自己在写很复杂的生命周期，考虑是否应该用 owned 类型代替
✏️ 示例分析与易错点

省略规则失败的典型场景：

RUSTelision_fail.rs
// ❌ 省略规则无法推断
// fn pick(a: &str, b: &str, flag: bool) -> &str {
//     if flag { a } else { b }
// }

// ✅ 手动标注
fn pick<'a>(a: &'a str, b: &'a str, flag: bool) -> &'a str {
    if flag { a } else { b }
}

fn main() {
    let a = String::from("alpha");
    let b = String::from("beta");
    println!("{}", pick(&a, &b, true));
    println!("{}", pick(&a, &b, false));
}
输出
alpha
beta
⚡ 快速检测
省略规则第 3 条适用于哪种场景？

03'static 生命周期

🎯 为什么需要了解这个

'static 是你会频繁遇到的生命周期，但也是最容易误解的。错误信息经常建议你加 'static，但盲目添加会导致设计问题。理解它的真正含义是避免 "生命周期地狱" 的关键。

🔍 核心思路

'static 有两种用法，含义不同：

  1. 引用类型 &'static T：这个引用指向的数据在整个程序运行期间有效
  2. trait bound T: 'static：T 要么是 owned 类型，要么内部所有引用都是 'static

关键区别：T: 'static 不意味着 T 永远不会被释放。String 满足 'static bound（因为它不含有非 'static 引用），但 String 随时可以被 drop。

📐 规则与语法

&'static 引用：

RUSTstatic_ref.rs
// 字符串字面值是 &'static str
let s: &'static str = "I live forever (in binary)";

// 全局常量也是 'static
static GREETING: &str = "Hello, world!";

// lazy_static 或 const 也产生 'static 数据
const MAX_SIZE: usize = 1024;

fn main() {
    println!("{}", s);
    println!("{}", GREETING);
    println!("max: {}", MAX_SIZE);
}
输出
I live forever (in binary)
Hello, world!
max: 1024

T: 'static bound：

RUSTstatic_bound.rs
use std::fmt::Display;

// T: Display + 'static 不表示 T 永远存在
// 它表示 T 内部没有短命的引用
fn print_it(val: T) {
    println!("{}", val);
}

fn main() {
    let s = String::from("owned data");
    print_it(s);    // ✅ String 满足 'static bound（owned，无内部引用）
    print_it(42);   // ✅ i32 满足 'static bound

    let local = String::from("temp");
    // print_it(&local);  // ❌ &local 不是 'static 引用
    print_it(local);      // ✅ move local 进去，owned
}
输出
owned data
42
temp
🌐 工程实践

常见 'static 使用场景：

  • std::thread::spawn 要求闭包 F: 'static——因为线程可能比创建它的作用域活得更久
  • 错误类型 Box<dyn Error + 'static>
  • 全局配置、常量字符串

常见误区：编译器建议 "consider adding 'static" 时，不要盲目添加。先想想：真的需要数据活到程序结束吗？还是应该用 owned 类型？

✏️ 示例分析与易错点

线程与 'static：

RUSTthread_static.rs
use std::thread;

fn main() {
    let name = String::from("Alice");

    // ❌ 闭包捕获了 &name，但线程可能比 main 更久
    // thread::spawn(|| { println!("{}", name); });

    // ✅ move 闭包：把 name 的所有权转移给线程
    let handle = thread::spawn(move || {
        println!("Hello from thread, {}!", name);
    });

    // name 已被 move，这里不能用了
    handle.join().unwrap();
}
输出
Hello from thread, Alice!

不要把所有东西都改成 'static：

RUSTavoid_static.rs
// ❌ 过度使用 'static — 所有参数都只能接受 'static 数据
fn bad_api(data: &'static str) -> &'static str { data }

// ✅ 使用泛型生命周期 — 更灵活
fn good_api<'a>(data: &'a str) -> &'a str { data }

fn main() {
    let owned = String::from("dynamic");
    // bad_api(&owned);  // ❌ owned 不是 'static
    good_api(&owned);    // ✅ 'a 推断为 owned 的生命周期
}
⚡ 快速检测
String 满足 T: 'static 约束吗？

04结构体中的生命周期

🎯 为什么需要了解这个

当你想在 struct 中存储引用而非 owned 数据时（例如存一个 &str 而非 String），Rust 需要知道：这个 struct 实例不能活得比引用指向的数据更久。生命周期参数就是表达这个约束的工具。

🔍 核心思路

struct 有引用字段 → 必须有生命周期参数。这个参数约束：struct 实例的生命周期 ≤ 引用数据的生命周期。

直觉：struct 在借用别人的数据，必须保证在 "还书" 之前它就不存在了。

📐 规则与语法

基本语法：

RUSTstruct_lifetime.rs
// 'a 约束：Excerpt 不能活得比 part 引用的数据更久
struct Excerpt<'a> {
    part: &'a str,
}

fn main() {
    let novel = String::from(
        "Call me Ishmael. Some years ago..."
    );
    let first_sentence = novel
        .split('.')
        .next()
        .expect("Could not find a '.'");

    let excerpt = Excerpt {
        part: first_sentence,
    };
    println!("Excerpt: {}", excerpt.part);
}
// novel 在这里 drop — 但 excerpt 先 drop（声明晚），所以没问题
输出
Excerpt: Call me Ishmael

为 struct 实现方法：

RUSTstruct_methods.rs
struct Excerpt<'a> {
    part: &'a str,
}

impl<'a> Excerpt<'a> {
    // 返回 &str — 省略规则第 3 条：&self 的生命周期赋给返回值
    fn content(&self) -> &str {
        self.part
    }

    // 返回 i32 — 不需要生命周期
    fn word_count(&self) -> usize {
        self.part.split_whitespace().count()
    }
}

fn main() {
    let text = String::from("Hello world foo bar");
    let exc = Excerpt { part: &text };
    println!("{} ({} words)", exc.content(), exc.word_count());
}
输出
Hello world foo bar (4 words)
🌐 工程实践

何时在 struct 中用引用？何时用 owned？

  • 用 owned (String)：struct 需要独立管理数据、需要跨线程传递、数据来源不确定
  • 用引用 (&str)：零拷贝解析（如 JSON parser 借用原始字符串）、性能关键路径
  • 用 Cow<'a, str>：有时借用、有时拥有——最灵活但最复杂

经验法则：初学者优先用 owned 类型（String, Vec<T>），等熟练后再为性能优化引入引用。

✏️ 示例分析与易错点

struct 活得比数据久——编译错误：

RUSTstruct_outlive.rs
struct Excerpt<'a> {
    part: &'a str,
}

fn main() {
    let excerpt;
    {
        let text = String::from("hello world");
        excerpt = Excerpt { part: &text };
        // text 在这里 drop
    }
    // println!("{}", excerpt.part);  // ❌ text 已经释放
}
编译错误
error[E0597]: `text` does not live long enough
 --> src/main.rs:8:36
  |
8 |         excerpt = Excerpt { part: &text };
  |                                    ^^^^ borrowed value does not live long enough
9 |     }
  |     - `text` dropped here while still borrowed
10|     println!("{}", excerpt.part);
  |                    ------------ borrow later used here

多个生命周期参数：

RUSTmulti_lifetime_struct.rs
// 两个引用字段可能有不同的来源/生命周期
struct Pair<'a, 'b> {
    first: &'a str,
    second: &'b str,
}

fn main() {
    let s1 = String::from("hello");
    let pair;
    {
        let s2 = String::from("world");
        pair = Pair { first: &s1, second: &s2 };
        println!("{} {}", pair.first, pair.second); // ✅ 都还活着
    }
    // pair 不能在这里用了（second 已失效）
}
输出
hello world
