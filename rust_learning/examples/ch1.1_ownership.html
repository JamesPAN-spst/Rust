Ch 1.1 Ownership

Ownership 是 Rust 最核心、也最容易让初学者卡住的机制。这个模板页展示如何把 `math_univers` 的卡片化、提问式、交互式设计迁移到 Rust 学习内容里。

🎯 学习目标

  • 理解“每个值有唯一所有者”的基本约束
  • 理解赋值、传参为何会触发 move
  • 区分 move、copy、clone 的语义差别
  • 理解 Drop 与资源释放之间的关系

📋 前置知识

建议先了解变量绑定、函数调用和 struct 的基本语法；否则会把语法问题误认为所有权问题。

1.1.1 为什么 Rust 要有 Ownership？

🤔

如果两个变量都以为自己“拥有”同一块堆内存，程序结束时谁来释放？如果都释放一次，会发生什么？

Rust 用 ownership 在编译期决定资源归属，从而避免双重释放、悬垂指针和一大类内存错误。

核心规则很短：每个值在任意时刻只有一个所有者；所有者离开作用域时值会被清理。

设计重点：Rust 不是把“谁负责释放资源”交给程序员记忆，而是把这件事变成语言规则。

1.1.2 move 语义为什么会出现？

🧠

把一个 `String` 赋值给另一个变量后，如果旧变量还能继续使用，编译器该如何判断谁负责释放堆内存？

对拥有堆资源的类型，赋值默认不是“复制资源”，而是“转移拥有权”。这就是 move。

Rustmove.rs
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;

    println!("{s2}");
    // println!("{s1}"); // 编译错误：value borrowed after move
}
输出 
hello
深拷贝会带来隐式分配和性能成本。Rust 默认选择显式所有权转移，把昂贵复制变成需要开发者主动表达的行为。

1.1.3 Copy、Clone、Move 怎么区分？

Move 是转移所有权，Copy 是按位复制且原值仍可用，Clone 是显式复制资源。

Rustcopy_clone.rs
fn main() {
    let x = 42;
    let y = x;
    println!("x = {x}, y = {y}");

    let a = String::from("rust");
    let b = a.clone();
    println!("a = {a}, b = {b}");
}
输出 
x = 42, y = 42
a = rust, b = rust

常见误区： `clone()` 不是“修复报错按钮”。在写下 `clone()` 前，先问自己是否真的需要复制所有权。

1.1.4 Drop 为什么重要？

Drop 决定值离开作用域时如何清理资源。文件句柄、网络连接、锁、堆内存都依赖这个机制安全释放。

Rustdrop.rs
struct LogOnDrop(&'static str);

impl Drop for LogOnDrop {
    fn drop(&mut self) {
        println!("dropping {}", self.0);
    }
}

fn main() {
    let _guard = LogOnDrop("resource");
    println!("working...");
}
输出 
working...
dropping resource
RAII 的关键思想是“资源和对象生命周期绑定”。Rust 用作用域 + Drop 让这个模式变成默认行为。

📌 章节总结

  • Ownership 用语言规则明确资源归属
  • 对堆资源类型，赋值和传参经常意味着 move
  • `Copy` 是轻量复制，`Clone` 是显式复制资源
  • `Drop` 负责把资源释放时机绑定到作用域结束
