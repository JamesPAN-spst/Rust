Unit 4 · 并发与异步

异步编程

异步是 Rust 处理大量 I/O 密集任务的核心方案。与线程的"一任务一线程"不同，异步让成千上万的任务在少数几个线程上交替执行。本章建立 async/await 的完整心智模型：从 Future 的惰性求值、状态机本质，到 Executor 的调度机制和"不能阻塞"的铁律。

01async/await 心智模型

🎯 为什么需要了解这个

一个 Web 服务器同时处理 10000 个连接，每个连接大部分时间在等待网络数据。如果每个连接一个线程，需要 10000 个 OS 线程（栈空间 ~20GB）。async/await 让一个线程交替处理多个连接——等待时切换到其他任务，数据来了再切回来。

🔍 核心思路

async/await 的核心是三个概念：

  • 惰性求值：async fn 调用不执行代码，只返回一个 Future 值
  • .await 暂停点：每个 .await 是一个可能暂停的点——如果内部 Future 未就绪，控制权交还调度器
  • 状态机：编译器在每个 .await 处将函数切割成状态，自动生成一个可暂停/恢复的状态机
RUSTmental_model.rs
// 你写的代码：
async fn fetch_and_process() -> String {
    let data = fetch_data().await;   // 暂停点 1
    let result = transform(data).await; // 暂停点 2
    result
}

// 编译器看到的（概念简化）：
// enum FetchAndProcess {
//     State0 { },                           // 初始：调用 fetch_data
//     State1 { fetch_future: FetchFuture },  // 等待 fetch_data 完成
//     State2 { transform_future: TransFut }, // 等待 transform 完成
//     Done,
// }
📐 规则与语法
RUSTasync_basic.rs
use tokio::time::{sleep, Duration};

async fn greet(name: &str) -> String {
    // sleep 返回 Future，.await 暂停直到定时器到期
    sleep(Duration::from_millis(100)).await;
    format!("Hello, {}!", name)
}

#[tokio::main]
async fn main() {
    // 调用 async fn 只构造 Future，不执行
    let future = greet("Rust");
    println!("Future 已创建");

    // .await 驱动执行
    let result = future.await;
    println!("{}", result);
}
输出 
Future 已创建
Hello, Rust!

关键规则：

  • .await 只能在 async 函数/块中使用
  • async fn 不能直接在 fn main() 中调用——需要运行时驱动
  • async 块 async { ... } 也返回 Future，可赋值给变量
🌐 工程实践
  • 并发执行多个 Future：用 tokio::join! 或 futures::join! 同时驱动多个 Future
  • 顺序 vs 并发：连续 .await 是顺序的；用 tokio::spawn 或 join! 才能并发
  • async 块的妙用：可以在非 async 函数中用 async { ... } 构造 Future 再传给 spawn
RUSTconcurrent_await.rs
use tokio::time::{sleep, Duration};

async fn task(id: u32, ms: u64) -> u32 {
    sleep(Duration::from_millis(ms)).await;
    println!("任务 {} 完成", id);
    id
}

#[tokio::main]
async fn main() {
    // ❌ 顺序执行——总时间 300ms
    // let a = task(1, 100).await;
    // let b = task(2, 200).await;

    // ✅ 并发执行——总时间 ~200ms（取最长的）
    let (a, b) = tokio::join!(task(1, 100), task(2, 200));
    println!("结果: {}, {}", a, b);
}
输出 
任务 1 完成
任务 2 完成
结果: 1, 2
✏️ 示例分析与易错点
RUSTlazy_future.rs
async fn side_effect() {
    println!("我被执行了！");
}

#[tokio::main]
async fn main() {
    let _f = side_effect(); // 只创建 Future，不执行！
    println!("main 继续");
    // _f 永远没被 await，"我被执行了！"永远不会打印
}
输出 
main 继续

易错点：Rust 的 Future 是惰性的（lazy）。如果你调用 async fn 但不 .await，编译器会警告 unused future，但不会阻止你——副作用直接丢失。与 JavaScript Promise（构造即执行）完全不同！

02Future 与 poll

🎯 为什么需要了解这个

.await 是语法糖，底层驱动 Future 的机制是 poll。理解 poll 模型才能真正搞清楚异步代码的执行流——为什么 .await 能暂停、怎么恢复、Waker 是什么。这是排查异步 bug 和理解性能的基础。

🔍 核心思路

Future trait 的核心是一个 poll 方法，每次调用时返回以下两种状态之一：

  • Poll::Ready(val)：完成了，这是结果
  • Poll::Pending：还没完成，等我通知你

关键组件：

  • Waker：Pending 时 Future 保存 Waker；就绪时调用 waker.wake() 通知 Executor 重新 poll
  • Pin：poll 的 self 是 Pin<&mut Self>，确保 Future 不被移动（因为编译器生成的状态机可能是自引用的）
📐 规则与语法
RUSTfuture_trait.rs
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

// Future trait 定义（简化）
// trait Future {
//     type Output;
//     fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
// }

// 一个永远立刻完成的 Future
struct Ready<T>(Option<T>);

impl<T> Future for Ready<T> {
    type Output = T;

    fn poll(mut self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll<T> {
        // take() 取出内部值，返回 Ready
        match self.0.take() {
            Some(val) => Poll::Ready(val),
            None => panic!("polled after completion"),
        }
    }
}

// 使用：
// let fut = Ready(Some(42));
// 被 .await 后立刻返回 42
RUSTpoll_flow.rs
// .await 的展开逻辑（伪代码）
// let result = some_future.await;
// 等价于：
// loop {
//     match some_future.poll(cx) {
//         Poll::Ready(val) => break val,
//         Poll::Pending => yield,  // 将控制权交回 Executor
//     }
// }
🌐 工程实践
  • 你几乎不需要手写 poll：编译器在 async fn 中自动生成。手写 Future 只在编写底层库（如定时器、I/O 驱动）时需要
  • 理解 poll 有助于排查问题：为什么 Future 没有推进？因为没人 wake 它。为什么 .await 没返回？因为内部 Future 仍是 Pending
  • Pin 的意义：async fn 生成的状态机可能有自引用字段。Pin 保证值不被移动，自引用指针不会失效
  • Future 只被 poll 一次 Ready：poll 返回 Ready 后不应再被 poll（行为未定义）
✏️ 示例分析与易错点
RUSTmanual_future.rs
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

/// 一个延迟 N 次 poll 后返回值的 Future
struct Delay {
    remaining: u32,
}

impl Future for Delay {
    type Output = String;

    fn poll(mut self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<String> {
        if self.remaining == 0 {
            Poll::Ready("done!".to_string())
        } else {
            self.remaining -= 1;
            cx.waker().wake_by_ref(); // 立刻再次唤醒自己
            Poll::Pending
        }
    }
}

#[tokio::main]
async fn main() {
    let result = Delay { remaining: 3 }.await;
    println!("{}", result); // "done!"
}
输出 
done!

易错点：忘记调用 waker.wake()！如果 Future 返回 Pending 但没有安排 wake，Executor 永远不会再 poll 它——任务就"饿死"了。

03Executor 运行时

🎯 为什么需要了解这个

Future 本身不做任何事——它需要被驱动。Executor（执行器）就是那个反复 poll Future、调度任务的组件。Rust 有意不在标准库中包含 Executor，留给社区提供（如 Tokio、async-std）。理解 Executor 的职责，才能选对运行时、调对配置。

🔍 核心思路

一个完整的异步运行时通常包含三个组件：

  • Executor：管理任务队列，把 Future 从任务队列取出并 poll
  • I/O 驱动：与操作系统的 I/O 多路复用（epoll/kqueue/IOCP）交互，在 I/O 就绪时唤醒 Future
  • 定时器：管理 sleep/timeout，在时间到达时唤醒 Future

执行流程：Executor 从就绪队列取出任务 → poll → 如果 Pending 则挂起等待 Waker → I/O 驱动或定时器触发 wake → 任务重新入队 → 循环。

📐 规则与语法
RUSTruntime_setup.rs
// 方式 1：宏自动创建运行时
#[tokio::main]
async fn main() {
    println!("在 Tokio 运行时中");
}

// 宏展开等价于：
// fn main() {
//     tokio::runtime::Runtime::new()
//         .unwrap()
//         .block_on(async {
//             println!("在 Tokio 运行时中");
//         });
// }

// 方式 2：手动创建运行时（更多控制）
fn manual_runtime() {
    let rt = tokio::runtime::Builder::new_multi_thread()
        .worker_threads(4)     // 4 个工作线程
        .enable_all()          // 启用 I/O 和定时器
        .build()
        .unwrap();

    rt.block_on(async {
        println!("手动创建的运行时");
    });
}
RUSTtokio_spawn.rs
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    // tokio::spawn 创建一个新的异步任务（类似 thread::spawn）
    let handle = tokio::spawn(async {
        sleep(Duration::from_millis(100)).await;
        "任务完成"
    });

    // .await JoinHandle 获取任务结果
    let result = handle.await.unwrap();
    println!("{}", result);
}
输出 
任务完成
🌐 工程实践
  • 单线程 vs 多线程运行时：#[tokio::main(flavor = "current_thread")] 适合简单场景或嵌入式；默认多线程适合服务端
  • tokio::spawn vs join!：spawn 创建独立任务（可在不同线程执行），join! 在当前任务内并发
  • 一个进程一个运行时：一般只创建一个 Tokio Runtime，避免嵌套 runtime（会 panic）
  • 为什么 Rust 不内置 runtime：不同场景需要不同运行时——高吞吐服务器用 Tokio，嵌入式用 embassy，WASM 用浏览器事件循环
✏️ 示例分析与易错点
RUSTnested_runtime.rs
// ❌ 错误：在异步上下文中再次 block_on 会 panic
#[tokio::main]
async fn main() {
    let rt = tokio::runtime::Runtime::new().unwrap();
    // rt.block_on(async { ... });
    // 运行时 panic: Cannot start a runtime from within a runtime
}

// ✅ 正确做法：用 tokio::spawn 或 .await
#[tokio::main]
async fn main() {
    let result = tokio::spawn(async {
        // 在已有运行时中创建新任务
        42
    }).await.unwrap();
    println!("{}", result);
}
输出 
42

易错点：不能在异步上下文中调用 block_on，这会嵌套运行时导致 panic。在异步代码中需要驱动另一个 Future 时，用 .await 或 tokio::spawn。

04为什么不能阻塞

🎯 为什么需要了解这个

异步运行时的每个工作线程可能同时负责成百上千个 Future。如果一个 Future 调用了 std::thread::sleep(10s)，这个线程整整 10 秒无法服务其他任何 Future——所有在这个线程上的任务都被冻结了。这是 async Rust 中最常犯的错误之一。

🔍 核心思路

黄金法则：永远不要在异步上下文中做长时间阻塞操作。

  • 阻塞操作 = 长时间占据线程不让出：同步 I/O、CPU 密集计算、std::sync::Mutex::lock、std::thread::sleep
  • 替代方案：使用异步版本（tokio::fs、tokio::time::sleep）或 spawn_blocking
  • spawn_blocking：将阻塞操作丢到专用的阻塞线程池，不影响异步工作线程
📐 规则与语法
RUSTblocking_bad.rs
use tokio::time::{sleep, Duration};

async fn bad_task() {
    // ❌ 同步 sleep 阻塞整个线程
    std::thread::sleep(Duration::from_secs(5));
    println!("bad task done");
}

async fn good_task() {
    // ✅ 异步 sleep，让出线程
    sleep(Duration::from_secs(1)).await;
    println!("good task done");
}

#[tokio::main]
async fn main() {
    // bad_task 会冻结线程 5 秒
    // good_task 得不到执行机会直到 bad_task 的 sleep 结束
    tokio::join!(bad_task(), good_task());
}
可能的输出（顺序异常） 
bad task done
good task done
RUSTspawn_blocking.rs
use tokio::task;

#[tokio::main]
async fn main() {
    // 将阻塞操作放到专用线程池
    let result = task::spawn_blocking(|| {
        // 这里可以安全地做同步 I/O 或 CPU 密集计算
        let mut sum: u64 = 0;
        for i in 0..1_000_000 {
            sum += i;
        }
        sum
    }).await.unwrap();

    println!("计算结果: {}", result);
}
输出 
计算结果: 499999500000
🌐 工程实践
阻塞操作 异步替代方案
std::thread::sleeptokio::time::sleep
std::fs::readtokio::fs::read
std::net::TcpStreamtokio::net::TcpStream
std::sync::Mutextokio::sync::Mutex（跨 .await 时）
CPU 密集计算tokio::task::spawn_blocking
  • std::sync::Mutex 的特殊情况：如果锁定时间极短且不跨 .await，用 std::sync::Mutex 性能更好（无 async 开销）
  • 检测工具：Tokio Console 和 tokio::time::timeout 可帮助发现意外阻塞
✏️ 示例分析与易错点
RUSTmutex_across_await.rs
use std::sync::Mutex;
use std::sync::Arc;

// ❌ 错误：MutexGuard 跨 .await 持有
async fn bad_lock(data: Arc<Mutex<Vec<i32>>>) {
    let mut guard = data.lock().unwrap();
    guard.push(1);
    tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
    // guard 在 .await 之后才 drop
    // 如果用 tokio::spawn 会编译错误：MutexGuard 不是 Send
}

// ✅ 正确：在 .await 前释放锁
async fn good_lock(data: Arc<Mutex<Vec<i32>>>) {
    {
        let mut guard = data.lock().unwrap();
        guard.push(1);
    } // guard 在这里 drop
    tokio::time::sleep(tokio::time::Duration::from_secs(1)).await;
}

易错点：std::sync::MutexGuard 不是 Send。如果持有它跨过 .await 点，tokio::spawn 会编译失败。解决方案：在 .await 前 drop guard，或使用 tokio::sync::Mutex。
