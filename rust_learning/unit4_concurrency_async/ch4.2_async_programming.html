Unit 4 · 并发与异步

Ch 4.2 异步编程

异步最容易被误解成“更快的线程”。这一章的目标不是背 API，而是建立 async/await 的工作直觉：Future 是什么、为什么需要 executor、以及为什么在 async 代码里阻塞线程会伤害整个系统。

🎯 学习目标

  • 理解 async/await 表达的是“可暂停的任务”，而不是自动创建线程
  • 建立对 Future、poll、wake 的基础直觉
  • 理解 executor 如何调度大量异步任务
  • 知道为什么 async 任务里应避免阻塞调用，并能做出基本选型

📋 前置知识

建议先完成线程并发章节，并熟悉函数、闭包和 Result。若还把“等待”理解成“线程空转卡住”，会很难真正理解 Future 为什么能在少量线程上承载大量 I/O 任务。

4.2.1 async / await：把“等待中的任务”写得像顺序代码

🤔

网络请求的大部分时间都在等对端回应，为什么还要一直占着一个线程什么也不做？

async 函数会返回一个 Future。它描述的是“这项工作最终会产出一个结果”，中间遇到等待点时可以先把执行权交出去，等条件满足后再继续。因此 async/await 的核心收益是复用等待时间，而不是凭空制造更多 CPU。

正式规则：async fn 调用后不会立刻执行到底，而是生成一个 Future；只有被运行时驱动时，这个 Future 才会逐步推进。

Rustasync_shape.rs
async fn fetch_label() -> String {
    String::from("ready")
}

async fn show() {
    let label = fetch_label().await;
    println!("{label}");
}
说明 
调用 fetch_label() 得到的是 Future；
await 代表“等它准备好后再继续”。

4.2.2 Future 为什么要被 poll？

🔄

如果 Future 还没准备好结果，它是怎么告诉系统“我现在先停一下，以后再叫我”的？

Rust 的 Future 是惰性的：它自己不会偷偷在后台跑。executor 会调用 poll 推进它一步；如果还没完成，Future 返回 Poll::Pending，并约定在条件满足时通过 waker 再次通知 executor。

心智模型：把 Future 想成一个“可恢复的状态机”。每次 poll 都是在问：“你现在能继续做一点吗？”

Rustfuture_poll_intuition.rs
use std::future::Future;
use std::pin::Pin;
use std::task::{Context, Poll};

struct OneShot(bool);

impl Future for OneShot {
    type Output = &'static str;

    fn poll(mut self: Pin<&mut Self>, _cx: &mut Context<'_>) -> Poll {
        if self.0 {
            Poll::Ready("done")
        } else {
            self.0 = true;
            Poll::Pending
        }
    }
}
说明 
第一次 poll 返回 Pending；
下一次再 poll，状态变成 Ready。
因为这样调度权掌握在 executor 手里。Future 只描述任务逻辑，不强行决定在哪里、何时、以什么策略运行。

4.2.3 executor：谁在管理这么多 Future？

🧩

如果 async 任务自己不会跑，那到底是谁负责不停 poll、登记 wake、安排先后顺序？

这就是 executor 的职责。它维护一批待运行的任务，并在它们可继续执行时进行 poll。你可以把 executor 看成“异步任务调度器”，而运行时 runtime 则通常还会附带定时器、I/O 驱动、任务生成等更完整能力。

区分概念：Future 描述任务，executor 调度任务，runtime 往往 = executor + I/O / timer / spawning 等基础设施。

Rustjoin_style.rs
async fn load_user() -> &'static str { "user" }
async fn load_orders() -> &'static str { "orders" }

async fn page() {
    let (user, orders) = tokio::join!(load_user(), load_orders());
    println!("{user} + {orders}");
}
说明 
两个任务不是各自占一个 OS 线程；
而是由运行时交替推进，谁先可继续就先 poll 谁。

4.2.4 为什么 async 代码里仍然要避免阻塞？

🚧

如果一个 async 任务里直接调用 std::thread::sleep 或做很重的 CPU 计算，会发生什么？

异步运行时通常希望少量线程能承载大量任务。如果某个任务在运行时线程上做阻塞调用，就会把这条线程卡住，连带影响同线程上的其他任务。I/O 等待应使用异步 API；重 CPU 任务则应考虑专门线程池或 spawn_blocking 一类机制。

判断原则：async 适合“等待很多、计算不重”的任务；若是纯 CPU 重活，线程或专门工作池往往更直接。

Rustavoid_blocking.rs
async fn good_style() {
    // 用异步 I/O 等待网络或文件事件
    // tokio::time::sleep(...).await;
}

async fn bad_style() {
    // 会阻塞运行时线程，拖慢同线程上的其他任务
    std::thread::sleep(std::time::Duration::from_secs(1));
}
说明 
await 会把执行权交还给运行时；
阻塞调用则会把运行时线程整段卡住。
不是。短小计算完全没问题；真正要避免的是长时间占住运行时线程的阻塞 I/O 或重 CPU 工作。关键是别让大量其他任务陪它一起等待。

📌 章节总结

  • async fn 返回的是 Future；await 表示“等结果就绪后继续”。
  • Future 是惰性状态机，需要 executor 通过 poll 推进。
  • executor 负责调度任务，runtime 往往还整合了 I/O 和时间驱动。
  • async 的优势来自非阻塞等待；重 CPU 和阻塞 I/O 需要另外安排执行位置。
