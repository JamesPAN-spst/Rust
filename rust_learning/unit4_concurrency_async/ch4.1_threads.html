Unit 4 · 并发与异步

线程并发

Rust 的并发模型被称为"无畏并发"——编译器通过所有权和类型系统在编译期阻止数据竞争。本章从 OS 线程出发，依次掌握 spawn、move 闭包、消息传递、共享状态和 Send/Sync 两大 marker trait，构建线程安全编程的完整心智模型。

01std::thread 与 spawn

🎯 为什么需要了解这个

现代 CPU 都是多核的。如果程序只跑在一个核心上，其余核心都在空转。线程是操作系统提供的最基本的并行执行单元——掌握它，才能让 Rust 程序真正利用多核能力。

🔍 核心思路

std::thread::spawn 创建一个新的 OS 线程并执行传入的闭包。它返回一个 JoinHandle<T>，调用 join() 可以等待线程结束并获取返回值。

  • 1:1 模型：每个 Rust 线程 = 一个 OS 线程，调度由操作系统管理
  • JoinHandle：join() 阻塞当前线程直到子线程完成；忽略它线程变成 detached
  • 线程 vs 异步：线程适合 CPU 密集型任务；异步适合大量 I/O 等待
📐 规则与语法
RUSTthread_basic.rs
use std::thread;
use std::time::Duration;

fn main() {
    // spawn 返回 JoinHandle<T>
    let handle = thread::spawn(|| {
        for i in 1..=3 {
            println!("子线程: {}", i);
            thread::sleep(Duration::from_millis(100));
        }
        42 // 线程的返回值
    });

    println!("主线程继续执行...");

    // join() 等待子线程结束，unwrap 取出返回值
    let result = handle.join().unwrap();
    println!("子线程返回: {}", result);
}
输出 
主线程继续执行...
子线程: 1
子线程: 2
子线程: 3
子线程返回: 42

关键签名：pub fn spawn<F, T>(f: F) -> JoinHandle<T> where F: FnOnce() -> T + Send + 'static, T: Send + 'static

🌐 工程实践
  • 始终 join：忽略 JoinHandle 会导致 main 退出时子线程被强制终止
  • 错误处理：join().unwrap() 在子线程 panic 时也会 panic；生产代码应匹配 Err
  • 线程数量：OS 线程有栈开销（默认 ~2–8MB），不要创建数千个；大量并行任务用线程池或异步
  • 命名线程：thread::Builder::new().name("worker".into()).spawn(...) 方便调试
✏️ 示例分析与易错点
RUSTthread_panic.rs
use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        panic!("子线程炸了！");
    });

    // join 返回 Err，不会让主线程 panic
    match handle.join() {
        Ok(_) => println!("正常结束"),
        Err(_) => println!("子线程发生了 panic"),
    }

    println!("主线程仍在运行");
}
输出 
子线程发生了 panic
主线程仍在运行

易错点：子线程的 panic 不会自动传播到主线程。如果不 join 不 match，子线程的错误会被静默吞掉。

02move 语义与线程

🎯 为什么需要了解这个

线程需要数据才能工作，但子线程可能比创建它的作用域活得更久。如果只是借用栈上数据，数据被释放后线程就拿着悬垂引用了。move 闭包是 Rust 解决这个问题的方式——编译期强制你交出所有权。

🔍 核心思路

move || { ... } 告诉编译器：闭包内使用的所有外部变量都要转移所有权到闭包中。这满足了 'static 约束——闭包不再借用任何临时数据。

  • 非 move：闭包默认借用（& 或 &mut）外部变量
  • move：强制将所有捕获的变量 move 进闭包
  • Copy 类型的特殊情况：move 对 Copy 类型是复制，对非 Copy 类型是转移
📐 规则与语法
RUSTmove_closure.rs
use std::thread;

fn main() {
    let name = String::from("Rust");
    let count = 3; // i32 是 Copy

    // move 将 name 的所有权和 count 的副本移入闭包
    let handle = thread::spawn(move || {
        println!("Hello from thread: {} (count={})", name, count);
    });

    // println!("{}", name); // ❌ 编译错误！name 已被 move
    println!("count = {}", count); // ✅ i32 是 Copy，原变量仍可用

    handle.join().unwrap();
}
输出 
count = 3
Hello from thread: Rust (count=3)
🌐 工程实践
  • 需要在多个线程中使用同一数据？先 clone 再 move 各自的副本进去
  • 大数据不想 clone？用 Arc 共享（详见 KP4）
  • 闭包内只用引用的一部分？可以在闭包外先 let 绑定子字段，再 move 进去，避免 move 整个结构体
RUSTclone_then_move.rs
use std::thread;

fn main() {
    let data = vec![1, 2, 3];

    // 两个线程都需要 data——各 clone 一份
    let data_clone = data.clone();
    let h1 = thread::spawn(move || {
        println!("线程1: {:?}", data_clone);
    });
    let h2 = thread::spawn(move || {
        println!("线程2: {:?}", data);
    });

    h1.join().unwrap();
    h2.join().unwrap();
}
✏️ 示例分析与易错点
RUSTmove_loop.rs
use std::thread;

fn main() {
    let mut handles = vec![];

    for i in 0..3 {
        // i 是 i32 (Copy)，每次循环 move 的是 i 的副本
        let handle = thread::spawn(move || {
            println!("线程 {}", i);
            i * 10
        });
        handles.push(handle);
    }

    for h in handles {
        println!("结果: {}", h.join().unwrap());
    }
}
输出 
线程 0
线程 1
线程 2
结果: 0
结果: 10
结果: 20

易错点：循环变量 i 是 Copy 类型，每次循环 move 的是副本。如果要传 String 类型到多个线程，必须在循环内 clone。

⚡ 快速检测
move 闭包捕获一个 i32 变量后，原变量还能用吗？

03channel 消息传递

🎯 为什么需要了解这个

Go 语言有句名言："不要通过共享内存来通信，而要通过通信来共享内存"。Rust 的 mpsc::channel 就是这种思路——数据通过管道在线程间传递，所有权跟着数据走，从类型系统层面杜绝数据竞争。

🔍 核心思路
  • mpsc = Multiple Producer, Single Consumer
  • channel() 返回 (Sender<T>, Receiver<T>)
  • tx.send(val) 转移 val 的所有权到 channel 中
  • rx.recv() 从 channel 取出数据（阻塞），获得所有权
  • Sender 可以 clone（多生产者），Receiver 不能 clone（单消费者）
📐 规则与语法
RUSTchannel_basic.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        let msg = String::from("hello from thread");
        tx.send(msg).unwrap();
        // println!("{}", msg); // ❌ msg 所有权已 send 出去
    });

    // recv() 阻塞直到收到消息
    let received = rx.recv().unwrap();
    println!("收到: {}", received);
}
输出 
收到: hello from thread
RUSTchannel_multi_producer.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    for id in 0..3 {
        let tx_clone = tx.clone();
        thread::spawn(move || {
            tx_clone.send(format!("来自线程 {}", id)).unwrap();
        });
    }
    drop(tx); // 原始 tx 不再需要，drop 掉避免 channel 不关闭

    // 用 for-in 迭代 rx——所有 sender drop 后自动结束
    for msg in rx {
        println!("{}", msg);
    }
}
输出（顺序可能不同） 
来自线程 0
来自线程 1
来自线程 2
🌐 工程实践
  • 记得 drop 原始 tx：clone 了 Sender 后，原始的如果不 drop，channel 永远不会关闭，rx 迭代会永远阻塞
  • 有界 channel：mpsc::sync_channel(cap) 限制缓冲区大小，发送端在满时阻塞——天然背压机制
  • 适用场景：日志收集、任务分发、pipeline 数据处理
  • 替代方案：crossbeam-channel 提供 MPMC 和 select!，更适合复杂场景
✏️ 示例分析与易错点
RUSTchannel_iter.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        for val in vec!["hello", "world", "!"] {
            tx.send(val.to_string()).unwrap();
        }
        // tx 在这里被 drop —— channel 关闭
    });

    // ⚠️ 安全写法：for 循环在 channel 关闭时自动结束
    for msg in rx {
        println!("{}", msg);
    }
    // ❌ 不要用 loop { rx.recv().unwrap() }——channel 关闭时会 panic
}
输出 
hello
world
!

易错点：rx.recv().unwrap() 在 channel 关闭时会 panic。生产代码应该用 for msg in rx 或 while let Ok(msg) = rx.recv()。

⚡ 快速检测
tx.send(val) 之后，val 还能在发送端线程中使用吗？

04Arc 与 Mutex 共享状态

🎯 为什么需要了解这个

有些场景下数据不适合通过 channel "传来传去"——比如共享配置、计数器、缓存。多线程需要读/写同一份数据。Rust 用 Arc（原子引用计数）+ Mutex（互斥锁）的组合来安全地共享可变状态。

🔍 核心思路
  • Arc<T>：原子引用计数，允许多个线程持有同一数据的共享所有权
  • Mutex<T>：互斥锁，保证同一时刻只有一个线程能访问内部数据
  • 组合使用：Arc<Mutex<T>> = 跨线程共享（Arc） + 安全可变（Mutex）
  • lock() 返回 MutexGuard——RAII 机制，出作用域自动释放锁
📐 规则与语法
RUSTarc_mutex_counter.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter); // 克隆 Arc，引用计数 +1
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap(); // 获取锁
            *num += 1;
            // MutexGuard 在此 drop，锁自动释放
        });
        handles.push(handle);
    }

    for h in handles {
        h.join().unwrap();
    }

    println!("最终计数: {}", *counter.lock().unwrap());
}
输出 
最终计数: 10
🌐 工程实践
  • 锁的粒度：持锁时间尽量短。用 { let mut g = m.lock().unwrap(); *g += 1; } 块限定 guard 生命周期
  • 避免死锁：不要嵌套获取多个锁；如果必须，保持一致的加锁顺序
  • RwLock：读多写少时用 RwLock<T>——允许多个读者同时访问，写者独占
  • 原子类型：简单计数器可用 AtomicUsize 等原子类型，无需 Mutex 开销
  • 中毒处理：lock().unwrap() 在持锁线程 panic 后会 panic——生产代码可用 .lock().unwrap_or_else(|e| e.into_inner())
✏️ 示例分析与易错点
RUSTmutex_guard_scope.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let data = Arc::new(Mutex::new(vec![1, 2, 3]));

    let data_clone = Arc::clone(&data);
    let handle = thread::spawn(move || {
        let mut vec = data_clone.lock().unwrap();
        vec.push(4);
        println!("子线程: {:?}", *vec);
        // MutexGuard 在函数结束时 drop
    });

    handle.join().unwrap(); // 等子线程释放锁

    let vec = data.lock().unwrap();
    println!("主线程: {:?}", *vec);
}
输出 
子线程: [1, 2, 3, 4]
主线程: [1, 2, 3, 4]

易错点：在 println!("{:?}", data.lock().unwrap()) 中，MutexGuard 的临时变量在整个语句结束后才 drop。如果链式调用不当，锁的持有时间可能比你预期的长。

05Send 与 Sync

🎯 为什么需要了解这个

前面讲的"spawn 要求 Send"、"Rc 不能跨线程"都涉及一个底层机制：Rust 用两个 marker trait——Send 和 Sync——在类型系统层面编码并发安全。理解它们，就能理解编译器为什么允许或拒绝某些跨线程操作。

🔍 核心思路
  • Send：T 的值可以安全地从一个线程 move 到另一个线程
  • Sync：&T 可以安全地被多个线程同时持有（即 T 的不可变引用可跨线程共享）
  • 等价关系：T: Sync ⟺ &T: Send
  • 自动推导：如果结构体的所有字段都是 Send，那结构体自动是 Send；Sync 同理
  • marker trait：没有方法，纯粹是编译期标记
📐 规则与语法
类型 Send? Sync? 原因
i32, bool, String✅✅无内部共享状态
Vec<T> (T: Send)✅✅所有字段 Send+Sync
Rc<T>❌❌引用计数非原子
Arc<T> (T: Send+Sync)✅✅原子引用计数
Mutex<T> (T: Send)✅✅锁保护了内部数据
Cell<T>✅❌内部可变但无锁保护
*mut T❌❌裸指针无安全保证
RUSTsend_sync_demo.rs
use std::sync::{Arc, Mutex};

// 泛型约束：T 可以跨线程传递和共享
fn share_across_threads<T: Send + Sync + 'static + std::fmt::Debug>(val: T) {
    let shared = Arc::new(Mutex::new(val));
    let c = Arc::clone(&shared);
    std::thread::spawn(move || {
        let data = c.lock().unwrap();
        println!("子线程看到: {:?}", *data);
    }).join().unwrap();
}

fn main() {
    share_across_threads(42);
    share_across_threads(String::from("hello"));
    // share_across_threads(std::rc::Rc::new(1));
    // ❌ 编译错误: Rc<i32> 不满足 Send
}
输出 
子线程看到: 42
子线程看到: "hello"
🌐 工程实践
  • 大部分类型自动满足 Send + Sync，你很少需要手动实现
  • unsafe impl Send/Sync：只有在封装 FFI 或裸指针时才需要手动标记，且必须确保安全性
  • 泛型约束：编写跨线程通用代码时，别忘了加 T: Send + 'static 约束
  • 编译器是你的盟友：如果忘了 Send/Sync 约束，编译器会明确告诉你哪个类型不满足
✏️ 示例分析与易错点
RUSTnot_send.rs
use std::rc::Rc;
use std::thread;

fn main() {
    let data = Rc::new(42);

    // ❌ 编译错误：Rc<i32> cannot be sent between threads safely
    // thread::spawn(move || {
    //     println!("{}", data);
    // });

    // ✅ 改用 Arc
    use std::sync::Arc;
    let data = Arc::new(42);
    let data_clone = Arc::clone(&data);
    thread::spawn(move || {
        println!("Arc 版本: {}", data_clone);
    }).join().unwrap();
}
输出 
Arc 版本: 42

易错点：编译错误 "Rc<T> cannot be sent between threads safely" 的根本原因是 Rc: !Send。解决方案是换成 Arc，而不是用 unsafe 绕过。
