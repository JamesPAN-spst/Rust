Ch 4.1 线程并发

这一章先把 Rust 的线程并发放回工程语境：为什么还需要 OS 线程、所有权为什么会在并发场景变得更严格，以及消息传递与共享状态两条路线分别适合什么问题。

🎯 学习目标

  • 理解 std::thread 的基本用法与线程 join 的意义
  • 掌握为什么闭包进入线程时经常需要 move
  • 理解 channel、Arc、Mutex 三类并发协作手段的区别
  • 建立对 Send / Sync 的直觉，知道编译器在阻止什么风险

📋 前置知识

建议先完成 Ownership、Borrowing、Trait 和错误处理章节。线程相关报错往往不是“并发语法难”，而是所有权、生命周期和共享可变性在并发环境下被放大了。

4.1.1 为什么在 Rust 里还要学线程？

🤔

既然 Unit 4 还会讲 async，为什么还不能直接把“并发 = 异步”画等号？

线程是操作系统调度的执行单元，适合 CPU 密集工作、阻塞式库、以及需要真正并行跑在多个核心上的任务。异步更像是在较少线程上高效复用等待时间，它不等于自动并行。

正式规则：std::thread::spawn 会启动新线程并返回 JoinHandle；调用 join() 可以等待子线程完成并拿回结果或 panic 信息。

Rustspawn_join.rs
use std::thread;

fn main() {
    let handle = thread::spawn(|| {
        for n in 1..=3 {
            println!("worker: {n}");
        }
    });

    handle.join().unwrap();
    println!("main done");
}
输出 
worker: 1
worker: 2
worker: 3
main done

4.1.2 为什么线程闭包经常要写 move？

🧠

如果主线程可能先结束，而子线程还在跑，那么子线程里借来的数据到底还能不能活着？

线程闭包往往会比当前作用域活得更久，因此直接借用外部局部变量通常不安全。move 会把需要的数据所有权转移进线程，让线程自己负责那份值的生命周期。

直觉重点：move 不是“为了语法通过”；它是在说“这份数据之后归新线程管理”。

Rustmove_into_thread.rs
use std::thread;

fn main() {
    let values = vec![10, 20, 30];

    let handle = thread::spawn(move || {
        println!("child got: {:?}", values);
    });

    handle.join().unwrap();
}
输出 
child got: [10, 20, 30]
通常不能，因为所有权已经转进了线程闭包。如果你还要在主线程保留一份，就需要显式 clone，或者改成借用且确保生命周期安全。

4.1.3 channel：让线程通过消息，而不是抢同一份数据

📨

两个线程协作时，一定要共享同一个可变变量吗？如果只是“把结果交给另一边”，有没有更干净的方式？

Rust 标准库的 channel 提供“发送者—接收者”模型。发送端把值 move 进去，接收端再取出来，这样资源归属会随着消息流动而转移，而不是两边同时修改同一份状态。

正式规则：std::sync::mpsc 是 multi-producer, single-consumer；每条消息都有明确所有权流向，适合任务分发和结果回传。

Rustchannel.rs
use std::sync::mpsc;
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();

    thread::spawn(move || {
        tx.send(String::from("job finished")).unwrap();
    });

    let message = rx.recv().unwrap();
    println!("received: {message}");
}
输出 
received: job finished

4.1.4 Arc<Mutex<T>>：共享状态路线的最小起点

🔒

如果多个线程确实需要共同维护一个计数器、缓存或队列，消息传递还够用吗？

这时就会走“共享状态”路线：Arc 负责让多个线程共享所有权，Mutex 负责保证同一时刻只有一个线程能修改内部数据。两者组合是 Rust 并发入门最常见的模式。

使用边界：Arc 解决的是“谁持有”，Mutex 解决的是“谁此刻能改”；它们不是同一个问题。

Rustarc_mutex.rs
use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..3 {
        let counter = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut value = counter.lock().unwrap();
            *value += 1;
        }));
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("count = {}", *counter.lock().unwrap());
}
输出 
count = 3
Rc 不是线程安全的引用计数，RefCell 也只做单线程下的运行时借用检查。跨线程共享状态时，需要 Arc 和真正的同步原语，如 Mutex。

4.1.5 Send / Sync：编译器如何表达“能不能跨线程”

🧭

为什么有的类型可以放心在线程间转移，有的类型一跨线程编译器就会拦下来？

Send 的直觉是“这个值的所有权可以安全地转移到另一个线程”；Sync 的直觉是“这个类型的共享引用 &T 可以安全地在多线程之间共享”。它们是 Rust 并发安全模型的重要边界说明。

记忆方式：Send 关注“搬过去是否安全”，Sync 关注“大家一起看这份值是否安全”。

Rustsend_sync_intuition.rs
use std::sync::Arc;
use std::thread;

fn main() {
    let data = Arc::new(vec![1, 2, 3]);
    let shared = Arc::clone(&data);

    let handle = thread::spawn(move || {
        println!("child sees {}", shared[0]);
    });

    handle.join().unwrap();
    println!("main sees {}", data[1]);
}
输出 
child sees 1
main sees 2

📌 章节总结

  • 线程适合真正并行的 CPU 工作，以及无法避免的阻塞式任务。
  • 线程闭包常用 move，因为子线程需要拥有自己使用的数据。
  • channel 解决消息传递，Arc<Mutex<T>> 解决共享状态，它们适用于不同协作模型。
  • Send / Sync 把“能否跨线程安全使用”编码进了类型系统。
