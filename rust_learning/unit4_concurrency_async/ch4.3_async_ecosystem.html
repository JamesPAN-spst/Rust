Unit 4 · 并发与异步

异步生态

Rust 的 async/await 只提供语言级原语——实际的运行时、HTTP 框架、数据库驱动都来自生态。本章梳理 Tokio 生态的分层架构，攻克 async trait 的核心挑战，掌握取消/超时/背压三大运维模式，最后建立同步 vs 异步的选型判断框架。

01Tokio 生态分层

🎯 为什么需要了解这个

刚接触 Rust 异步时，你会遇到一堆库名：tokio、hyper、axum、tower、tonic、reqwest……它们是什么关系？谁依赖谁？理解生态分层，才能在选型时做出正确决策，排查问题时知道去哪一层找原因。

🔍 核心思路

Rust 异步生态是分层架构，从底到顶：

层次 职责 代表
语言层Future trait, async/await 语法Rust 编译器 + std
运行时层Executor + I/O 驱动 + 定时器Tokio, async-std, smol
基础设施层HTTP、gRPC、数据库协议Hyper, Tonic, sqlx
中间件层Service / Layer 抽象Tower (timeout, retry, etc.)
应用框架层路由、提取器、响应Axum, Actix-web, Poem
📐 规则与语法
TOMLCargo.toml
# 典型 Web 服务的依赖层次
[dependencies]
tokio = { version = "1", features = ["full"] }  # 运行时层
hyper = { version = "1", features = ["full"] }   # HTTP 基础设施
tower = "0.4"                                     # 中间件
axum = "0.7"                                      # 应用框架
sqlx = { version = "0.7", features = ["runtime-tokio", "postgres"] }
reqwest = { version = "0.12", features = ["json"] } # HTTP 客户端
tokio-util = "0.7" # 实用工具：codec、compat 等
RUSTaxum_hello.rs
use axum::{routing::get, Router};

async fn hello() -> &'static str {
    "Hello, World!"
}

#[tokio::main]
async fn main() {
    let app = Router::new().route("/", get(hello));

    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000")
        .await
        .unwrap();
    axum::serve(listener, app).await.unwrap();
}
curl http://localhost:3000 
Hello, World!
🌐 工程实践
  • 统一运行时：一个项目中尽量只用一个运行时（通常是 Tokio）。混用 async-std 和 Tokio 会出兼容性问题
  • features 按需开启：Tokio 的 features 是模块化的——只开 rt、net、time 等你需要的，full 在开发时方便但会拉入所有功能
  • Tower 中间件是可组合的：超时、限流、重试可以像乐高一样叠加，不需要每个框架重复实现
  • Tonic：基于 Tokio + Hyper 的 gRPC 框架，适合微服务间通信
✏️ 示例分析与易错点
RUSTtower_middleware.rs
use axum::{routing::get, Router};
use tower::ServiceBuilder;
use tower_http::timeout::TimeoutLayer;
use std::time::Duration;

async fn slow_handler() -> &'static str {
    tokio::time::sleep(Duration::from_secs(10)).await;
    "done"
}

#[tokio::main]
async fn main() {
    let app = Router::new()
        .route("/slow", get(slow_handler))
        .layer(
            ServiceBuilder::new()
                .layer(TimeoutLayer::new(Duration::from_secs(5)))
                // 可以继续叠加：.layer(RateLimitLayer::new(...))
        );

    // 访问 /slow 会在 5 秒后超时，而非等 10 秒
    let listener = tokio::net::TcpListener::bind("0.0.0.0:3000")
        .await.unwrap();
    axum::serve(listener, app).await.unwrap();
}

易错点：layer 的叠加顺序很重要——外层先执行。如果 Timeout 在 RateLimit 外面，超时包含了排队时间；如果在里面，只计算实际处理时间。

02async trait 挑战

🎯 为什么需要了解这个

想定义一个异步接口——比如 trait HttpClient { async fn get(&self, url: &str) -> Response; }——你会发现事情比同步 trait 复杂得多。async trait 的挑战是 Rust 异步生态中的经典痛点，也是面试高频问题。

🔍 核心思路

核心矛盾：async fn 返回的 Future 是编译器生成的匿名类型，每个实现者的 Future 大小不同。而 trait 方法需要统一的返回类型签名。

  • Rust 1.75 之前：trait 中不能写 async fn，需要用 async-trait 宏
  • Rust 1.75+：原生支持 async fn in trait（通过 RPITIT），静态分发场景直接可用
  • 动态分发（dyn Trait）：仍然需要 Box<dyn Future> 或 trait-variant 等方案
📐 规则与语法
RUSTnative_async_trait.rs
// Rust 1.75+：原生 async fn in trait
trait Fetcher {
    async fn fetch(&self, url: &str) -> String;
}

struct HttpFetcher;

impl Fetcher for HttpFetcher {
    async fn fetch(&self, url: &str) -> String {
        // 实际项目中用 reqwest 等
        format!("Response from {}", url)
    }
}

// ✅ 静态分发：编译器知道具体类型，可以直接使用
async fn use_fetcher(f: &impl Fetcher) {
    let result = f.fetch("https://example.com").await;
    println!("{}", result);
}

// ⚠️ 动态分发：dyn Fetcher 不能直接用
// async fn use_dyn_fetcher(f: &dyn Fetcher) { ... }
// 编译错误！Future 大小不确定
RUSTasync_trait_macro.rs
// 使用 async-trait 宏（支持动态分发 dyn Trait）
use async_trait::async_trait;

#[async_trait]
trait Storage {
    async fn save(&self, key: &str, value: &str);
    async fn load(&self, key: &str) -> Option<String>;
}

#[async_trait]
impl Storage for MemoryStore {
    async fn save(&self, key: &str, value: &str) {
        // ...
    }
    async fn load(&self, key: &str) -> Option<String> {
        // ...
        None
    }
}

// 宏展开后大致等价于：
// fn save(&self, key: &str, value: &str)
//     -> Pin<Box<dyn Future<Output = ()> + Send + '_>>
struct MemoryStore;
🌐 工程实践
  • 优先用原生 async trait：如果只需要静态分发（impl Fetcher 或泛型 T: Fetcher），直接用原生语法
  • 需要 dyn Trait 时：使用 async-trait 宏或 trait-variant crate
  • 性能考量：async-trait 的 Box<dyn Future> 有一次堆分配。hot path 中可考虑 enum dispatch 避免
  • Send bound：默认 async-trait 生成的 Future 是 Send 的。如果不需要跨线程，可用 #[async_trait(?Send)]
✏️ 示例分析与易错点
RUSTasync_trait_send.rs
use async_trait::async_trait;

#[async_trait]  // 默认：Future: Send
trait Worker {
    async fn run(&self);
}

// 如果实现中持有 !Send 类型跨 await：
struct LocalWorker {
    // data: Rc<String>,  // Rc 不是 Send
}

// ❌ 如果 Worker 要求 Send，但你在 async 方法中用了 Rc，会编译错误
// ✅ 解决方案 1：改用 Arc
// ✅ 解决方案 2：用 #[async_trait(?Send)] 取消 Send 要求

#[async_trait(?Send)]
trait LocalWorkerTrait {
    async fn run(&self);
    // 生成的 Future 不要求 Send，可以持有 Rc 等非 Send 类型
}

易错点：async-trait 默认要求返回的 Future 是 Send。如果你的实现需要持有 Rc、Cell 等非 Send 类型，要么改用 Send 等价物（Arc），要么用 ?Send 标注。

03取消、超时与背压

🎯 为什么需要了解这个

生产环境中的异步系统必须处理三个关键问题：任务超时了怎么办？不需要的任务怎么取消？上游发太快下游处理不过来怎么办？这三个模式是构建健壮异步服务的基础。

🔍 核心思路
  • 取消（Cancellation）：Rust 的模型是 drop = cancel。drop 一个 Future 就停止其执行
  • 超时（Timeout）：用 tokio::time::timeout 包装 Future，超时后 drop 内部 Future
  • 背压（Backpressure）：用有界 channel 或 Semaphore 限制并发数，防止内存无限增长
📐 规则与语法
RUSTtimeout_pattern.rs
use tokio::time::{timeout, Duration};

async fn slow_operation() -> String {
    tokio::time::sleep(Duration::from_secs(10)).await;
    "完成".to_string()
}

#[tokio::main]
async fn main() {
    match timeout(Duration::from_secs(3), slow_operation()).await {
        Ok(result) => println!("成功: {}", result),
        Err(_) => println!("超时！操作已取消"),
        // slow_operation 的 Future 被 drop（取消）
    }
}
输出 
超时！操作已取消
RUSTcancellation_token.rs
use tokio_util::sync::CancellationToken;
use tokio::time::{sleep, Duration};

#[tokio::main]
async fn main() {
    let token = CancellationToken::new();

    // 工作任务
    let worker_token = token.clone();
    let worker = tokio::spawn(async move {
        tokio::select! {
            _ = worker_token.cancelled() => {
                println!("收到取消信号，清理并退出");
            }
            _ = async {
                loop {
                    println!("工作中...");
                    sleep(Duration::from_secs(1)).await;
                }
            } => {}
        }
    });

    // 3 秒后取消
    sleep(Duration::from_secs(3)).await;
    token.cancel();
    worker.await.unwrap();
    println!("工作任务已优雅退出");
}
输出 
工作中...
工作中...
工作中...
收到取消信号，清理并退出
工作任务已优雅退出
RUSTbackpressure.rs
use tokio::sync::{mpsc, Semaphore};
use std::sync::Arc;

#[tokio::main]
async fn main() {
    // 方式 1：有界 channel 自然背压
    let (tx, mut rx) = mpsc::channel::<i32>(10); // 容量 10

    tokio::spawn(async move {
        for i in 0..100 {
            // 当 channel 满时，send 会暂停（背压）
            tx.send(i).await.unwrap();
            println!("发送: {}", i);
        }
    });

    while let Some(val) = rx.recv().await {
        println!("处理: {}", val);
        tokio::time::sleep(tokio::time::Duration::from_millis(50)).await;
    }
}
🌐 工程实践
  • Semaphore 限流：Arc<Semaphore> 可限制同时运行的任务数——如限制并发数据库连接为 20
  • select! 超时模式：tokio::select! 可以同时等待操作完成或超时，比 timeout() 更灵活
  • 优雅退出：CancellationToken + select! 是实现优雅关闭（graceful shutdown）的标准模式
  • 注意 drop 的副作用：取消可能在任意 .await 点发生。如果你在两个 .await 之间做了半个操作（如写了一半数据），需要确保 Drop 能清理
✏️ 示例分析与易错点
RUSTselect_timeout.rs
use tokio::time::{sleep, Duration};

async fn fetch_data() -> String {
    sleep(Duration::from_secs(5)).await;
    "data".to_string()
}

#[tokio::main]
async fn main() {
    tokio::select! {
        result = fetch_data() => {
            println!("获取到: {}", result);
        }
        _ = sleep(Duration::from_secs(2)) => {
            println!("超时！fetch_data 被取消");
            // fetch_data 的 Future 被 drop
        }
    }
}
输出 
超时！fetch_data 被取消

易错点：在 select! 中，没胜出的分支的 Future 会被 drop（取消）。如果这个 Future 中有需要完成的清理工作（如提交事务），需要特别处理——可以用 tokio::pin! 固定 Future 然后在多次 select 中复用。

04同步 vs 异步选型

🎯 为什么需要了解这个

异步不是银弹。它增加了代码复杂度（生命周期更难处理、错误信息更晦涩、调试更困难）。选错了模型，项目可能既慢又复杂。掌握选型判断框架，才能在正确的场景用正确的工具。

🔍 核心思路
维度 同步（线程） 异步
并发模型一任务一线程M:N（多任务少线程）
调度OS 抢占式协作式（.await 让出）
适合场景CPU 密集、少量并发I/O 密集、大量并发
代码复杂度低（线性思维）高（状态机、Pin、生命周期）
内存开销每线程 ~2-8MB 栈每任务几百字节
调试难度较易（标准调用栈）较难（异步栈帧、状态机）
📐 规则与语法

选型决策树：

  1. 并发连接/任务数 > 几百？→ 考虑异步
  2. 主要是 I/O 等待（网络、数据库）？→ 异步更合适
  3. 主要是 CPU 密集计算？→ 同步线程 + rayon 并行更合适
  4. 需要调用大量阻塞 C 库？→ 同步更简单（或 spawn_blocking）
  5. 团队对 async Rust 不熟悉？→ 先用同步，后续按需迁移
RUSTsync_vs_async.rs
// ✅ 场景 1：CPU 密集——用线程 + rayon
use rayon::prelude::*;

fn process_data_sync(data: &[i64]) -> Vec<i64> {
    data.par_iter()          // rayon 并行迭代器
        .map(|x| x * x + 1) // CPU 密集计算
        .collect()
}

// ✅ 场景 2：大量网络 I/O——用异步
async fn fetch_all_async(urls: Vec<String>) -> Vec<String> {
    let mut handles = vec![];
    for url in urls {
        handles.push(tokio::spawn(async move {
            // reqwest::get(&url).await... (模拟)
            format!("Response from {}", url)
        }));
    }
    let mut results = vec![];
    for h in handles {
        results.push(h.await.unwrap());
    }
    results
}

// ✅ 场景 3：混合——异步外壳 + spawn_blocking 内核
async fn mixed_workload(url: &str) -> i64 {
    let data = fetch_data(url).await;        // I/O：用异步
    let result = tokio::task::spawn_blocking(move || {
        heavy_computation(&data)               // CPU：用 blocking 线程池
    }).await.unwrap();
    result
}

async fn fetch_data(url: &str) -> Vec<i64> { vec![1,2,3] }
fn heavy_computation(data: &[i64]) -> i64 { data.iter().sum() }
🌐 工程实践
  • "异步传染性"：一旦用了 async，调用链上所有函数都需要是 async。起步时想清楚，后续改动成本大
  • 混合模式最常见：大多数真实项目是 "异步入口 + spawn_blocking 处理 CPU 密集 + 同步库调用"
  • 测试：异步测试用 #[tokio::test]；同步测试更容易编写和调试
  • 简单命令行工具？同步就够了——异步的机器开销和心智开销都不值得
  • rayon vs tokio：rayon 用于数据并行（并行处理一个大数组），tokio 用于任务并发（同时等待多个 I/O）
✏️ 示例分析与易错点
RUSTwrong_choice.rs
// ❌ 错误选型：用异步做 CPU 密集计算
#[tokio::main]
async fn main() {
    // 这里没有任何 I/O，异步毫无优势
    // 反而增加了 Future 状态机的开销
    let result = async {
        let mut sum: u64 = 0;
        for i in 0..10_000_000 {
            sum += i;
        }
        sum
    }.await;
    println!("{}", result);
}

// ✅ 正确选型：同步直接计算
fn main() {
    let mut sum: u64 = 0;
    for i in 0..10_000_000 {
        sum += i;
    }
    println!("{}", sum);
}

易错点：不要因为"异步很酷"就到处用 async。纯 CPU 计算的代码包一层 async 块不会让它变快——反而有额外开销。async 只在有 I/O 等待时才有意义。
