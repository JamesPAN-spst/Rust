Unit 4 · 并发与异步

Ch 4.3 异步生态

学会 async 语法只是第一步。真正做项目时，你会面对运行时、网络栈、数据库驱动、取消语义、背压控制和 API 设计取舍。本章帮助你建立生态视角，而不是只记某个库的调用方式。

🎯 学习目标

  • 理解 Tokio-like 异步生态通常由哪些层组成
  • 知道 async trait 设计为何仍有现实 caveat，如何保持 API 判断力
  • 建立对取消、超时、背压的基础直觉
  • 能根据负载类型、库生态和维护成本判断同步或异步方案

📋 前置知识

建议先完成线程并发与异步编程两章。本章默认你已经知道 Future、executor、await 的基本直觉，现在要把这些概念映射到实际生态与工程选型上。

4.3.1 Tokio-like 异步生态通常分成哪几层？

🗺️

项目里说“我们用 Tokio”，到底是在说一个库、一个运行时，还是整套异步基础设施？

现实中的异步生态通常是分层的：最底层是 Future / Pin / Waker 这类语言与标准库抽象；再往上是 runtime / executor；再上面是网络、时间、同步原语、任务管理；最外层则是 HTTP、数据库、消息队列等应用框架和驱动。

实战直觉：Tokio 更像“运行时 + 常用基础设施平台”，而不是只提供一个 .await 语法糖。

生态分层mental_model.txt
语言层：async / await, Future
运行时层：executor, timer, I/O driver
基础设施层：task spawning, async sync primitives, channels
应用层：HTTP server, database client, gRPC, queues
说明 
写业务时你常直接碰到应用层；
排查性能和阻塞问题时则会一路追到运行时层。

4.3.2 async trait 为什么总让人觉得“有一点绕”？

🧵

既然 async fn 很自然，为什么一进 trait 设计就开始冒出 boxing、宏、对象安全这些词？

async fn 背后会生成一个具体但匿名的 Future 类型，而 trait 方法需要在接口层表达返回类型。现实项目里经常借助关联类型、Pin<Box<dyn Future<...>>> 或宏方案来桥接，这会带来分配、对象安全或可读性上的权衡。

设计重点：async trait 不是不能用，而是要明确自己是在追求零成本静态分发、接口稳定性，还是快速统一多实现的工程便利。

Rustasync_trait_shape.rs
use std::future::Future;
use std::pin::Pin;

trait Loader {
    fn load(&self) -> Pin<Box<dyn Future<Output = String> + Send + '_>>;
}
说明 
这类签名能工作，但可读性和分配成本
都值得在设计时主动权衡。
先问接口是否真的需要 trait 抽象。如果只有一两个实现，直接具体类型或泛型往往更简单；只有在插件化、多后端或测试替身需求明确时，再承受 async trait 的复杂度更划算。

4.3.3 取消、超时与 backpressure：异步系统不是“无限接活”

🌊

如果请求来得比处理速度快，会发生什么？如果调用方已经不关心结果了，还要继续把任务跑完吗？

异步系统的难点之一是任务数量很容易暴涨，所以必须有取消、超时和背压控制。取消让不再需要的工作尽快停下；超时避免永远等待；backpressure 则是在生产速度高于消费速度时主动施加限制，而不是让队列无边界膨胀。

工程直觉：吞吐不是“接得越多越好”，而是“系统在压力上升时仍能以可控方式退化”。

Rustbackpressure_intuition.rs
// 伪代码：限制并发和等待时间
// semaphore.acquire().await?;
// timeout(Duration::from_secs(2), handle_request()).await?;
// bounded_channel.send(job).await?;
说明 
限制并发、加超时、使用有界队列，
都是常见的背压与取消策略。

4.3.4 什么时候该选同步，什么时候该选异步？

⚖️

既然异步听起来吞吐更高，为什么很多 CLI、小工具甚至一部分后端服务仍然坚持同步写法？

异步最值得投入的场景，是高并发 I/O、连接数多、等待时间长、生态成熟且团队能驾驭运行时复杂度。若程序规模小、并发需求有限、依赖 mostly blocking、或主要瓶颈在 CPU，同步代码往往更简单、更直接，也更容易调试。

选型原则：先看负载模型和依赖生态，再看团队维护成本；不要因为“async 看起来先进”就无条件全盘切换。

选型清单sync_vs_async.txt
偏向同步：
- CLI、批处理、小型服务
- CPU 密集或阻塞库很多
- 并发规模有限，代码简单最重要

偏向异步：
- 高并发网络 I/O
- 连接数大，等待时间长
- 需要在少量线程上管理大量任务
说明 
架构选择不是站队；
而是看工作负载和系统约束。
“性能敏感 = 一定要 async” 是误区。若瓶颈是 CPU 算法、序列化或压缩，线程、SIMD、批处理策略可能比 async 更关键。

📌 章节总结

  • Tokio-like 生态是分层的：语言抽象、运行时、基础设施和应用库共同组成项目体验。
  • async trait 的难点主要来自匿名 Future 的接口表达与分发权衡。
  • 取消、超时和 backpressure 是异步系统可靠性的关键，而不是附加功能。
  • 同步还是异步要看负载类型、依赖生态和团队维护成本，不是单纯追新。
