Ch 6.C 系统编程方向

系统编程是 Rust 的"主场"。从 CLI 工具到操作系统抽象，从嵌入式 no_std 到 FFI 和内核模块探索，这一章展示 Rust 如何在最底层的工程场景中替代 C，同时提供更强的安全保障。

🎯 学习目标

  • 理解 Rust 在系统编程中替代 C/C++ 的核心动机和现实案例
  • 掌握用 clap 构建健壮 CLI 工具的方法
  • 了解文件系统操作和进程管理的 Rust API 设计
  • 建立对 FFI、no_std 嵌入式开发和 Linux 内核 Rust 支持的基本认知

📋 前置知识

建议已掌握所有权、Trait、错误处理（Result/Option）、以及 unsafe 基础。系统编程经常与 C 交互、操作裸指针、处理底层错误码——这些都需要对 Rust 安全边界有清晰认识。

6.C.1 为什么用 Rust 做系统编程？

🤔

操作系统、驱动、嵌入式固件——这些代码运行在最底层，一个内存错误就可能导致系统崩溃或安全漏洞。C 语言统治这个领域 50 年了，Rust 凭什么介入？

Rust 提供了与 C 相同的底层控制能力（裸指针、内联汇编、零成本抽象），同时通过所有权系统在编译期排除内存错误。这不是"用安全换性能"的妥协，而是"两者兼得"的设计目标。Linux 内核从 6.1 版本开始正式支持 Rust，Android、Windows 也在关键组件中引入 Rust——这标志着系统编程范式的转变。

现实信号：Linux 6.1+ 支持 Rust 内核模块；Android 新增安全组件使用 Rust；Cloudflare、Discord 等公司用 Rust 重写关键系统服务。

Rustos_info.rs
use std::env;

fn main() {
    // OS 级别信息获取
    println!("OS: {}", env::consts::OS);
    println!("Arch: {}", env::consts::ARCH);
    println!("Family: {}", env::consts::FAMILY);

    // 环境变量读取
    match env::var("HOME") {
        Ok(home) => println!("HOME = {home}"),
        Err(_) => println!("HOME not set"),
    }
}
输出 
OS: linux
Arch: x86_64
Family: unix
HOME = /home/user

6.C.2 用 clap 构建 CLI 工具

⌨️

命令行工具是系统编程最常见的交付形式。手动解析 std::env::args() 既冗长又容易出错——有没有声明式的方式定义参数，同时自动生成帮助信息和错误提示？

clap 是 Rust 生态中最流行的命令行参数解析库。通过 derive 宏，你可以用结构体声明参数，clap 自动处理解析、验证、帮助文本生成。ripgrep、fd、bat 等知名 CLI 工具都基于 Rust 构建，展示了 Rust CLI 工具的高性能和用户体验。

核心模式：用 #[derive(Parser)] 标注结构体，字段即参数。clap 自动生成 --help、类型验证和错误信息。

Rustclap_example.rs
// 需要在 Cargo.toml 添加: clap = { version = "4", features = ["derive"] }
use clap::Parser;

/// 一个简单的文件搜索工具
#[derive(Parser, Debug)]
#[command(name = "minigrep", about = "Search for a pattern in files")]
struct Args {
    /// 要搜索的模式
    pattern: String,

    /// 要搜索的文件路径
    path: String,

    /// 忽略大小写
    #[arg(short, long, default_value_t = false)]
    ignore_case: bool,
}

fn main() {
    // 实际运行: minigrep "hello" src/main.rs --ignore-case
    // 这里演示结构定义
    println!("clap 的 derive 模式让 CLI 参数定义变成了结构体声明");
    println!("字段类型即验证规则，doc comment 即帮助文本");
}
输出 
clap 的 derive 模式让 CLI 参数定义变成了结构体声明
字段类型即验证规则，doc comment 即帮助文本
  • clap — 参数解析（derive 或 builder 模式）
  • serde + toml/serde_json — 配置文件读写
  • indicatif — 进度条
  • colored / ansi_term — 终端着色
  • dialoguer — 交互式输入（选择、确认）
  • tracing — 结构化日志

6.C.3 文件系统操作与进程管理

📂

系统工具需要遍历目录、读写文件、启动子进程并管理输出。Rust 标准库提供了哪些 OS 级抽象？和 C 的 POSIX API 有什么不同？

Rust 的 std::fs 和 std::process 模块封装了操作系统 API，但与 C 不同的是：路径用 Path/PathBuf（区分 UTF-8 和 OS 字符串），文件操作返回 Result（强制处理错误），进程管理通过 Command builder 模式安全构造。

跨平台注意：Path 使用 OsStr 而非 &str，因为文件系统路径不保证是合法 UTF-8。用 .to_str() 转换时要处理 None。

Rustfs_and_process.rs
use std::fs;
use std::process::Command;
use std::path::Path;

fn main() -> std::io::Result<()> {
    // 遍历目录
    let dir = Path::new(".");
    for entry in fs::read_dir(dir)? {
        let entry = entry?;
        let kind = if entry.file_type()?.is_dir() { "DIR" } else { "FILE" };
        println!("{kind}: {}", entry.file_name().to_string_lossy());
    }

    // 执行子进程并捕获输出
    let output = Command::new("echo")
        .arg("hello from subprocess")
        .output()?;

    println!("subprocess: {}", String::from_utf8_lossy(&output.stdout));
    Ok(())
}
输出 
DIR: src
FILE: Cargo.toml
...
subprocess: hello from subprocess

6.C.4 FFI、no_std 与内核模块探索

🔧

系统编程有时需要直接调用 C 库、在没有操作系统的裸机上运行、甚至编写内核模块。Rust 能做到这些吗？安全边界在哪里？

FFI（Foreign Function Interface）让 Rust 调用 C 函数或被 C 调用。extern "C" 声明 C ABI 函数，#[no_mangle] 导出符号。no_std 模式去掉标准库依赖，只保留 core 和可选的 alloc，适合嵌入式和内核场景。Linux 从 6.1 开始在内核中支持 Rust 模块，使用 kernel crate 包装内核 API。

安全边界：FFI 调用和 no_std 中的裸指针操作必须在 unsafe 块中。这不是 Rust 的"漏洞"，而是明确标记安全责任由开发者承担的边界。

Rustffi_example.rs
use std::ffi::CString;
use std::os::raw::c_char;

// 声明要调用的 C 函数
extern "C" {
    fn strlen(s: *const c_char) -> usize;
}

fn safe_strlen(s: &str) -> usize {
    let c_str = CString::new(s).expect("string contains null byte");
    // unsafe 块明确标记：我们正在跨越安全边界
    unsafe { strlen(c_str.as_ptr()) }
}

fn main() {
    let length = safe_strlen("Hello, FFI!");
    println!("C strlen says: {length}");
}
输出 
C strlen says: 11

#![no_std] 告诉编译器不链接标准库（std），只使用 core（无堆分配、无 OS 依赖）。嵌入式开发中，你需要自行提供 panic handler 和可能的内存分配器。典型目标如 ARM Cortex-M 微控制器：

#![no_std]
#![no_main]

use core::panic::PanicInfo;

#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    loop {}
}

#[no_mangle]
pub extern "C" fn _start() -> ! {
    // 嵌入式入口：直接操作硬件寄存器
    loop {}
}

📌 章节总结

  • Rust 提供 C 级底层控制力的同时，在编译期保证内存安全——这是系统编程的范式转变。
  • clap 让 CLI 工具的参数定义变成声明式结构体，配合 Rust 生态可以快速构建高质量命令行工具。
  • std::fs 和 std::process 提供类型安全的文件系统和进程管理 API，Path 处理跨平台路径差异。
  • FFI 通过 extern "C" 和 unsafe 实现 C 互操作；no_std 支持嵌入式和内核开发；Linux 已在内核中支持 Rust。
