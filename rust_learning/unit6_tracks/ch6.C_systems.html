Ch 6.3 系统编程

系统编程是 Rust 的"本命领域"——从命令行工具到操作系统内核，Rust 让底层编程既安全又高效。这是 C 语言统治了 50 年的领域，Rust 正在成为有力的挑战者。

01系统编程优势

🎯 为什么需要了解这个

系统编程意味着直接与硬件和 OS 交互——内存管理、进程控制、设备驱动、性能优化。Rust 让你在不牺牲安全的前提下做这些事。

🔍 核心思路
  • C 级性能：无 GC、零成本抽象、可直接操作内存
  • 编译期安全：不需要 Valgrind、AddressSanitizer 来找内存 bug
  • 现代工具链：Cargo 包管理、交叉编译、测试框架——C 从未有过这么好的开发体验
  • 渐进式 unsafe：安全代码是默认模式，需要时才用 unsafe 下降一层
📐 规则与语法
RUSTsystem_programming.rs
// Rust 系统编程的日常：
// 1. 精确控制内存布局
#[repr(C)]
struct PacketHeader {
    version: u8,
    length: u16,
    flags: u8,
}

// 2. 位操作
fn extract_flags(byte: u8) -> (bool, bool, bool) {
    let read   = byte & 0b100 != 0;
    let write  = byte & 0b010 != 0;
    let exec   = byte & 0b001 != 0;
    (read, write, exec)
}

// 3. 零开销迭代
fn sum_even(data: &[i32]) -> i32 {
    data.iter().filter(|&&x| x % 2 == 0).sum()
}

fn main() {
    let header = PacketHeader { version: 1, length: 64, flags: 0b101 };
    let (r, w, x) = extract_flags(header.flags);
    println!("rwx: {r}{w}{x}");
    println!("sum even: {}", sum_even(&[1,2,3,4,5,6]));
}
输出
rwx: truefalsetrue
sum even: 12
🌐 工程实践

成功案例：ripgrep（比 grep 更快的搜索工具）、fd（find 替代）、bat（cat 替代）、tokei（代码统计）、zoxide（cd 替代）。这些工具证明 Rust CLI 在性能和用户体验上都超越前辈。

✏️ 示例分析与易错点

#[repr(C)]：与 C 交互的结构体必须使用 C 内存布局，否则 Rust 可能重排字段顺序。

02CLI 工具开发

🎯 为什么需要了解这个

CLI 工具是系统编程最常见的产出。Rust 的静态编译（单一二进制）+ 极快的启动速度 + clap 的声明式参数解析让它成为 CLI 开发的理想语言。

🔍 核心思路
  • clap：参数解析。derive 模式让参数定义和数据结构合一
  • anyhow / eyre：错误处理。自动转换不同类型的错误
  • indicatif：进度条。长时间任务的用户反馈
  • colored / owo-colors：终端彩色输出
📐 规则与语法
RUSTcli_tool.rs
use clap::Parser;
use std::fs;

/// 简单的文件搜索工具
#[derive(Parser)]
#[command(name = "search", about = "在文件中搜索文本")]
struct Args {
    /// 搜索的文本模式
    pattern: String,

    /// 输入文件路径
    #[arg(short, long)]
    file: String,

    /// 忽略大小写
    #[arg(short, long, default_value_t = false)]
    ignore_case: bool,
}

fn main() -> anyhow::Result<()> {
    let args = Args::parse();
    let content = fs::read_to_string(&args.file)?;

    for (i, line) in content.lines().enumerate() {
        let matches = if args.ignore_case {
            line.to_lowercase().contains(&args.pattern.to_lowercase())
        } else {
            line.contains(&args.pattern)
        };
        if matches {
            println!("{}:{}", i + 1, line);
        }
    }
    Ok(())
}
// cargo run -- -f data.txt "hello" -i
🌐 工程实践

分发：cargo build --release 生成静态链接的单一二进制，无需运行时依赖。跨平台编译用 cross crate。

自动补全：clap 支持自动生成 bash/zsh/fish/PowerShell 的 tab 补全脚本。

✏️ 示例分析与易错点

注意：anyhow::Result 适用于应用程序（万能错误类型），库代码应使用自定义 Error 类型。

03文件系统与进程管理

🎯 为什么需要了解这个

系统编程的日常：读写文件、遍历目录、启动子进程、解析配置文件。Rust 标准库提供了类型安全的 API，第三方 crate 补充更强大的功能。

🔍 核心思路
  • 文件操作：std::fs（读写）、std::path（路径处理）、walkdir（递归遍历）
  • 进程管理：std::process::Command（子进程）、nix（Unix API）
  • 配置管理：serde + 格式 crate（toml、json、yaml）
📐 规则与语法
RUSTfs_process.rs
use std::fs;
use std::process::Command;
use std::path::Path;

fn main() -> std::io::Result<()> {
    // 文件读写
    fs::write("hello.txt", "Hello, System Programming!")?;
    let content = fs::read_to_string("hello.txt")?;
    println!("文件内容: {}", content);

    // 路径操作
    let path = Path::new("/usr/local/bin/rustc");
    println!("文件名: {:?}", path.file_name());
    println!("扩展名: {:?}", path.extension());
    println!("父目录: {:?}", path.parent());

    // 遍历目录
    for entry in fs::read_dir(".")? {
        let entry = entry?;
        let meta = entry.metadata()?;
        println!("{:?} ({}B)", entry.file_name(), meta.len());
    }

    // 子进程
    let output = Command::new("echo")
        .arg("Hello from subprocess!")
        .output()?;
    println!("子进程输出: {}", String::from_utf8_lossy(&output.stdout));

    fs::remove_file("hello.txt")?;
    Ok(())
}
🌐 工程实践

BufReader/BufWriter：处理大文件时必须使用缓冲 I/O，否则每次 read/write 都是系统调用。

tempfile crate：创建临时文件和目录，自动在 scope 结束时清理。

✏️ 示例分析与易错点

跨平台注意：路径分隔符在 Windows 是 \，Unix 是 /。始终用 Path::join 而非字符串拼接

04FFI 与外部集成

🎯 为什么需要了解这个

系统编程中经常需要调用 C 库（如 OpenSSL、SQLite、系统 API），或者让你的 Rust 库被其他语言调用。FFI 是 Rust 融入现有生态的关键桥梁。

🔍 核心思路
  • 调用 C：extern "C" { fn ... } + unsafe 调用
  • 被 C 调用：#[no_mangle] pub extern "C" fn
  • 自动绑定：bindgen（C→Rust）、cbindgen（Rust→C）
  • -sys crate 惯例：裸绑定在 foo-sys，安全封装在 foo
📐 规则与语法
RUSTffi_integration.rs
use std::ffi::{CStr, CString};
use std::os::raw::{c_char, c_int};

// 声明 C 函数
extern "C" {
    fn strlen(s: *const c_char) -> usize;
    fn getenv(name: *const c_char) -> *const c_char;
}

fn safe_getenv(name: &str) -> Option<String> {
    let c_name = CString::new(name).ok()?;
    unsafe {
        let ptr = getenv(c_name.as_ptr());
        if ptr.is_null() {
            None
        } else {
            CStr::from_ptr(ptr).to_str().ok().map(|s| s.to_owned())
        }
    }
}

fn main() {
    let c_str = CString::new("Hello, FFI!").unwrap();
    let len = unsafe { strlen(c_str.as_ptr()) };
    println!("strlen = {}", len);

    if let Some(home) = safe_getenv("HOME") {
        println!("HOME = {}", home);
    }
}
🌐 工程实践

build.rs：用 cc crate 在构建时编译 C 代码，或用 pkg-config 查找系统库。

PyO3：Rust ↔ Python FFI，可以用 Rust 写 Python 扩展模块（maturin 打包）。

✏️ 示例分析与易错点

安全抽象模式：safe_getenv 示例展示了标准的做法——unsafe 内部调用 C 函数，外部提供安全的 Rust API。

05no_std 与内核编程

🎯 为什么需要了解这个

系统编程的终极形态：没有操作系统的环境——裸机嵌入式、操作系统内核、引导加载器。Rust 的 no_std 让你在这些极端环境中仍然享受类型安全。

🔍 核心思路
  • #![no_std]：不链接 std 标准库，只用 core（不依赖 OS）
  • core：提供基础类型、trait、迭代器等。没有堆分配。
  • alloc：可选引入。提供 Vec、String 等，但需要全局分配器。
  • 嵌入式：embedded-hal trait 抽象硬件，cortex-m 支持 ARM 芯片。
📐 规则与语法
RUSTno_std_example.rs
#![no_std]
#![no_main]

use core::panic::PanicInfo;

// 裸机需要自定义 panic handler
#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    loop {}  // 死循环——裸机没有 OS 可以退回去
}

// 入口点
#[no_mangle]
pub extern "C" fn _start() -> ! {
    // 裸机代码——直接操作硬件
    let mut counter: u32 = 0;
    loop {
        counter = counter.wrapping_add(1);
        // 在实际嵌入式中，这里操作 GPIO、定时器等
    }
}

🌐 工程实践

Rust for Linux：Linux 6.1+ 支持用 Rust 编写内核模块。目标是在驱动开发中减少内存安全漏洞。

嵌入式开发：Rust 已支持 ARM Cortex-M、RISC-V 等平台。probe-rs 提供调试和烧录工具。

学习资源：Philipp Oppermann 的 "Writing an OS in Rust" 博客系列是最好的入门教程。

✏️ 示例分析与易错点

no_std 不等于 no safety：即使没有 std，借用检查、类型安全等所有编译期检查仍然有效。unsafe 仍然是明确标注的。
