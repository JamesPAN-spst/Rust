Ch 6.1 游戏开发

Rust 正在成为游戏开发的新势力——零成本抽象、无 GC 暂停、内存安全让它天然适合高性能实时应用。Bevy 引擎展示了 Rust 游戏开发的现代范式。

01Rust 游戏开发优势

🎯 为什么需要了解这个

游戏是计算机最苛刻的应用之一：16ms 内完成一帧的所有计算和渲染。理解 Rust 在这个场景下的独特优势，才能判断它是否适合你的项目。

🔍 核心思路
  • 无 GC：不存在"GC 暂停导致掉帧"的问题（Unity/C# 常见痛点）
  • 零成本抽象：高级 API 不损失底层性能
  • 编译期安全：消除 use-after-free、数据竞争等常见 bug
  • 跨平台：编译到 Windows、Linux、macOS、WASM
📐 规则与语法
RUSTgame_advantages.rs
// Rust 的性能保证：
// 1. 没有隐式堆分配
// 2. 没有运行时类型检查
// 3. 编译期决定一切

// 对比 C# (Unity):
// - void Update() { 每帧 GC 可能触发 }
// - Rust: fn update(query: Query<...>) { 无 GC，确定性性能 }

fn frame_budget_example() {
    let target_fps = 60;
    let budget_ms = 1000.0 / target_fps as f64;
    println!("每帧预算: {:.2}ms", budget_ms);
    println!("Rust: 没有 GC 暂停偷走你的帧时间");
}
🌐 工程实践

适合的游戏类型：性能敏感的游戏（实时策略、大规模模拟）、WASM 网页游戏、嵌入式游戏设备。

挑战：生态不如 Unity/Unreal 成熟、学习曲线较陡、可视化编辑器仍在发展。

✏️ 示例分析与易错点

常见误解："Rust 太安全不适合游戏" → ECS 模式完美契合 Rust 所有权。Bevy 的 Query 系统自动处理借用关系。

02Bevy 引擎与 ECS 架构

🎯 为什么需要了解这个

Bevy 是 Rust 游戏生态中最活跃的引擎，也是 ECS（Entity-Component-System）模式的现代实现。ECS 完美契合 Rust 的所有权系统——数据和逻辑分离，并行处理自然。

🔍 核心思路

ECS 三要素：

  • Entity：唯一 ID，只是一个标签
  • Component：纯数据，附加到 Entity（如 Position、Velocity、Sprite）
  • System：纯逻辑函数，通过 Query 筛选 Entity

对比传统 OOP 的继承树：ECS 用组合代替继承，更灵活。

📐 规则与语法
RUSTbevy_ecs_demo.rs
use bevy::prelude::*;

// Component 定义——纯数据
#[derive(Component)]
struct Position { x: f32, y: f32 }

#[derive(Component)]
struct Velocity { dx: f32, dy: f32 }

#[derive(Component)]
struct Health(f32);

// System——纯逻辑
fn movement_system(mut query: Query<(&Velocity, &mut Position)>) {
    for (vel, mut pos) in &mut query {
        pos.x += vel.dx;
        pos.y += vel.dy;
    }
}

fn health_display(query: Query<(&Health, &Position)>) {
    for (hp, pos) in &query {
        println!("HP: {:.0} at ({:.1}, {:.1})", hp.0, pos.x, pos.y);
    }
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Startup, setup)
        .add_systems(Update, (movement_system, health_display))
        .run();
}

fn setup(mut commands: Commands) {
    // 生成 Entity，组合不同 Component
    commands.spawn((
        Position { x: 0.0, y: 0.0 },
        Velocity { dx: 1.0, dy: 0.5 },
        Health(100.0),
    ));
}
🌐 工程实践

组合 vs 继承：需要飞行能力？加 Flyable Component。需要伤害？加 DamageTaker。不需要设计类层次。

自动并行：Bevy 分析 System 的 Query 声明，无冲突的 System 自动在不同线程执行。

✏️ 示例分析与易错点

初学者常见疑问："Entity 似乎什么都没有？" —— 对！Entity 就是个 ID，Component 才有数据。这种分离是 ECS 的精髓。

03游戏循环与渲染

🎯 为什么需要了解这个

游戏的核心是循环——每秒 30/60/144 次重复"输入→更新→渲染"。理解这个循环是理解所有游戏引擎的基础。

🔍 核心思路

标准游戏循环：

  1. 输入处理：键盘、鼠标、手柄事件
  2. 更新逻辑：物理、AI、动画、碰撞检测
  3. 渲染：把当前状态画到屏幕上

Bevy 使用 Schedule 管理这些阶段：Startup（运行一次）、Update（每帧）、FixedUpdate（固定时间步）。

📐 规则与语法
RUSTgame_loop.rs
use bevy::prelude::*;

fn input_system(keys: Res<ButtonInput<KeyCode>>) {
    if keys.just_pressed(KeyCode::Space) {
        println!("Space pressed!");
    }
}

fn physics_system(
    time: Res<Time>,
    mut query: Query<(&Velocity, &mut Position)>
) {
    let dt = time.delta_secs();
    for (vel, mut pos) in &mut query {
        pos.x += vel.dx * dt;
        pos.y += vel.dy * dt;
    }
}

fn render_system(query: Query<(&Position, &Sprite)>) {
    // Bevy 的内置渲染器处理大部分工作
    // 这里可以做自定义渲染逻辑
    for (pos, _sprite) in &query {
        // 渲染在 Transform 组件的位置
        let _ = pos; // Bevy 自动渲染 Sprite
    }
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Update, (input_system, physics_system))
        .run();
}
🌐 工程实践

FixedUpdate vs Update：物理模拟用 FixedUpdate（固定时间步，确保确定性），视觉渲染用 Update（每帧，流畅感）。

delta time：逻辑计算必须乘以 time.delta_secs()，否则速度取决于帧率。

✏️ 示例分析与易错点

常见错误：忘记用 delta time，导致高帧率电脑上物体飞速移动。

04资源管理

🎯 为什么需要了解这个

一个中等规模的游戏可能有上千个纹理、模型和音频文件，总大小几个 GB。如何加载、共享、卸载这些资源是游戏开发的核心挑战之一。

🔍 核心思路
  • 异步加载：资源在后台线程加载，不阻塞游戏主循环
  • Handle：Bevy 用 Handle<T> 引用资源，多个 Entity 共享同一资源
  • 热重载：开发时修改纹理文件，游戏实时刷新，不需重启
📐 规则与语法
RUSTasset_loading.rs
use bevy::prelude::*;

fn setup(
    mut commands: Commands,
    asset_server: Res<AssetServer>,
) {
    // 异步加载——返回 Handle，资源在后台加载
    let texture: Handle<Image> = asset_server.load("sprites/player.png");
    let font: Handle<Font> = asset_server.load("fonts/FiraSans-Bold.ttf");

    // 多个 Entity 共享同一纹理
    commands.spawn(Sprite {
        image: texture.clone(),
        ..default()
    });
    commands.spawn(Sprite {
        image: texture,  // 同一个 Handle
        ..default()
    });
}

fn check_loading(asset_server: Res<AssetServer>) {
    // 可以检查加载状态
    // LoadState::Loaded, Loading, Failed ...
}
🌐 工程实践

资源目录：Bevy 默认从 assets/ 目录加载，路径相对于此。

加载屏幕：在 Startup 中发起所有加载，用一个 System 检查加载进度，完成后切换到游戏状态。

✏️ 示例分析与易错点

常见错误：忘记 clone Handle——Handle 实现了 Clone（内部只是个 ID），但移动后原变量就不能用了。

🎯 为什么需要了解这个

选择合适的工具和库能大幅提升开发效率。Rust 的游戏生态虽然年轻但正在快速成长。
