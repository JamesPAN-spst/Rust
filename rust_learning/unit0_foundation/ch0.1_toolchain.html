Ch 0.1 工具链

学 Rust 之前，先把“你在和谁打交道”搞清楚：安装器是谁、构建器是谁、编译器是谁、 为什么同一段代码在 debug 和 release 下表现不同，以及当编译器报错时应该先看哪里。

🎯 学习目标

  • 区分 rustup、cargo、rustc 在日常开发中的职责
  • 理解 toolchain、channel、edition 不是同一个概念
  • 知道何时用 debug 构建、何时切到 release 构建
  • 能把 rustfmt、clippy、编译器错误信息纳入日常反馈循环

📋 前置知识

本章默认你会打开终端、能运行命令，并知道“源码需要先编译再执行”。 还不要求你已经掌握所有权或借用；如果看到 Rust 代码片段里有陌生语法，先把重点放在工具链行为上。

0.1.1 rustup、cargo、rustc 分别在做什么？

🤔

你执行 cargo run 时，到底是谁负责安装 Rust、谁负责组织项目、谁真正把源码编译成二进制？

直觉上可以把它们看成三层：rustup 管“安装与切换工具链”， cargo 管“项目与工作流”，rustc 管“真正的编译”。 日常开发多数时候直接面对 cargo，但 cargo 最终仍会调用 rustc。

正式规则： rustup 管理 compiler/toolchain，cargo 管理 package/build/test/run，rustc 是底层编译器。

Rustsrc/main.rs
fn main() {
    println!("toolchain ready");
}
配套命令 
rustup show
cargo run
rustc --version

上面的 Rust 代码本身很简单，但它刚好能帮你观察分工：cargo run 会构建并运行，rustc --version 只告诉你编译器版本，rustup show 会告诉你当前工具链来源。

0.1.2 toolchain、channel、edition 为什么要分开理解？

🧭

很多初学者会把“我装的是 stable/nightly”与“这个项目是 2021 edition”混成一件事，这会直接影响排错。

toolchain / channel 说的是你拿哪套编译器工作，常见有 stable、beta、nightly； edition 说的是这个 crate 采用哪一代语言表面规则。 edition 会写在 Cargo.toml，它不是“安装了新版 Rust 就自动升级”的东西。

正式规则： toolchain 决定“用哪套编译器”，edition 决定“这个项目按哪代语言规则解释源码”。

项目配置： edition 通常写在 Cargo.toml 里，例如 edition = "2021"。

Rustsrc/main.rs
fn main() {
    // r#async 是 raw identifier：当名字和关键字冲突时可这样书写
    let r#async = "edition aware";
    println!("{r#async}");
}
说明 
raw identifier 说明某些标识符会受 edition 规则影响；
edition 本身仍由 Cargo.toml 声明。
因为 edition 是 crate 元数据，不是安装器全局开关。你可以用新的 stable 编译器继续维护一个 edition = "2018" 的项目。

0.1.3 debug 与 release 的差别，为什么不能只看一种结果？

⚙️

为什么 cargo run 很适合学习与调试，但你做性能判断时却常被提醒切到 --release？

cargo 默认使用 debug profile：编译更快、保留更多调试信息、优化较少。 cargo run --release 会启用 release profile：编译更慢，但优化更强，更接近最终发布表现。

正式规则： 不带 --release 时 cargo 使用 debug profile；带 --release 时切换到 release profile。

常见误区： “程序能跑” 不等于 “性能结论可信”。学习阶段先在 debug 看行为，性能判断再切 release。

Rustsrc/main.rs
fn main() {
    println!("debug_assertions = {}", cfg!(debug_assertions));
    debug_assert!(2 + 2 == 4);
}
运行观察 
cargo run            -> debug_assertions = true
cargo run --release  -> debug_assertions = false

0.1.4 rustfmt 与 clippy：它们不是语法课附属品，而是反馈工具

🛠️

“代码能编译” 只是底线；怎样尽早发现风格漂移和可改进写法？

rustfmt 负责把格式统一，减少无意义差异； clippy 负责给出 lint 建议，指出可读性、惯用法和潜在问题。 对新手来说，它们最重要的价值不是“通过检查”，而是帮助建立 Rust 社区常见写法的直觉。

正式定义： rustfmt 是格式化工具，clippy 是 lint 工具；二者都不替代编译器，但都能明显提升反馈质量。

Rustsrc/lib.rs
pub fn has_items(items: &[i32]) -> bool {
    if items.len() > 0 {
        true
    } else {
        false
    }
}
常见反馈 
cargo fmt
cargo clippy

Clippy 可能提示：
- 使用 !items.is_empty()
- 可以直接返回布尔表达式
可以写成 pub fn has_items(items: &[i32]) -> bool { !items.is_empty() }。逻辑更直接，也更贴近标准习惯。

0.1.5 读编译错误时，先抓“错误码、主语句、修复方向”

🔍

Rust 编译器输出看起来很长，初学者容易一眼只看到红字，却不知道先读哪三行。

一个实用顺序是：先看 error code / 主错误句，再看编译器标出的 发生位置，最后看 help / note。Rust 的报错通常不是只告诉你“坏了”，而是尽量把语言规则翻译成修复线索。

正式规则： 排错时先定位主错误，再结合 error code 与 help；后续连锁报错通常会随着首个错误修复而减少。

Rustsrc/main.rs
fn main() {
    let name = String::from("Ferris");
    let moved = name;
    println!("{}", name);
    println!("{}", moved);
}
你应该读到的重点 
error[E0382]: borrow of moved value: `name`
help: consider cloning the value if the performance cost is acceptable

真正的问题不是 println!，而是 name 的所有权已经移动。

读错题技巧： 不要先搜整段报错；先回答“是谁出问题、违反了哪条规则、编译器给了什么方向”。

📌 章节总结

  • rustup 管工具链，cargo 管项目工作流，rustc 才是底层编译器。
  • stable/nightly 是 toolchain 维度，edition 是项目语言时代维度，二者不要混淆。
  • debug 构建适合快速学习与排错，release 构建才更适合性能观察。
  • rustfmt 和 clippy 应该尽早进入日常循环，不要等到“写完再补”。
  • 阅读编译错误时优先抓主错误、定位和 help，而不是被长输出吓住。

🧪 课后测试

  1. ⭐ 基础： 用你自己的话解释 rustup、cargo、rustc 的职责差异，并写出各自最常用的一条命令。
  2. ⭐⭐ 进阶： 新建一个项目，分别运行 cargo run 与 cargo run --release，记录 cfg!(debug_assertions) 的差异并解释原因。
  3. ⭐⭐⭐ 挑战： 设计一个最小示例，让 clippy 给出一条惯用法建议；然后改写代码，并说明这条建议改善了什么。
  4. 🏆 附加： 选一条你最近遇到的 Rust 编译错误，按“错误码 → 主语句 → 规则 → 修复方案”的格式做一次排错复盘。
