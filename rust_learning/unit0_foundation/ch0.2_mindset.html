Unit 0 · 环境与心态

Ch 0.2 心态建设

Rust 的难点不只在语法，而在它会持续逼你解释“谁拥有数据、谁只借用、什么时候释放”。 本章的目标不是让你少犯错，而是帮你把这些错误变成可重复利用的学习反馈循环。

🎯 学习目标

  • 理解 Rust 学习曲线为什么常常是“前期陡、后期稳”
  • 建立“编译器是老师，不是敌人”的阅读报错方式
  • 提前预期所有权与借用阶段的正常挫败感
  • 形成 cargo check / cargo test / cargo clippy 的迭代工作流
  • 遇到 borrow checker 时优先重画数据流，而不是立刻堆 clone()

📋 前置知识

建议你已经完成 Ch 0.1，对 rustup、cargo、clippy 和基本项目结构有印象。 本章不要求你彻底掌握 ownership；相反，它会告诉你在还没掌握时，应该如何练习和排错。

0.2.1 Rust 的学习曲线为什么会“前陡后缓”？

🤔

为什么很多语言都能先“写着跑起来”，Rust 却常常在很早阶段就要求你回答所有权、借用和生命周期问题？

Rust 把很多原本可能在运行期暴露的问题，提前挪到了编译期。你感受到的“难”，很多时候不是语法更复杂， 而是语言要求你更早表达资源关系。前期会慢，是因为你在补一套别的语言经常推迟处理的工程约束。

正式理解： Rust 的学习曲线通常是前期投入较大，因为编译器会尽早要求程序员明确数据所有权、可变性和作用域边界。

Rustsrc/main.rs
fn print_name(name: &str) {
    println!("hello, {name}");
}

fn main() {
    let name = String::from("Ferris");
    print_name(&name);
    println!("still usable: {name}");
}
输出 
hello, Ferris
still usable: Ferris

这段代码说明：一旦你开始主动区分“只读借用”和“转移所有权”，Rust 反而会让后续行为更可预测。

0.2.2 把编译器当老师：先读规则，再读情绪

🧭

看到一大段红字时，你是先紧张，还是先定位“谁被 move 了、谁借用冲突了、编译器建议了什么”？

Rust 编译器最有价值的地方，是它往往不仅指出错误，还会告诉你被违反的规则和可能的修改方向。 真正有效的习惯不是“背所有错误码”，而是用固定顺序读：主错误句 → 标注位置 → help / note。

正式规则： 报错输出首先回答的是“哪条语言规则被破坏”，不是“你这个人不适合学 Rust”。

Rustborrow_error.rs
fn main() {
    let text = String::from("rust");
    let moved = text;
    println!("{}", text);
    println!("{}", moved);
}
你应该先读到的重点 
error[E0382]: borrow of moved value: `text`
help: consider borrowing or cloning if ownership should not move

重点不是 println! 坏了，
而是 text 的所有权已经转移。
1）谁拥有这份数据？ 2）当前这段代码只是读取，还是需要修改？ 3）这个值之后还要不要继续使用？ 这三个问题通常能直接缩小修复方向。

0.2.3 对 ownership 挫败感要有正常预期

🪨

为什么你明明“知道自己想做什么”，编译器却一直说不能同时借用、不能再次使用、生命周期不够长？

这是 Rust 初学阶段最正常的体验之一。你脑中也许已经有业务流程，但编译器还在要求你把数据流说清楚。 尤其进入 ownership / borrowing 阶段时，挫败感往往意味着：你的需求没错，只是表达方式还没对齐 Rust 的资源模型。

重要预期： 连续几天反复遇到 move、borrow、lifetime 报错，并不代表你“学不会”；这正是 Rust 把隐性工程问题显式化的过程。

Rustmutable_borrow.rs
fn main() {
    let mut msg = String::from("hi");
    let first = &msg;
    let second = &mut msg;
    println!("{first} {second}");
}
你该得到的直觉 
这里会报借用冲突。
不是因为 String 特殊，
而是同一时刻你要求了“只读视角”和“独占可变视角”。

把这类报错看成“模型不一致提示”会更有效：你需要决定当前阶段到底是读，还是改，而不是要求编译器同时允许两种互斥视角。

0.2.4 用 cargo 建立小步快跑的反馈循环

🔁

如果你总是写一大坨再一次性编译，Rust 的报错会不会显得特别“难收拾”？

很多 Rust 学习挫败来自迭代粒度太大。更实用的做法是：功能没写完前先跑 cargo check 看类型与借用， 行为有预期后跑 cargo test，代码能工作后再用 cargo clippy 清理惯用法问题。

正式工作流： cargo check 解决“能否通过规则检查”，cargo test 解决“行为是否正确”，cargo clippy 解决“写法是否更 Rusty”。

Rustsrc/lib.rs
pub fn add(a: i32, b: i32) -> i32 {
    a + b
}

#[cfg(test)]
mod tests {
    use super::add;

    #[test]
    fn adds_numbers() {
        assert_eq!(add(2, 3), 5);
    }
}
建议循环 
cargo check   # 先看编译与借用
cargo test    # 再看行为是否符合预期
cargo clippy  # 最后清理惯用法与潜在坏味道
因为 Rust 报错解决后，不代表逻辑一定正确。测试把“编译通过”和“行为正确”拆开，让你知道自己卡在哪一层。

0.2.5 少用 clone：先判断是不是只需要借用

🧪

编译器提示可以 clone() 时，你是真的需要一份独立副本，还是只是想绕过当前的所有权设计？

clone() 当然合法，也经常有用；问题在于新手容易把它当“消灭报错按钮”。 更好的学习习惯是先问：这段代码到底要“拥有”数据，还是只要“读一下”？如果只是读，很多时候传 &str、&String、&T 就够了。

正式规则： 只有当你确实需要一份独立拥有的副本时再 clone()；如果只是读取、格式化、检查长度、做匹配，通常应优先考虑借用。

Rustborrow_instead_of_clone.rs
fn print_uppercase(name: &str) {
    println!("{}", name.to_uppercase());
}

fn main() {
    let name = String::from("ferris");
    print_uppercase(&name);
    println!("original: {name}");
}
输出 
FERRIS
original: ferris

这里没有必要先写 let copy = name.clone();，因为函数只是读取名字并打印变换结果，没有接管 ownership。

📌 章节总结

  • Rust 的早期学习阻力，很多来自它把资源规则提前暴露，而不是因为你“不适合”。
  • 编译器报错应该被当成规则提示：先读主错误句，再决定是改所有权、借用还是作用域。
  • ownership / borrowing 阶段出现挫败感很正常，关键是把它转成数据流重构问题。
  • cargo check → cargo test → cargo clippy 是非常适合新手的小步反馈循环。
  • clone() 是明确语义工具，不该成为逃避 borrow checker 的默认动作。

🧪 课后测试

  1. ⭐ 基础： 用你自己的话解释为什么 Rust 的“前期陡”并不等于它整体更低效，并写出一个你准备坚持的排错阅读顺序。
  2. ⭐⭐ 进阶： 写一个最小示例，先触发一次 borrow of moved value，再改成借用版本，并说明你改变了哪条数据流。
  3. ⭐⭐⭐ 挑战： 给一个你最近写过的函数做微重构：把“先 clone() 再传参”的写法改成借用，并说明这样是否改变了 API 语义。
  4. 🏆 附加： 为一个小项目制定 15 分钟迭代节奏：每个时间片分别安排 cargo check、cargo test、cargo clippy 在什么时机执行，并解释原因。
